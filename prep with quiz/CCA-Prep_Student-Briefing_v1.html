Claude Certified Architect — Foundations · Student Briefing

What this project is and how to get the most from it

A direct orientation to the CCA-F prep system — what's in it, in what order to use it, and how it responds to repeated work.

What you're getting

What this project contains

The project is split into three layers that build on one another. You use them in order: course material first, domain practice second, mock exams third.

Layer What it is Format
Course modules Browser-based lessons across the 5 exam domains, with embedded exercises, concept explanations, and source citations. Two main courses (Claude 101 and Introduction to Subagents) with extension modules covering depth topics that appear on the exam. HTML — open in browser
Domain practice bank 51 original exam-style questions across all 5 domains, with an answer key per domain. Questions are answered first (no peeking), then you score yourself and log misses. Includes 3 cross-domain integration checkpoints that combine two domains in one scenario. Markdown — read in any viewer or text editor
Mock exams One 30-question warm mock (Mock Test #1) with interactive scoring and per-question explanations. Five domain-level held-out mock banks used only once each, as cold-run tests — not study material. HTML (warm mock) + Markdown (held-out banks)

There is also a condensed exam cram sheet (EXAM-DIGEST) and a gap tracker (GAPS.md) that you maintain yourself as you work through the material.

What is not yet available: The MCQ practice app that would generate fresh questions on demand is specified but not built. Until it ships, all practice questions are from the static bank and the mock test files listed above.
How to start

Entry point and recommended sequence

Open index.html in your browser. That is the Launchpad — it lists all available courses and links you to the practice materials. Everything else is accessible from there.

The sequence below is the one that delivers the most exam-relevant recall. Do not skip straight to the mock test — you will score poorly and have nothing actionable to do with the result.

Step 1
Complete the course modules in order. Start with Claude 101, then Introduction to Subagents. After each main course, work through its extension modules — they cover the exam's depth topics: prompt caching economics, MCP at a builder level, prompt engineering depth, context management, tool use and the agent loop, orchestration patterns, orchestrator-worker at scale, and subagents as an SDK primitive. Extensions are linked from the course pages.
Step 2
Work the domain practice bank, one domain at a time. Open the relevant file in the practice/ folder, write down your answers before scrolling to the key, then score yourself. Log every miss in GAPS.md: what was asked, what you picked, the correct mental model, and the one-line pattern to remember. Target ≥70% per domain. If you score below that, re-read the relevant module section before moving on. Start with D1 (Agentic Architecture — it carries 27% of the exam and underpins every other domain).
Step 3
Do the three cross-domain integration checkpoints. These are in the practice/ folder as CCA-F_checkpoint_A/B/C.md. Each checkpoint combines two domains in a single scenario — the same collision pattern the exam uses in its hardest questions. Do these cold (design the architecture from memory before reading the model answer). The checkpoints are:

A — Customer Support Agent (D1 × D5: escalation + context/caching)
B — CI/CD with Claude Code (D3 × D4: headless config + structured output)
C — Multi-Agent Research (D1 × D2: orchestration + tool/error design)
Step 4
Run Mock Test #1 as a warm mock. This is a 30-question interactive test covering all 5 domains in proportion to the exam blueprint. It takes roughly 60 minutes. Treat it like a real sitting: no notes, no looking things up. The final screen tells you your two weakest domains — revisit those before attempting the held-out mocks.
Step 5
Use the held-out mocks as cold-run tests. The files in practice/held-out-mocks/ contain per-domain question sets labeled by scenario, type, and difficulty. These have never appeared in the study material. Take each domain's held-out set once, under exam conditions, only after you have finished all study and practice for that domain. Do not use them for study or revision — they are your one clean signal of where you actually stand.
Repeated use

How the system responds to multiple passes

The content in this project is static — the same questions, the same explanations, the same course material each time you open a file. The improvement mechanism is yours to operate, not the system's.

Three things drive real gains across multiple passes:

Gap tracking

GAPS.md is the engine

Every miss you log in GAPS.md is a retrieval target. At the start of each new session, re-read your open gaps before touching new material. Move a gap to "Resolved" only after you get it right in three separate sessions without the note in front of you.

Spaced repetition

Wait before re-testing

Re-running a domain file the same day you studied it tests short-term recall, not exam-ready retention. Return to a domain after at least one day's gap. The spaced repetition only works if you let the gap exist.

Score discipline

The 70% threshold

The exam passing score scales to around 72% (720/1000). Do not advance to the next domain until you hit ≥70% on the current domain's practice bank. Tracking per-domain scores across runs shows you whether you are actually improving or just recognising questions.

Mock cadence

Hold the held-out mocks

After your first run of Mock Test #1, re-read only the explanations for questions you got wrong — not the full mock. On a second pass, score any domain that came in below 70% from its practice bank, not by re-running the same mock questions from memory.

The held-out mocks are one-shot instruments. Do not use them for repeated practice. They exist to give you one clean, uncontaminated reading of your readiness before the real exam.

Getting the maximum out of it

Tips and tricks

→ Answer before scrolling. In the practice bank files, the Questions section and the Answer Key are in the same file. Write your letter choices down before you scroll past the divider. The moment you see the answer, the question stops being a test and becomes a study note. You only get one clean shot at each question.
→ Scenario and anti-pattern questions carry more weight on the real exam. The practice bank deliberately over-indexes on these — especially in D1. When you miss one, the answer key tells you the sub-topic and why the tempting distractor fails. That "why the wrong answer feels right" explanation is more valuable than the correct answer itself. Read it carefully.
→ D1 first, every time. Agentic Architecture is 27% of the exam and its concepts appear as context in questions from every other domain. If your D1 mental models are shaky, D2, D3, D4, and D5 questions that cross into orchestration territory will cost you points you do not expect to lose.
→ Use the exam scenario names in your notes. The real exam draws 4 of 6 named scenarios per sitting: Customer Support Resolution Agent, Code Generation with Claude Code, Multi-Agent Research System, Developer Productivity with Claude, Claude Code for CI/CD, and Structured Data Extraction. Map your study gaps to these scenario names. If you have a gap in "structured handoff payload," that lives in the Customer Support scenario.
→ EXAM-DIGEST is for the final 48 hours, not the start. Open EXAM-DIGEST only after you have completed the course modules and at least one pass through the practice bank. It is a condensed cram sheet, not an introduction. Reading it first means you will not have the mental models to attach the information to — it will evaporate.
→ The mock test explanation is the lesson, not the score. When you get a question wrong in Mock Test #1, the explanation tells you why the correct answer is correct and what makes the wrong answer plausible. Sit with that for thirty seconds before hitting Next. The score is a signal; the explanation is the material.
→ Do not skip the cross-domain checkpoints. Checkpoints A, B, and C are where most exam candidates lose points — the exam's hardest questions combine two domains in a single scenario. The checkpoints force you to design the architecture cold before seeing the model answer. That cold attempt is the exercise; reading the answer without attempting it first is not.
→ Hard rules go in code, not prompts — learn that one cold. The "compliance in code vs compliance in prompt" concept appears in D1 and in D4, and it is a high-yield exam pattern. The wrong answer always sounds reasonable. Practise explaining to yourself why a hard threshold must live inside the tool, not in the system prompt, before exam day.
