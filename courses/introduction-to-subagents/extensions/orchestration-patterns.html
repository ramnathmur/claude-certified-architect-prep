🧱 Foundations

The line, the building block

Before the patterns: the one architectural distinction the exam keeps testing, and the unit every pattern is built from.

🪢 Section 1⏱ ~7 min · concept

Workflows vs. Agents

What you'll be able to do
  • State the hard architectural line Anthropic draws between workflows and agents
  • Use "who controls the path?" to classify any given system
  • Place the five named patterns and the autonomous loop on the right side of that line

Why before how: if you cannot tell a workflow from an agent, every later "which pattern?" decision is built on sand. The exam tests this distinction directly, so it goes first.

Anthropic groups everything under the umbrella term "agentic systems" but draws a hard line between two sub-types:

  • Workflows — "systems where LLMs and tools are orchestrated through predefined code paths."
  • Agents — "systems where LLMs dynamically direct their own processes and tool usage, maintaining control over how they accomplish tasks."

The dividing question is who controls the path — the developer's code (workflow) or the model itself at runtime (agent). Hold onto that single test; it is the most reliable classifier on the exam.

Analogy A workflow is a railway: the track is laid in advance and the train can only go where the rails go. An agent is a taxi driver: you give a destination, and the driver chooses the route, reroutes around traffic, and decides when to stop for fuel. (Author analogy, not a product fact.)

🤔 Predict first
A team builds a system: a script calls the model to extract fields, then always calls it again to format them into JSON — same two steps every time, in code. Workflow or agent?
Workflow. The steps are orchestrated through predefined code paths — the script controls the route, not the model. It would only become an agent if the model itself decided, at runtime, what steps to take and which tools to use.
❓ Quick check
What single question best classifies a system as workflow vs. agent?
Workflows orchestrate through predefined code paths; agents let the model dynamically direct its own process. Tool use, call count, and parallelism don't decide it — control of the path does.
🔑 Key takeaway▼

Both are "agentic systems." The line is control of the path: workflow = predefined code paths; agent = model dynamically directs itself. The five patterns ahead are all workflows; only the loop in Section 6 is a true agent.

⚙️ Section 2⏱ ~5 min · concept

The Augmented LLM — the building block

What you'll be able to do
  • Name the three augmentations that turn a raw model into the unit every pattern composes
  • Explain why you get the single augmented call right before reaching for any multi-step pattern

Every pattern in this lesson is composed from one repeated unit: the augmented LLM — "an LLM enhanced with augmentations such as retrieval, tools, and memory." Current models "can actively use these capabilities — generating their own search queries, selecting appropriate tools, and determining what information to retain."

Why this matters first: the patterns compose augmented LLMs; they don't replace the need to get one right. Before reaching for any multi-step pattern, make sure the single augmented call is well-built — good tooling, retrieval, and a tailored interface.

❓ Quick check
Which three augmentations define the "augmented LLM" building block?
The augmented LLM is "enhanced with augmentations such as retrieval, tools, and memory" — and current models actively use them (generating queries, selecting tools, deciding what to retain).
🧩 The five patterns

Five workflow patterns

Each is a way to compose augmented LLMs through predefined code paths. Learn each by its fit signal — the situation that makes it the right call.

🔗 Pattern 1⏱ ~8 min · pattern

Prompt Chaining

Definition. Prompt chaining "decomposes a task into a sequence of steps, where each LLM call processes the output of the previous one."

Gate checks. "You can add programmatic checks (a 'gate' in the diagram) on any intermediate steps to ensure the process is still on track." The gate is plain code — it can stop, retry, or reroute the chain before bad output propagates.

When to use. "Ideal for situations where the task can be easily and cleanly decomposed into fixed subtasks." It trades "off latency for higher accuracy by making each LLM call an easier task."

Concrete examples (from the source):

  • "Generating marketing copy, then translating it into a different language."
  • "Writing an outline of a document, checking that the outline meets certain criteria, then writing the document based on the outline."

Maps to subagents. A chain is typically implemented in code paths (a predefined sequence), not by delegation — but any step that needs context isolation can itself be a tool-restricted subagent.

🤔 Predict first
Chaining "trades off latency for higher accuracy." Why would breaking one prompt into several sequential calls raise accuracy?
Because it makes "each LLM call an easier task." A single mega-prompt asks the model to do everything at once; splitting it lets each call focus on one cleanly-decomposable subtask — at the cost of more round-trips (latency).
❓ Quick check
What is the purpose of a "gate" in a prompt chain?
The gate is a programmatic check on intermediate steps. Parallel runs = parallelization; voting = a parallelization variant; classify-and-dispatch = routing.
🔀 Pattern 2⏱ ~8 min · pattern

Routing

Definition. Routing "classifies an input and directs it to a specialized followup task."

Why it helps. "This workflow allows for separation of concerns, and building more specialized prompts." Without routing, optimizing for one kind of input tends to hurt others.

When to use. "Complex tasks where there are distinct categories that are better handled separately, and where classification can be handled accurately." That second clause is the trap: if the classifier is unreliable, routing hurts.

Concrete examples (from the source): "Directing different types of customer service queries (general questions, refund requests, technical support) into different downstream processes, prompts, and tools." A second example — routing easy/common questions to a smaller model and hard/unusual ones to a more capable model — optimizes cost and speed.

Maps to subagents. This is exactly how Claude decides which subagent to invoke — "Claude determines whether to invoke them based on each subagent's description field." A well-written description is a router classifier. The base-course Lesson 3 insight ("the description controls when a subagent launches") is, in pattern terms, routing.

❓ Quick check
Routing is the right pattern when…
Routing = classify-then-dispatch into distinct categories, and it depends on accurate classification. Unpredictable subtasks → orchestrator-workers; multi-attempt confidence → parallelization (voting); previous-output chaining → prompt chaining.
⫴ Pattern 3⏱ ~9 min · pattern

Parallelization — Sectioning & Voting

Definition. "LLMs can sometimes work simultaneously on a task and have their outputs aggregated programmatically." There are two variations — and the exam wants you to tell them apart:

  • Sectioning — "breaking a task into independent subtasks run in parallel." (Different work, in parallel.)
  • Voting — "running the same task multiple times to get diverse outputs." (Same work, for redundancy/confidence.)

When to use. Effective "when the divided subtasks can be parallelized for speed, or when multiple perspectives or attempts are needed for higher confidence results."

Concrete examples (from the source):

  • Sectioning: guardrails where one model instance processes the user query while another screens it for inappropriate content (separate attention per concern); also automated evals where each LLM call assesses a different aspect of performance.
  • Voting: "Reviewing a piece of code for vulnerabilities, where several different prompts review and flag the code if they find a problem."

Maps to subagents (SDK + Claude Code). "Multiple subagents can run concurrently, dramatically speeding up complex workflows." The docs' example runs style-checker, security-scanner, and test-coverage simultaneously, "reducing review time from minutes to seconds." Each parallel worker gets isolation: "Each subagent runs in its own context window."

🤔 Predict first
You run the same vulnerability-review prompt three times with slightly different framings and flag the code if any run finds a problem. Sectioning or voting?
Voting. It's the same task run multiple times for higher confidence — the verbatim source example. Sectioning would instead split the review into different independent subtasks (e.g., one call per concern) running in parallel.
❓ Quick check
The clearest distinction between sectioning and voting is:
Sectioning = independent subtasks in parallel (split the work). Voting = the same task repeated for diverse outputs / higher confidence (redundancy). Both are variations of parallelization.
🎯 Pattern 4⏱ ~10 min · pattern · most exam-relevant

Orchestrator-Workers

Definition. "A central LLM dynamically breaks down tasks, delegates them to worker LLMs, and synthesizes their results."

The key difference vs. parallelization. "Its flexibility — subtasks aren't pre-defined, but determined by the orchestrator based on the specific input." In parallelization you (the developer) decide the splits up front; here the orchestrator decides them at runtime. That one word — predefined vs. determined-at-runtime — is the exam's favorite distractor.

When to use. "Well-suited for complex tasks where you can't predict the subtasks needed."

Concrete examples (from the source): "Coding products that make complex changes to multiple files each time"; also search tasks that gather and synthesize information from multiple sources.

Maps to subagents. This is the native shape of Claude Code / Agent SDK subagent delegation: a main agent spawns focused workers via the Agent tool and receives "the subagent's final message verbatim as the Agent tool result." If you remember one pattern↔subagent mapping for the exam, make it this one.

🤔 Predict first
A coding agent must edit "an unknown number of files that depend on the bug" — you can't list the files before you start. Orchestrator-workers or parallelization?
Orchestrator-workers. The subtasks (which files to change) "can't be predicted" and must be "determined by the orchestrator based on the specific input." Parallelization requires you to define the splits up front, which you can't do here. Coding changes across multiple files is the verbatim source example for this pattern.
❓ Quick check
What is the single key difference between orchestrator-workers and parallelization?
"The key difference from parallelization is its flexibility — subtasks aren't pre-defined, but determined by the orchestrator based on the specific input." Speed and worker-count aren't the defining line.
♻️ Pattern 5⏱ ~8 min · pattern

Evaluator-Optimizer

Definition. "One LLM call generates a response while another provides evaluation and feedback in a loop."

When to use. "Particularly effective when we have clear evaluation criteria, and when iterative refinement provides measurable value." Two fit signals: (1) responses can be demonstrably improved by feedback, and (2) the LLM can provide that feedback — "analogous to a human writer's revision process."

Concrete examples (from the source): "Literary translation where there are nuances that the translator LLM might not capture initially, but where an evaluator LLM can provide useful critiques." Also complex search requiring multiple rounds, where the evaluator decides whether further searches are warranted.

Maps to subagents. Implement the evaluator as a separate tool-restricted subagent — e.g., a read-only reviewer that critiques the generator's output before the loop continues.

❓ Quick check
Evaluator-optimizer is "particularly effective" when…
Clear criteria + measurable value from iteration are the fit signals — plus the LLM being able to provide useful feedback (like a writer's revision process). The others describe orchestrator-workers, routing, and parallelization.

🏆 Quiz A — The five patterns

0 / 5
Q1
A support system classifies each ticket as billing / technical / general and sends it to a tailored prompt. Which pattern?
Classify-then-dispatch into distinct categories is routing — the verbatim customer-service example.
Q2
A translator LLM produces a draft; an evaluator LLM critiques nuance; the translator revises, looping until quality criteria are met. Which pattern?
Generate-and-critique in a loop with clear criteria = evaluator-optimizer; literary translation is its verbatim example.
Q3
Three differently-framed prompts each independently review the same code for vulnerabilities and any of them can flag it. Which pattern (and variation)?
Same task run multiple times for confidence = voting, a variation of parallelization. It's the source's verbatim code-vulnerability example.
Q4
A coding agent makes "complex changes to multiple files each time" and the affected files can't be listed in advance. Which pattern?
Subtasks determined at runtime by a central LLM = orchestrator-workers; multi-file coding changes is its verbatim example. Sectioning needs pre-defined splits.
Q5
"Write an outline, check it meets criteria, then write the doc from the outline" — each step consuming the last. Which pattern?
Each LLM call processes the previous call's output, with a programmatic gate checking the outline = prompt chaining. It's the verbatim source example.
🚦 Choosing

The agent loop & the decision framework

When the path can't be predefined at all — and the simplicity-first ladder for choosing the least-complex option.

🛰️ Section 6⏱ ~7 min · concept

Autonomous Agents — the loop

The five patterns are workflows. The autonomous agent is the other side of the line from Section 1 — and it operates as a loop, not a fixed sequence.

How they operate. Agents "begin their work with either a command from, or interactive discussion with, the human user. Once the task is clear, agents plan and operate independently, potentially returning to the human for further information or judgement." Critically: "During execution, it's crucial for the agents to gain 'ground truth' from the environment at each step (such as tool call results or code execution) to assess its progress."

When to use. "Open-ended problems where it's difficult or impossible to predict the required number of steps, and where you can't hardcode a fixed path."

The caution. Autonomy means higher costs and the potential for compounding errors. Anthropic recommends extensive testing in sandboxes plus appropriate guardrails. This caution is the bridge to the decision framework: more autonomy is not free.

⚠️ The cost of autonomy▼

Agents trade predictability for flexibility. Their autonomy means higher costs and compounding errors (a wrong step early can cascade). That's why "ground truth at each step" matters — it's the feedback that keeps the loop honest — and why Anthropic pairs agents with sandbox testing + guardrails. Don't reach for an agent loop when a fixed workflow would do.

❓ Quick check
Why must an autonomous agent gain "ground truth" from the environment at each step?
Ground truth (tool results, code execution) lets the agent assess progress on an open-ended task with no hardcodable path — that feedback is what keeps the loop on track and limits compounding errors.
🚦 Section 7⏱ ~10 min · framework · high-yield

The Decision Framework — simplicity first

What you'll be able to do
  • State the simplicity-first rule and the single-call default in the source's own words
  • Walk the 7-rung escalation ladder and justify stopping at the lowest rung that works
  • State the workflow-vs-agent tradeoff that governs the top of the ladder

The core rule (verbatim). "When building applications with LLMs, we recommend finding the simplest solution possible, and only increasing complexity when needed. This might mean not building agentic systems at all."

The single-call default. "For many applications… optimizing single LLM calls with retrieval and in-context examples is usually enough." Most problems never need a pattern.

Escalation guidance (verbatim). "Start with simple prompts, optimize them with comprehensive evaluation, and add multi-step agentic systems only when simpler solutions fall short."

The governing tradeoff. "Workflows offer predictability and consistency for well-defined tasks, whereas agents are the better option when flexibility and model-driven decision-making are needed at scale."

The 7-rung escalation ladder

Synthesized from the guidance above: climb only as high as you must. Each rung is justified by a fit signal; stop at the lowest rung that solves the problem.

🤔 Predict first
A product manager proposes a multi-agent system for a task that a single well-built LLM call with retrieval already handles at the needed quality. What does the framework say?
Don't build it. "Find the simplest solution possible, and only increase complexity when needed — this might mean not building agentic systems at all." For many applications, "optimizing single LLM calls with retrieval and in-context examples is usually enough." Escalate only when simpler solutions fall short.

The workflow-vs-agent tradeoff, per the source, is best stated as:
That's the verbatim tradeoff. The framework's spirit is simplicity-first: prefer the predictable workflow unless flexibility/model-driven decisions at scale justify an agent.

🔬 Case study⏱ ~12 min · production reference

Orchestrator-Workers at Scale: the Multi-Agent Research System

Anthropic's multi-agent research post is the production reference architecture for Pattern 4. It turns the abstract orchestrator-workers diagram into a shipped system — with measured numbers you should be able to recall on the exam.

The reference architecture

  • A lead agent analyzes the query, develops a strategy, and spawns subagents.
  • Subagents operate in parallel, each with its own context window: "Subagents facilitate compression by operating in parallel with their own context windows, exploring different aspects of the question simultaneously."
  • Subagents perform web searches, use interleaved thinking to evaluate results, and return findings to the lead agent, which synthesizes and decides whether more research is needed.
  • A dedicated CitationAgent then "processes the documents and research report to identify specific locations for citations," ensuring proper attribution.

Why multi-agent works here. "The essence of search is compression: distilling insights from a vast corpus." Parallel subagents with separate context windows each compress a slice — together exceeding what a single context window could hold.

Effort-scaling heuristics (given to the lead agent, verbatim)

Query typeSuggested allocation
Simple fact-finding"1 agent with 3-10 tool calls"
Direct comparisons"2-4 subagents with 10-15 calls each"
Complex research"more than 10 subagents with clearly divided responsibilities"

Purpose: "These guidelines help the lead agent allocate resources efficiently and prevent overinvestment in simple queries." (Same simplicity-first spirit as the decision ladder — applied inside a running system.)

The economic gate — when multi-agent is worth it

"There's a real tension between autonomy and reliability." Multi-agent systems "require tasks where the value of the task is high enough to pay for the increased performance." They excel at "valuable tasks that involve heavy parallelization, information that exceeds single context windows." They are not suited to tasks where all agents need to share the same context, or where there are many tight dependencies between agents.

🔑 The multi-agent decision, in one line▼

Multi-agent earns its ~15× token cost only when the task is high-value, heavily parallelizable, and exceeds a single context window — and when agents don't need to share context or carry tight inter-agent dependencies. If agents must share context or depend tightly on each other, a single agent or a simpler workflow is the better call.

🤔 Predict first
A task needs all sub-tasks to share one evolving context and has tight step-to-step dependencies, but it's only moderately valuable. Good fit for multi-agent?
No. Multi-agent is "not suited to tasks where all agents need to share the same context or have many tight dependencies between agents," and it "requires tasks where the value… is high enough to pay for the increased performance" (~15× tokens). This task fails on shared context, dependencies, and value. Use a single agent or a simpler workflow.
❓ Quick check
By how much did the multi-agent system (Opus 4 lead + Sonnet 4 subagents) outperform single-agent Opus 4 on the internal research eval?
90.2% on the internal research eval. (4× and 15× are the token-use multipliers for agents and multi-agent systems vs. chat; 40% is the tool-description completion-time figure.)
🧰 Implementation⏱ ~12 min · how-to

How Subagents Implement These Patterns

What you'll be able to do
  • Map routing, parallelization, and orchestrator-workers onto concrete subagent mechanics
  • State exactly what crosses the parent→subagent context boundary (and what doesn't)
  • Apply the no-nesting constraint and know when to graduate to the Workflow tool

What a subagent is. "Subagents are separate agent instances that your main agent can spawn to handle focused subtasks" — to "isolate context for focused subtasks, run multiple analyses in parallel, and apply specialized instructions without bloating the main agent's prompt." In Claude Code: "Each subagent runs in its own context window with a custom system prompt, specific tool access, and independent permissions."

How subagents realize each pattern

PatternSubagent mechanic
RoutingClaude matches a task to a subagent's description (a router classifier); or you invoke it explicitly by name ("Use the code-reviewer agent to…").
Parallelization"Multiple subagents can run concurrently"; each runs in an isolated context window so parallel work doesn't pollute the main conversation.
Orchestrator-workersThe main agent delegates via the Agent tool and synthesizes the verbatim final messages returned by workers.

A filesystem subagent definition (Claude Code)

Subagent config — YAML frontmatter + markdown body
---
name: code-reviewer
description: Reviews code for quality and best practices
tools: Read, Glob, Grep
model: sonnet
---
You are a code reviewer. When invoked, analyze the code and provide
specific, actionable feedback on quality, security, and best practices.

"The frontmatter defines the subagent's metadata and configuration. The body becomes the system prompt." Key AgentDefinition fields (SDK): description (required — drives routing), prompt (required — system prompt), tools/disallowedTools (capability restriction), model (sonnet/opus/haiku/inherit or full ID), plus skills, memory, mcpServers, maxTurns, background, effort, permissionMode.

Context isolation — what crosses the boundary

This is an architecture-decision point. "A subagent's context window starts fresh (no parent conversation)… The only channel from parent to subagent is the Agent tool's prompt string." The subagent receives its own system prompt + the Agent tool prompt + project CLAUDE.md (if settingSources loads it) + tool defs. It does not receive the parent's conversation history, tool results, or system prompt.

🚫 Critical constraint — no nesting▼

"Subagents cannot spawn their own subagents." The orchestration tree is therefore one level deep by default — a main agent and its workers, not arbitrarily nested agents. When you design an orchestrator-workers system on subagents, the orchestrator is the only layer that delegates.

📈 Scaling beyond a few workers — the Workflow tool▼

"Subagents work well for a few delegated tasks per turn. For runs that coordinate dozens to hundreds of agents, use the Workflow tool, which moves the orchestration into a script the runtime executes outside the conversation context." This is the graduation path: a handful of workers → subagents; large-scale fan-out → Workflow tool.

Built-in subagents (Claude Code): Explore (Haiku, read-only, codebase search), Plan (inherits model, read-only, used in plan mode), General-purpose (inherits model, all tools, complex multi-step work).

Managing them — the /agents command: opens a tabbed interface; the Running tab shows live subagents and the Library tab creates/edits/deletes them ("the recommended way"). Scope precedence (highest→lowest): managed settings (org) → --agents CLI flag (session) → .claude/agents/ (project) → ~/.claude/agents/ (user) → plugin agents/ directory.

⚠️ VERIFY (naming, preserved from source): the delegation tool was "renamed from Task to Agent in Claude Code v2.1.63"; the current SDK emits Agent in tool_use blocks but still uses Task in the system:init tools list. Verify against the current version before citing a specific version number.
🤔 Predict first
You want a worker subagent to coordinate its own team of sub-workers, and you also need to fan out across ~200 agents. What does the source tell you?
Two facts collide. First, "subagents cannot spawn their own subagents" — the tree is one level deep, so a worker can't run its own sub-workers. Second, for runs coordinating "dozens to hundreds of agents," you graduate to the Workflow tool, which runs the orchestration in a script outside the conversation context — not by nesting subagents.
❓ Quick check
What is the ONLY channel from a parent agent into a subagent's fresh context?
"The only channel from parent to subagent is the Agent tool's prompt string." The subagent gets its own system prompt + that prompt + tool defs + CLAUDE.md (if loaded) — never the parent's history, results, or system prompt.
