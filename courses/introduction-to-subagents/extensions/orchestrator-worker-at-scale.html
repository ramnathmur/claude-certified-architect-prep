🔁 Recap & the line

Dynamic decomposition, not a fixed fan-out

What orchestrator-worker actually is, why it is categorically different from static parallelization, and how the SDK subagent primitive realizes it.

🔁 Section 1⏱ ~7 min · concept

Recap — what orchestrator-worker is

What you'll be able to do
  • Recite Anthropic's verbatim definition of the orchestrator-workers workflow
  • Draw the line between dynamic decomposition and static parallelization
  • Connect the pattern to the SDK subagent primitive from S-E2

Why before how: before we worry about what breaks at scale, we need the pattern crisp. Anthropic's definition is exact — in the orchestrator-workers workflow, "a central LLM dynamically breaks down tasks, delegates them to worker LLMs, and synthesizes their results." Three verbs do all the work: break down, delegate, synthesize. Hold onto them — Sections C, D, and E are basically "what goes wrong with each verb, and how to fix it."

It "is well-suited for complex tasks where you can't predict the subtasks needed" — where "subtasks aren't pre-defined, but determined by the orchestrator based on the specific input." That word determined-at-runtime is the whole game.

The line that defines the pattern: dynamic vs. static

Plain parallelization comes in two flavors Anthropic names verbatim: "Sectioning: Breaking a task into independent subtasks run in parallel. Voting: Running the same task multiple times to get diverse outputs." In both, the subtasks are fixed in code — you wrote the fan-out ahead of time. In orchestrator-workers, the central LLM decides at runtime what the subtasks even are. That dynamic, LLM-driven decomposition is the defining feature: the orchestrator adapts the breakdown to each specific input rather than executing a predetermined fan-out.

Analogy Static parallelization is a printed assembly line — every station is bolted down before the first part arrives. Orchestrator-worker is a newsroom editor who reads the story that just broke, then decides on the spot how many reporters to assign and to what angles. Same "many workers in parallel" shape; completely different decision-maker. (Author analogy, not a product fact.)

The SDK realization — subagents are the primitive

From S-E2 you already know the mechanism. In the Claude Agent SDK, "Subagents are separate agent instances that your main agent can spawn to handle focused subtasks." You use them "to isolate context for focused subtasks, run multiple analyses in parallel, and apply specialized instructions without adding to the main agent's prompt." That is exactly the orchestrator-worker pattern expressed as code: the main agent is the orchestrator; each spawned subagent is a worker.

🤔 Predict first
You hard-code a function that always spawns exactly 3 workers — one per fixed section of a report — regardless of the input. Is that orchestrator-worker?
No — that's static parallelization (sectioning). The subtasks are fixed in code, not "determined by the orchestrator based on the specific input." It only becomes orchestrator-worker when a central LLM decides at runtime how to break the task down — which may be 1 worker, or 12.
❓ Quick check
What single feature most distinguishes orchestrator-workers from static parallelization?
The defining line is dynamic decomposition: "subtasks aren't pre-defined, but determined by the orchestrator based on the specific input." Static parallelization fixes the subtasks in code ahead of time.
🔑 Key takeaway▼

Orchestrator-worker = break down → delegate → synthesize, where the breakdown is decided at runtime by a central LLM. If the fan-out is hard-coded, it's static parallelization, not this pattern. The SDK realizes it with the subagent primitive: main agent = orchestrator, spawned subagents = workers.

📈 The economics

Why scale to multi-agent — and what it costs

Context isolation, separation of concerns, and parallelism are the upside. The ~15× token bill is the downside. The numbers that decide whether the trade is worth it.

📈 Section 2⏱ ~11 min · core · economics

Why scale to multi-agent

What you'll be able to do
  • State the three rationales for going multi-agent — and ground each in a source
  • Recall the headline numbers: ~90.2% lift, ~15× tokens, 80% of variance from tokens
  • Explain why multi-agent wins — the token story underneath the architecture

Anthropic productionized orchestrator-workers in their Research feature: "a lead agent coordinates the process while delegating to specialized subagents that operate in parallel." Why bother, given the cost? Three reasons — and they map directly onto Domains 1 and 5.

The three rationales

  1. Context isolation / compression (Domain 5). "Subagents facilitate compression by operating in parallel with their own context windows, exploring different aspects of the question simultaneously before condensing the most important tokens for the lead research agent." The SDK echoes the mechanism: "Each subagent runs in its own fresh conversation. Intermediate tool calls and results stay inside the subagent; only its final message returns to the parent." That last clause is the compression engine — the lead never has to hold the workers' raw exploration in its own window.
  2. Separation of concerns (Domain 1). "Each subagent also provides separation of concerns—distinct tools, prompts, and exploration trajectories—which reduces path dependency and enables thorough, independent investigations."
  3. Parallelism / speed. "Parallel tool calling transforms speed and performance. These changes cut research time by up to 90% for complex queries, allowing Research to do more work in minutes instead of hours."

The token economics, visualized

💡 The why underneath: it's a token story▼

The deep reason multi-agent wins is almost embarrassingly mechanical. More agents exploring in parallel = more tokens spent. Since token usage explains ~80% of the performance variance, the multi-agent architecture wins largely because it is a mechanism for spending more tokens productively in parallel. That reframing is the exam-relevant insight: it tells you exactly when the architecture won't help — when the task can't absorb more parallel token spend.

❓ Quick check
Which single factor, by itself, explained ~80% of the performance variance in Anthropic's analysis?
"Token usage by itself explains 80% of the variance." Three factors together explained 95% (token usage, number of tool calls, model choice), but token usage dominates. VERIFY figures.
🤔 Predict first
If a multi-agent system costs ~15× a chat in tokens, what must be true of a task for that spend to be justified?
It must be valuable AND heavily parallelizable. "Multi-agent systems excel at valuable tasks that involve heavy parallelization." If the task is low-value, or fundamentally sequential / shared-state, the 15× spend buys little — and you should fall back to a single agent or a fixed workflow (Section F).
🎛️ The orchestrator

The orchestrator's job at scale

Decomposition with detailed delegation, effort heuristics tied to query complexity, synthesis, and a separate citation/verification stage — plus the named prompt-engineering principles.

🎛️ Section 3⏱ ~12 min · core

What the orchestrator actually does

What you'll be able to do
  • List the four jobs of a scaled orchestrator and what each requires
  • Apply the effort heuristics (agent + tool-call counts) to query complexity
  • Explain why verification is its own architectural stage (the CitationAgent)
  • Name Anthropic's six prompt-engineering principles for the lead agent

In the Research architecture, the lead agent "analyzes [the query], develops a strategy, and spawns subagents to explore different aspects simultaneously." Subagents work "iteratively using search tools to gather information" and return findings; the system then "passes all findings to a CitationAgent" before returning results. Four jobs hide in that sentence.

Job 1 — Decomposition + detailed delegation

"Each subagent needs an objective, an output format, guidance on the tools and sources to use, and clear task boundaries. Without detailed task descriptions, agents duplicate work, leave gaps, or fail to find necessary information." Anthropic learned this the hard way: vague instructions like "research the semiconductor shortage" caused subagents to misinterpret the task or perform "the exact same searches as other agents." Detailed delegation is not politeness — it is the mechanism that prevents failure mode #5 (duplicated work).

Job 2 — Effort heuristics scaled to complexity

"Agents struggle to judge appropriate effort for different tasks, so we embedded scaling rules in the prompts." The explicit rules:

The SDK exposes effort directly as a field: effort accepts 'low' | 'medium' | 'high' | 'xhigh' | 'max' or a number, and maxTurns caps "Maximum number of agentic turns before the agent stops."

SDK — effort + maxTurns on a worker (escaped) · VERIFY enum values, SDK-version-sensitive
// A worker scaled for a "direct comparison" task
agents: {
  "comparison-worker": {
    description: "Compares two options across given dimensions.",
    prompt: "You compare exactly the two items named...",
    tools: ["Read", "Grep", "Glob"],
    effort: "medium",        // 'low'|'medium'|'high'|'xhigh'|'max' | number
    maxTurns: 15            // cap agentic turns (≈ the 10–15 calls heuristic)
  }
}

Job 3 — Synthesis

The lead agent gathers subagent findings and condenses them. Crucially, subagents are designed to return only "the most important tokens for the lead research agent" — the compression happens at the boundary, so the lead synthesizes condensed findings, not raw transcripts.

Job 4 — Citation / verification stage

A dedicated "CitationAgent" "processes the documents and research report to identify specific locations for citations," ensuring claims are attributed before delivery. This is a distinct worker role: verification is architecturally separated from research. The agents that find information are not the agent that checks it — a separation-of-concerns move that doubles as a reliability control.

🤔 Predict first
A user asks a simple, single-fact question ("what year did X launch?"). How many agents and tool calls should the orchestrator deploy, per Anthropic's embedded rules?
Just 1 agent with 3–10 tool calls. "Simple fact-finding" gets a single agent and a small call budget. Spinning up 4 subagents for a one-fact query is exactly the over-spend the effort heuristics exist to prevent. VERIFY counts.
❓ Quick check
Why is the CitationAgent a separate stage rather than folded into each research subagent?
A dedicated CitationAgent "processes the documents and research report to identify specific locations for citations." Separating find from verify is both separation of concerns and a reliability control.
❓ Quick check
What does a subagent need in its task description to avoid duplicating other workers' efforts?
"Each subagent needs an objective, an output format, guidance on the tools and sources to use, and clear task boundaries." Vague delegation is what caused real subagents to run "the exact same searches as other agents."
💥 The heart

Failure modes at scale

Eight ways orchestrator-worker systems break in production — each as name → mechanism → symptom → mitigation — plus the academic MAST cross-reference (handled carefully).

💥 Section 4⏱ ~16 min · core · the heart

The eight failure modes

What you'll be able to do
  • Name all eight failure modes by mechanism, symptom, and mitigation
  • Compute the coordination-interaction growth n·(n−1)/2
  • Cite the MAST academic taxonomy correctly — without overstating the unconfirmed percentages

Big picture first: single-agent failures are mostly "the model got it wrong." Multi-agent failures are systemic — they emerge from state, coordination, and economics, not just from any one model's mistake. That is why this is the heart of the lesson and the heart of Domain 5. Each mode below is given as name → mechanism → symptom → mitigation.

① Error compounding in stateful, multi-step agents▼

Mechanism: "Agents are stateful and errors compound. Agents can run for long periods of time, maintaining state across many tool calls." Chaining steps means errors propagate forward instead of canceling out. [SECONDARY] Galileo confirms: "errors often compound rather than cancel when you chain agents sequentially."

Symptom: "Without effective mitigations, minor system failures can be catastrophic for agents."

Mitigation: Durable execution + resume-from-failure; deterministic safeguards (retries, checkpoints) — see Section E.

② Context-window exhaustion across workers▼

Mechanism: Long-horizon runs blow past the context window. Anthropic: "if the context window exceeds 200,000 tokens it will be truncated and it is important to retain the plan." VERIFY 200,000-token figure. "As conversations extend, standard context windows become insufficient."

Symptom: Plan/context truncated; the agent loses the thread of its own strategy; degraded or incoherent output.

Mitigation: (a) External memory — "The LeadResearcher begins by thinking through the approach and saving its plan to Memory to persist the context." (b) Summarize-and-store — "agents summarize completed work phases and store essential information in external memory before proceeding to new tasks." (c) Clean-context handoffs — "When context limits approach, agents can spawn fresh subagents with clean contexts while maintaining continuity through careful handoffs." (d) Subagent context isolation by design — intermediate results never enter the parent context.

③ Token-cost blowup (the 15× economics)▼

Mechanism: Multi-agent systems "use about 15× more tokens than chats." VERIFY Every spawned subagent multiplies token consumption.

Symptom: Cost per query that may not justify the result; the architecture stops paying for itself on low-value or non-parallelizable tasks.

Mitigation: Gate on value + parallelizability — "Multi-agent systems excel at valuable tasks that involve heavy parallelization." Scale effort to complexity (Section C); fall back to single-agent or a fixed workflow when the task doesn't clear the gate (Section F).

④ Coordination / synchronization overhead▼

Mechanism: "Multi-agent systems have key differences from single-agent systems, including a rapid growth in coordination complexity." Anthropic's lead agents "execute subagents synchronously, waiting for each set of subagents to complete before proceeding… the entire system can be blocked while waiting for a single subagent to finish searching." [SECONDARY] Galileo quantifies the interaction explosion: "2 agents = 1 potential interaction; 4 agents = 6 potential interactions; 10 agents = 45 potential interactions" — i.e. n·(n−1)/2 pairwise growth.

Symptom: Latency bottlenecks (slowest worker blocks the batch); the lead "can't steer subagents [and] subagents can't coordinate."

Mitigation: Synchronous execution as a deliberate simplification ("This simplifies coordination"); parallel tool calling to cut wall-clock time. Async would add parallelism but "adds challenges in result coordination, state consistency, and error propagation across the subagents."

⑤ Duplicated or conflicting work between workers▼

Mechanism: Vague delegation → overlapping mandates. Real Anthropic example: "One subagent explored the 2021 automotive chip crisis while 2 others duplicated work investigating current 2025 supply chains, without an effective division of labor." Subagents do not share context with each other, so they cannot self-deconflict at runtime.

Symptom: Wasted tokens, gaps in coverage, redundant or contradictory findings.

Mitigation: Detailed task descriptions with "clear task boundaries" and explicit division of labor in the orchestrator's delegation (Job 1, Section C).

⑥ Partial failure — one worker fails▼

Mechanism: In synchronous fan-out, a single failing subagent can block or corrupt the batch; cascading downstream failures occur when one agent's bad output feeds others. [SECONDARY] Galileo names "Single Agent Failures Cascading Downstream."

Symptom: Whole run stalls or returns incomplete results despite most workers succeeding.

Mitigation: Resume-from-failure rather than full restart; model-driven graceful degradation — "letting the agent know when a tool is failing and letting it adapt works surprisingly well"; deterministic retry logic + checkpoints.

⑦ Non-determinism making debugging hard▼

Mechanism: Agents make different decisions on different runs; failures are hard to reproduce. [SECONDARY] Galileo lists "Inadequate Observability and Debugging" as a core failure mode.

Symptom: Cannot reproduce or root-cause failures from logs alone.

Mitigation: "Adding full production tracing let us diagnose why agents failed and fix issues systematically." Evaluate end-state, not every step (Section E).

⑧ Deployment coordination of long-running stateful agents▼

Mechanism: Deploying a new version mid-run can disrupt agents that are hours into a stateful task.

Symptom: Running agents broken or lost on deploy.

Mitigation: "we use rainbow deployments to avoid disrupting running agents, by gradually shifting traffic from old to new versions" — keep old and new versions running simultaneously and migrate traffic gradually so in-flight agents finish on the version they started on.

Secondary cross-reference — the MAST academic taxonomy

🎓 Academic, [SECONDARY] — read carefully. Cemri et al., "Why Do Multi-Agent LLM Systems Fail?" (UC Berkeley group) identify 14 unique failure modes across 3 categories — "(i) system design issues, (ii) inter-agent misalignment, and (iii) task verification" — derived from 1600+ annotated traces across 7 MAS frameworks, with inter-annotator agreement kappa = 0.88 on a 150-trace subset. VERIFY The commonly-cited 41.8% / 36.9% / 21.3% category split was NOT confirmed on the arXiv abstract — do not state those percentages as fact. Only the 14 modes, 3 categories, 1600+ traces, and kappa = 0.88 are confirmed here.
🤔 Predict first
Going from 4 coordinating agents to 10 — how many potential pairwise interactions does that jump from, and to?
From 6 to 45. Using n·(n−1)/2: 4·3/2 = 6; 10·9/2 = 45. [SECONDARY] More than doubling the agents multiplies coordination surface ~7.5×. This super-linear growth is exactly why "rapid growth in coordination complexity" is a named failure driver.
❓ Quick check
Subagents explore the same topic and return overlapping, redundant findings. Which failure mode is this, and what fixes it?
This is mode ⑤ (duplicated/conflicting work). Because subagents don't share context, they can't self-deconflict — the fix is upstream, in the orchestrator's delegation: objectives, output formats, and "clear task boundaries."
❓ Quick check
Which is accurately stated about the Cemri et al. MAST taxonomy?
Confirmed: 14 modes, 3 categories, 1600+ traces, kappa = 0.88 — a [SECONDARY] academic source. The 41.8/36.9/21.3 split was NOT confirmed on the abstract, so don't state it as fact.
🛡️ Reliability

Reliability patterns to design around the failures

Durable resume, checkpoints, graceful degradation, external memory, tracing, end-state evals, rainbow deploys, worker limits, and the Workflow escape hatch.

🛡️ Section 5⏱ ~14 min · core · Domain 5

Designing reliability in

What you'll be able to do
  • Map each reliability pattern to the failure mode(s) it neutralizes
  • Implement durable resume via session_id + resume in the SDK
  • Bound a worker's blast radius with maxTurns, effort, and a restricted tools list
  • Know when to graduate a huge fan-out to the Workflow tool

Why before how: Section D was the disease; this is the medicine. The elegant thing about Anthropic's design is that the same handful of patterns covers most modes. Here is the mapping, then the mechanics.

1 · Durable / resumable execution

"When errors occur, we can't just restart from the beginning: restarts are expensive and frustrating for users. Instead, we built systems that can resume from where the agent was when the errors occurred." The SDK supports this concretely: "Subagents can be resumed to continue where they left off. Resumed subagents retain their full conversation history… The subagent picks up exactly where it stopped rather than starting fresh," via capturing session_id + agentId and passing resume: sessionId.

SDK — capture session, then resume (escaped) · VERIFY field names SDK-version-sensitive
// 1) capture the subagent's session id from its messages
let sessionId;
for await (const msg of query({ prompt, options })) {
  if (msg.session_id) sessionId = msg.session_id;
}

// 2) later, resume from exactly where it stopped (not from scratch)
for await (const msg of query({
  prompt: "continue the analysis",
  options: { resume: sessionId }   // retains full history
})) { /* ... */ }

2 · Checkpointing + retries + deterministic safeguards

"We combine the adaptability of AI agents built on Claude with deterministic safeguards like retry logic and regular checkpoints." The pattern name to remember: adaptive core, deterministic guardrails. The model is flexible; the scaffolding around it is boringly reliable.

3 · Model-driven graceful degradation

"We also use the model's intelligence to handle issues gracefully: for instance, letting the agent know when a tool is failing and letting it adapt works surprisingly well." Instead of hiding failures from the agent, you tell it — and let it route around the damage.

4 · External memory / context persistence

Save the plan and completed-phase summaries to memory so context-window truncation doesn't destroy strategy. SDK note: "When the main conversation compacts, subagent transcripts are unaffected. They're stored in separate files," and persist within their session. This is the direct mitigation for mode ② (context exhaustion).

5 · Full production tracing / observability

"Adding full production tracing let us diagnose why agents failed and fix issues systematically." In the SDK, subagent invocation is observable via tool_use blocks named "Agent" and a parent_tool_use_id field on messages from within a subagent. VERIFY the tool name was renamed from "Task" to "Agent" in Claude Code v2.1.63; older SDKs emit "Task".

6 · End-state (not step-by-step) evaluation + LLM-as-judge

"evaluate whether it achieved the correct final state… focus on end-state evaluation rather than turn-by-turn analysis." For complex workflows: "break evaluation into discrete checkpoints where specific state changes should have occurred, rather than attempting to validate every intermediate step." This is the answer to mode ⑦ — you can't reproduce every non-deterministic path, so you grade the destination, not the route.

7 · Rainbow deployments

Gradually shift traffic old → new to avoid disrupting in-flight stateful agents (mode ⑧). In-flight agents finish on the version they started on.

8 · Worker effort/turn limits + tool restrictions

Set maxTurns, effort, and a restricted tools list per subagent to bound cost and blast radius; e.g., a read-only analysis agent gets only ["Read","Grep","Glob"].

SDK — a least-privilege, bounded worker (escaped)
agents: {
  "read-only-analyst": {
    description: "Read-only static analysis. No writes, no shell.",
    prompt: "Analyze; never modify files.",
    tools: ["Read", "Grep", "Glob"],  // allow-list = blast-radius cap
    effort: "low",
    maxTurns: 8                            // bound cost (mode ③)
  }
}

9 · Move large fan-outs out of the conversation — the Workflow tool

"Subagents work well for a few delegated tasks per turn. For runs that coordinate dozens to hundreds of agents, use the Workflow tool, which moves the orchestration into a script the runtime executes outside the conversation context." VERIFY the Workflow tool requires the TypeScript Agent SDK v0.3.149+. This is the escape hatch when coordination overhead (mode ④) and context pressure (mode ②) would otherwise overwhelm an in-conversation orchestrator.

🧪 Connect to your own work (Claude Code & agentic workflows)▼

If you've built agentic workflows in Claude Code, you've already met these patterns in miniature: a long task that compacts mid-run is mode ②; a flaky MCP tool that the agent works around is graceful degradation; a .claude/agents/*.md worker with a tight tools allow-list is pattern #8. The leap to "at scale" is mostly about making these deliberate and instrumented rather than incidental — durable resume instead of "just rerun it," end-state evals instead of eyeballing, rainbow deploys instead of "deploy and hope nothing's mid-run."

🤔 Predict first
A 40-minute research run crashes at minute 35. Why is "resume from session" dramatically better than "rerun from the start" — beyond just time?
Because restarts are "expensive and frustrating," and because a fresh run is non-deterministic. Rerunning doesn't just cost the 35 minutes again — it may take a different path and not reproduce the partial progress at all. Resume "picks up exactly where it stopped," retaining full history, so you keep the work already done.
❓ Quick check
Which reliability pattern most directly addresses context-window exhaustion (mode ②)?
External memory: "saving its plan to Memory to persist the context" and storing phase summaries before proceeding. Truncation can't destroy strategy that lives outside the window.
❓ Quick check
You need to coordinate ~200 agents in a rerunnable orchestration whose intermediate results should stay out of the conversation. You should…
Subagents suit "a few delegated tasks per turn." For dozens–hundreds, graduate to the Workflow tool, which "moves the orchestration into a script the runtime executes outside the conversation context." VERIFY TS SDK v0.3.149+.
❓ Quick check
Why does Anthropic evaluate end-state rather than every intermediate step?
"focus on end-state evaluation rather than turn-by-turn analysis." Because runs are non-deterministic (mode ⑦), grading every step is brittle; grading the destination (with checkpoints for complex flows) is robust.

When NOT to use it

Default to the simplest thing that works. Climb the complexity ladder — single agent → fixed workflow → orchestrator-worker — only when the task forces you to. Coding, with its tight coupling and few truly parallel subtasks, is Anthropic's own example of a poor fit. The architect's instinct is suspicion of the fan-out, not enthusiasm for it.
