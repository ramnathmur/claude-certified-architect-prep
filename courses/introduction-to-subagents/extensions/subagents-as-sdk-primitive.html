🧱 The primitive

From a CLI feature to an object you declare

What a subagent is once you stop clicking /agents and start writing code: a typed record with a description, a prompt, a tool allow-list, and a handful of operational flags.

🚪 Section 1⏱ ~6 min · concept

Hook — subagents as a thing you program

What you'll be able to do
  • Explain the shift from the /agents CLI UI to subagents-as-code
  • Name the three ways a subagent can come into existence
  • Say why this is the heart of the 27% Domain 1

Why before how: in the base course you met subagents through the /agents command — a tabbed UI where you click "create," fill in a description, pick tools, and save. That's the operator's view. The architect's view is different: a subagent is a typed object you can construct, parameterize, and pass into a query() at runtime. Once you see it that way, every orchestration pattern from S-E1 becomes something you assemble in code, not something you configure by hand.

The docs make the shift explicit. Subagents can be created three ways:

  • Programmatically — the agents parameter in query() options. This is "the documented recommended approach for SDK apps."
  • Filesystem-based — Markdown files at .claude/agents/*.md with YAML frontmatter.
  • The built-in general-purpose subagent — always invocable via the Agent tool, with no definition at all.

Analogy The CLI /agents UI is like hiring through a form on a careers page; the SDK AgentDefinition is like writing a job description in code that a factory function can generate on demand — same role, but now you can stamp out a stricter variant with a more capable model whenever a task warrants it. (Author analogy, not a product fact.)

❓ Quick check
Which way of creating a subagent is the documented recommended approach for SDK applications?
For SDK apps the recommended path is programmatic — pass an agents record/dict of AgentDefinitions into query(). Filesystem files suit version-controlled team agents; general-purpose needs no definition; /agents is the CLI operator view.
🔑 Key takeaway▼

A subagent is not a UI artifact — it's a declarable object. Three doors lead to it (programmatic / filesystem / built-in general-purpose), but they all resolve to the same runtime worker. Architects reach for the programmatic door because it lets them parameterize agents at runtime.

🧱 Section 2⏱ ~11 min · core · code-heavy

The AgentDefinition object — every field

What you'll be able to do
  • Recite the required vs. optional fields of AgentDefinition
  • Write the TS and Python shapes — and avoid the camelCase-in-Python TypeError
  • Explain what description drives, and the "omit tools → inherit all" rule

A subagent is declared as an AgentDefinition: a record (TS) / dataclass (Python) holding the subagent's routing description, system prompt, tool allow-list, model, and operational flags. The orchestrator decides whether to invoke it based on the description field — so that field is doing double duty as documentation and as the router signal.

Only two fields are required: description and prompt. Everything else is optional. Here is the exact TypeScript shape, verbatim from the TS reference:

TypeScript — VERIFY field roster against the live TS reference
type AgentDefinition = {
  description: string;
  tools?: string[];
  disallowedTools?: string[];
  prompt: string;
  model?: string;
  mcpServers?: AgentMcpServerSpec[];
  skills?: string[];
  initialPrompt?: string;
  maxTurns?: number;
  background?: boolean;
  memory?: "user" | "project" | "local";
  effort?: "low" | "medium" | "high" | "xhigh" | "max" | number;
  permissionMode?: PermissionMode;
  criticalSystemReminder_EXPERIMENTAL?: string;
};

And the Python dataclass shape, verbatim from the Python reference:

Python — note the field order differs; required fields first
@dataclass
class AgentDefinition:
    description: str
    prompt: str
    tools: list[str] | None = None
    disallowedTools: list[str] | None = None
    model: str | None = None
    skills: list[str] | None = None
    memory: Literal["user", "project", "local"] | None = None
    mcpServers: list[str | dict[str, Any]] | None = None
    initialPrompt: str | None = None
    maxTurns: int | None = None
    background: bool | None = None
    effort: EffortLevel | int | None = None
    permissionMode: PermissionMode | None = None
🐍 The camelCase-in-Python gotcha (it raises a TypeError)▼

AgentDefinition field names use camelCase even in Python — disallowedTools, permissionMode, maxTurns, initialPrompt — to match the wire format shared with the TS SDK. This differs from ClaudeAgentOptions, which uses Python snake_case (disallowed_tools, permission_mode). Because AgentDefinition is a dataclass, passing a snake_case keyword raises a TypeError at construction time. So: AgentDefinition(maxTurns=5) ✅ — AgentDefinition(max_turns=5) ✗.

🤔 Predict first
You define a subagent with no tools field at all. What tool access does it get?
All of them. "If omitted, tools inherits all tools." To restrict a subagent, you must explicitly list an allow-list (or use disallowedTools as a denylist). Omitting tools is the "full access" combo — which is the opposite of least privilege.
❓ Quick check
In the Python SDK, which constructor call is correct?
AgentDefinition fields are camelCase even in Python — maxTurns, not max_turns. A snake_case keyword raises a TypeError at construction. (The snake_case style belongs to ClaudeAgentOptions, not AgentDefinition.)
📦 Section 3⏱ ~9 min · core · code-heavy

The agents parameter — programmatic vs. filesystem

What you'll be able to do
  • Write the agents type for TS and Python, and identify what the dict key means
  • State the precedence rule (programmatic vs. filesystem) and the full filesystem ladder
  • Choose programmatic vs. filesystem definitions for a given scenario

You pass programmatic subagents through one option. Its type is small but exact:

  • TypeScript Options: agents?: Record<string, AgentDefinition>
  • Python ClaudeAgentOptions: agents: dict[str, AgentDefinition] | None = None

In both, the key is the agent's name and the value is its definition. Here is the verbatim Python invocation — note "Agent" in allowed_tools (we'll see why in §4):

Python — the agents={...} query example (verbatim)
import asyncio
from claude_agent_sdk import query, ClaudeAgentOptions, AgentDefinition

async def main():
    async for message in query(
        prompt="Review the authentication module for security issues",
        options=ClaudeAgentOptions(
            # Auto-approve these tools, including Agent for subagent invocation
            allowed_tools=["Read", "Grep", "Glob", "Agent"],
            agents={
                "code-reviewer": AgentDefinition(
                    description="Expert code review specialist. Use for quality, security, and maintainability reviews.",
                    prompt="""You are a code review specialist...""",
                    tools=["Read", "Grep", "Glob"],
                    model="sonnet",
                ),
                "test-runner": AgentDefinition(
                    description="Runs and analyzes test suites. Use for test execution and coverage analysis.",
                    prompt="""You are a test execution specialist...""",
                    tools=["Bash", "Read", "Grep"],
                ),
            },
        ),
    ):
        if hasattr(message, "result"):
            print(message.result)

asyncio.run(main())

The TS equivalent uses allowedTools: [..., "Agent"] and agents: { "code-reviewer": { description, prompt, tools, model } } — the same shape, camelCase throughout.

Precedence — who wins on a name clash

The headline rule: programmatically defined agents take precedence over filesystem-based agents with the same name. Within the filesystem world there's a fuller ladder (highest → lowest):

When to use each

Use programmatic agents when…Use filesystem .claude/agents/*.md when…
Building an SDK appYou want reusable, version-controlled team agents
Agents are dynamic / runtime-parameterized (e.g. a factory returning a stricter agent on a more capable model)Prompts are complex / long
The definition is generated in codeWindows workaround: long programmatic prompts can hit the 8191-char command-line limit — use file-based instead
🤔 Predict first
A project ships a .claude/agents/code-reviewer.md, and your SDK app also passes agents={"code-reviewer": ...}. Which definition runs?
The programmatic one. "Programmatically defined agents take precedence over filesystem-based agents with the same name." The file is shadowed for that name as long as your code defines it.
❓ Quick check
In agents: Record<string, AgentDefinition>, what does the key represent?
The key is the agent's name; the value is its AgentDefinition. In filesystem terms, the key corresponds to frontmatter name / the file's identity.
🔁 The delegation model

One channel in, one channel out

How a delegation actually runs: the Agent tool fires, a fresh subagent context spins up, work happens in isolation, and only a final message comes back. This is the most-tested mechanic in Domain 1.

🛰️ Section 4⏱ ~9 min · core

Runtime delegation — the Agent (formerly Task) tool

What you'll be able to do
  • Describe how the parent delegates via the Agent tool, and what returns
  • Distinguish automatic vs. explicit invocation
  • Explain why Agent must be in allowedTools to auto-approve delegation

The main agent delegates by calling the Agent tool (the built-in subagent / Task tool). The subagent then runs in its own fresh conversation: "Intermediate tool calls and results stay inside the subagent; only its final message returns to the parent."

SVG (c) — the delegation flow

⚠️ VERIFY — the Task → Agent rename (v2.1.63): "The tool name was renamed from "Task" to "Agent" in Claude Code v2.1.63." Current SDK releases emit "Agent" in tool_use blocks but still use "Task" in the system:init tools list and in result.permission_denials[].tool_name — check both "Task" and "Agent" for compatibility. At the filesystem/CLI level, existing Task(...) references in settings and agent definitions still work as aliases. Reconfirm the exact version and emit behavior against the live docs.

Auto-approve: include Agent in allowedTools

This is the single most common cause of "Claude won't delegate." To auto-approve delegation, include Agent in allowedTools. Without it, Agent invocations fall through to the canUseTool callback, or are denied in dontAsk mode. That's why every example in §3 lists "Agent" alongside Read/Grep/Glob.

Automatic vs. explicit invocation

  • Automatic — Claude matches the task to a subagent's description and delegates on its own.
  • Explicit — you name it in the prompt: "Use the code-reviewer agent to check the authentication module" — which bypasses automatic matching.

Detection. Subagent invocations appear as tool_use blocks where name is "Agent" (or legacy "Task"); messages from inside a subagent carry a parent_tool_use_id; block.input.subagent_type names the agent.

🤔 Predict first
Your subagents are well-described but Claude keeps doing the work itself instead of delegating. Your allowedTools is ["Read","Grep","Glob"]. What's the most likely fix?
Add "Agent" to allowedTools. Without it, the delegation tool isn't auto-approved — invocations fall to the canUseTool callback or get denied in dontAsk mode. (Secondary fixes: name the subagent explicitly, or sharpen its description.)
❓ Quick check
After the v2.1.63 rename, where does the legacy name "Task" still appear in SDK output?
Current SDK releases emit "Agent" in tool_use blocks but still use "Task" in the system:init tools list and in result.permission_denials[].tool_name — so robust code checks for both.
🪢 Section 5⏱ ~8 min · core · most exam-relevant

The prompt string is the ONLY channel — "fresh but not empty"

What you'll be able to do
  • State the single-channel rule and its design consequence (self-contain every delegation)
  • List what a subagent receives vs. does not receive
  • Explain "fresh but not empty" context inheritance

This is the rule the exam tests most. "The only channel from parent to subagent is the Agent tool's prompt string, so include any file paths, error messages, or decisions the subagent needs directly in that prompt." There is no shared memory between parent and subagent within a single delegation.

But "fresh" doesn't mean "empty." The subagent's context starts with no parent history, yet it inherits its own system prompt, the Agent tool's prompt string, project CLAUDE.md (via settingSources), and tool definitions per its definition.

SVG (d) — what crosses the boundary vs. what doesn't

The subagent receivesThe subagent does NOT receive
Its own system prompt (AgentDefinition.prompt) + the Agent tool's promptThe parent's conversation history or tool results
Project CLAUDE.md (loaded via settingSources)Preloaded skill content, unless listed in AgentDefinition.skills
Tool definitions (inherited from parent, or the subset in tools)The parent's system prompt

Analogy Delegating to a subagent is like emailing a contractor who has never seen your project: they know their own trade (system prompt) and the company handbook (CLAUDE.md), but they did not sit in your earlier meetings. If a decision was made in a meeting they missed, you must restate it in the email — or it never reaches them. (Author analogy, not a product fact.)

🧰 Design rule — self-contain every delegation▼

Pass concrete file paths, exact error strings, and prior decisions in the invocation prompt. If a rule must reach the subagent (e.g. "ignore vendor/"), restate it in the delegation prompt — the subagent does not see the parent's earlier context.

↩️ Return-value caveat: the parent receives the subagent's final message verbatim as the Agent tool result, but may summarize it in its own response. To preserve subagent output verbatim in the user-facing answer, instruct that in the main query() prompt / systemPrompt.
❓ Quick check
A subagent needs to skip vendor/, a rule established earlier in the parent's conversation. How does it learn the rule?
The prompt string is the only channel in. The subagent doesn't see the parent's history or system prompt, so any rule established earlier must be restated directly in the delegation prompt.
🛡️ Section 6⏱ ~7 min · concept

Context isolation — why subagents protect the main thread

What you'll be able to do
  • Explain how isolated context windows save the main thread's budget
  • Recognize the "filtering through large datasets" fit signal
  • Recall the persistence detail (separate transcript files, immune to main compaction)

The single channel out is also a context-management feature. Because "intermediate tool calls and results stay inside the subagent; only its final message returns," a research-assistant can "explore dozens of files without any of that content accumulating in the main conversation. The parent receives a concise summary, not every file the subagent read."

The engineering framing: "Subagents use their own isolated context windows, and only send relevant information back to the orchestrator" — ideal for "tasks requiring filtering through large datasets where most information won't be useful." This ties to the agent loop: gather context → take action → verify work → repeat. (This connects to context management — see C-E2 if you've covered it.)

💾 Persistence detail▼

Subagent transcripts are stored in separate files and are unaffected by main-conversation compaction; they persist within their session and are cleaned up per cleanupPeriodDays (default 30 days). Subagents support auto-compaction at ~95% capacity (override via CLAUDE_AUTOCOMPACT_PCT_OVERRIDE).

When to reach for it. Use a subagent "when a side task would flood your main conversation with search results, logs, or file contents you won't reference again" — e.g. "run the test suite and report only the failing tests with their error messages."

❓ Quick check
What is the primary context-management benefit of delegating a file-heavy search to a subagent?
Isolation means the search results/files accumulate in the subagent's own window, and only a concise summary returns — protecting the main thread's budget. (Subagents can't spawn subagents, and their transcripts are immune to main-conversation compaction.)

🚫 Section 7⏱ ~11 min · core

Guardrails — no nesting, least privilege, model, permissions, turns

What you'll be able to do
  • State the no-nesting rule and what to do instead
  • Apply least-privilege tool allow-lists (and the denylist-first ordering)
  • Reason about per-subagent model selection, permission-mode precedence, and maxTurns

The core rule: no nested subagents

"Subagents cannot spawn their own subagents. Don't include Agent in a subagent's tools array." If your workflow needs nested delegation, "use Skills or chain subagents from the main conversation." This is also why plan mode uses the read-only Plan subagent — "This prevents infinite nesting."

SVG (e) — the no-nesting rule

Note: Agent(agent_type) allow-listing only matters for an agent running as the main thread via claude --agent; it "has no effect in subagent definitions."

Least privilege — tool allow-lists

Omit tools → inherits all; specify tools → allow-list only. Ordering matters: disallowedTools is a denylist applied first, then tools resolves against the remainder; a tool in both is removed.

SVG (f) — least-privilege tool combos

🔒 Never-available tools: some UI/session tools are never available to subagents even if listed — Agent, AskUserQuestion, EnterPlanMode, ExitPlanMode (unless permissionMode: plan), ScheduleWakeup, WaitForMcpServers.

Per-subagent model selection

Set model to sonnet/opus/haiku/inherit or a full ID; it defaults to inherit if omitted. Resolution order: CLAUDE_CODE_SUBAGENT_MODEL env var → per-invocation model param → definition model → main model. Cost lever: route routine work to Haiku.

Permission-mode precedence

A subagent may override permissionMode, except a parent in bypassPermissions or acceptEdits takes precedence and cannot be overridden; under parent auto mode the subagent inherits auto and its frontmatter permissionMode is ignored.

maxTurns and timeout

maxTurns caps agentic turns before the subagent stops. VERIFY: no explicit wall-clock timeout field is documented for AgentDefinition — verify timeout behavior against the live docs.

🤔 Predict first
A teammate wants a "manager" subagent that spawns its own team of worker subagents. What does the source say, and what's the right alternative?
Not allowed. "Subagents cannot spawn their own subagents" — never put Agent in a subagent's tools. The right alternatives: use Skills for nested behavior, or chain subagents from the main conversation. For large fan-out, graduate to the Workflow tool (§9).
❓ Quick check
When both disallowedTools and tools are set, how are they resolved?
The denylist is applied first, then the allow-list resolves against what's left. A tool named in both ends up removed.

📈 Section 9⏱ ~10 min · core

The Workflow-tool graduation — who holds the plan

What you'll be able to do
  • State when a few delegations should graduate to a Workflow
  • Apply the "who holds the plan" distinction (Claude's context vs. script variables)
  • Recall the caps and the version gates

"A dynamic workflow is a JavaScript script that orchestrates subagents at scale. Claude writes the script for the task you describe, and a runtime executes it in the background while your session stays responsive." The orchestration loop, branching, and intermediate results live in script variables, not Claude's context — "so Claude's context holds only the final answer."

When to graduate. "Subagents work well for a few delegated tasks per turn. For runs that coordinate dozens to hundreds of agents, use the Workflow tool, which moves the orchestration into a script the runtime executes outside the conversation context." Reach for it for a codebase-wide bug sweep, a 500-file migration, cross-checked research, or multi-angle planning.

Patterns it unlocks. A workflow can have "independent agents adversarially review each other's findings before they're reported, or draft a plan from several angles and weigh them against each other." The bundled /deep-research workflow "fans out web searches across several angles, fetches and cross-checks the sources it finds, votes on each claim" — fan-out + cross-check barrier + voting. This maps to the orchestrator-workers and parallelization (sectioning/voting) patterns from S-E1.

⚠️ VERIFY — caps & version gates: the Workflow runtime caps at ≤16 concurrent agents (fewer on limited cores) and 1,000 agents total per run. The Workflow tool is in the TypeScript Agent SDK v0.3.149+; the dynamic-workflows feature in the CLI is a research preview requiring Claude Code v2.1.154+ (all paid plans). Both version numbers are fast-moving — reverify against the live docs. (The 16/1,000 caps are workflow-runtime limits; whether any per-turn cap applies to ordinary subagent parallelism outside workflows is itself VERIFY.)
📁 Where workflows live & how they run▼

Saved workflows live in .claude/workflows/ (project) or ~/.claude/workflows/; project wins on a name clash; input is passed via the args global. Runtime constraints: no mid-run user input; no direct filesystem/shell access from the script itself (agents do the I/O, the script coordinates). Subagents a workflow spawns always run in acceptEdits mode and inherit the tool allowlist regardless of session mode. To auto-approve, include Workflow in allowedTools.

🤔 Predict first
You need to migrate 500 files with a coordinated, rerunnable orchestration that doesn't blow up the conversation's context. Subagents or a Workflow — and why?
A Workflow. Subagents suit "a few delegated tasks per turn." A 500-file migration coordinates far more agents than one conversation can hold, and you want the orchestration "codified as a script you can read and rerun." The script holds the plan and intermediate results in variables — Claude's context holds only the final answer.
❓ Quick check
The decisive distinction between subagents and workflows is…
The defining line is where the orchestration lives: Claude decides turn-by-turn with results in its context (subagents), vs. a script decides with results in variables, run outside the conversation (workflows).

Design heuristics — subagent vs. a single longer prompt

What you'll be able to do
  • Decide when a subagent beats a single longer prompt — and when to stay in the main conversation
  • Apply "one responsibility per subagent"
  • Write a description that routes correctly
Use a subagent when…Stay in the main conversation when…
The task produces verbose output you don't need in your main contextThe task needs frequent back-and-forth
You want tool restrictions / permissions enforcedMultiple phases share significant context (planning → implementation → testing)
The work is self-contained and can return a summaryIt's a quick targeted change, or latency matters (subagents start fresh and need time to gather context)

For a quick question about existing context, prefer /btw over a subagent — spinning up a fresh worker just to answer something already in the conversation is wasted latency.

One responsibility per subagent

"Design focused subagents: each subagent should excel at one specific task." Define a custom subagent "when you keep spawning the same kind of worker with the same instructions."

Description-writing for correct routing

"Claude uses each subagent's description to decide when to delegate tasks… write a clear description so Claude knows when to use it." Add "use proactively" to encourage delegation; write detailed descriptions so Claude matches tasks correctly. If Claude won't delegate: (1) include Agent in allowedTools; (2) name the subagent explicitly; (3) sharpen the description.

🔑 The description is a router classifier▼

Recall from S-E1: routing = "classify an input and direct it to a specialized followup." A subagent's description is the classifier the orchestrator uses to route. Vague description → poor routing → Claude does the work itself. A sharp, "use proactively" description is the cheapest reliability lever you have.

❓ Quick check
Which scenario is the WEAKEST fit for a subagent?
Stay in the main conversation when multiple phases share significant context — the single-channel isolation would force you to re-pass everything. The other three are textbook subagent fits (verbose output, summary return, enforced tool restriction).

🧠 Final Knowledge Check (10 questions)

0 / 10
Q1
Which two fields are REQUIRED on an AgentDefinition?
Only description and prompt are required; everything else is optional. (name is the dict key in the agents record, not a field of the object itself.)
Q2
In Python, AgentDefinition(max_turns=5) will…
AgentDefinition fields are camelCase even in Python (to match the wire format). Because it's a dataclass, a snake_case keyword raises a TypeError. Use maxTurns=5.
Q3
If you omit the tools field on a subagent, it…
"If omitted, tools inherits all tools." To enforce least privilege you must list an explicit allow-list (or use disallowedTools).
Q4
What is the ONLY channel from a parent into a subagent's fresh context?
"The only channel from parent to subagent is the Agent tool's prompt string." Include file paths, errors, and decisions directly in it — there's no shared memory.
Q5
Which does a subagent DOES NOT receive?
The parent's history/results and system prompt never cross. The subagent does get its own prompt, the Agent tool's prompt, CLAUDE.md, and tool defs — "fresh but not empty."
Q6
To AUTO-APPROVE delegation so Claude will actually invoke subagents, you must…
Include Agent in the main allowedTools. (Putting Agent in a subagent's tools is forbidden — that would enable nesting.)
Q7
Which is TRUE about subagent nesting?
"Subagents cannot spawn their own subagents." Never put Agent in a subagent's tools. For nested behavior, use Skills or chain subagents from the main conversation.
Q8
Programmatic vs. filesystem agents with the same name — which wins?
"Programmatically defined agents take precedence over filesystem-based agents with the same name." The filesystem ladder (managed > --agents > project > user > plugin) sits below the programmatic layer.
Q9
In a .claude/agents/*.md file, the Markdown body maps to which SDK field?
The body = the system prompt = SDK prompt. Frontmatter name → the dict key; description → description. hooks/isolation/color are filesystem-only.
Q10
You must coordinate hundreds of agents in a rerunnable orchestration whose intermediate results stay out of the conversation. You should use…
Subagents suit "a few delegated tasks per turn." For dozens–hundreds, graduate to the Workflow tool: the script decides what runs next, results live in variables, and it runs outside the conversation (caps: ≤16 concurrent, 1,000 per run).
