🔁 The defining line

Dynamic decomposition is the whole pattern

A 90-second recap of what orchestrator-worker is — then straight into why it behaves the way it does. We move fast here; the depth lives in B–F.

🔁 Section 1⏱ ~6 min · recap + mechanism

The line that defines the pattern

By the end you can
  • State the one feature that separates orchestrator-worker from static parallelization — and why it matters for everything downstream
  • Map the pattern's three verbs onto the rest of this lesson

Why before how. Anthropic's definition is exact: in the orchestrator-workers workflow, "a central LLM dynamically breaks down tasks, delegates them to worker LLMs, and synthesizes their results." (source: Building Effective Agents) Three verbs — break down → delegate → synthesize — and the entire rest of this lesson is "what goes wrong with each verb under load, and how Anthropic designed around it." Hold that scaffold; we'll return to it in every section.

The defining feature is the word dynamically. Static parallelization comes in two named flavors: "Sectioning: Breaking a task into independent subtasks run in parallel. Voting: Running the same task multiple times to get diverse outputs." (source: Building Effective Agents) In both, the fan-out is fixed in code. In orchestrator-worker, the central LLM decides at runtime what the subtasks even are — it "is well-suited for complex tasks where you can't predict the subtasks needed." That runtime decision is not a detail; it is the source of both the power (adapts to each input) and every failure mode in Section D (the LLM can decide wrong).

Analogy Static parallelization is a printed assembly line — every station bolted down before the first part arrives. Orchestrator-worker is a newsroom editor who reads the story that just broke, then decides on the spot how many reporters to assign and to what angles. Same "many workers in parallel" silhouette; a completely different decision-maker — and the editor can mis-assign. (Author analogy, not a product fact.)

🤔 Predict first
A function always spawns exactly 3 workers — one per fixed report section — regardless of input. Orchestrator-worker, or not? And which failure modes can it not have?
Not orchestrator-worker — that's static sectioning. The subtasks are fixed in code, not "determined by the orchestrator based on the specific input." Because nothing is decided at runtime, it structurally cannot suffer the LLM-decision failures: duplicated/conflicting work from vague delegation (#5) or wrong effort scaling. It can still hit token cost (#3) and partial failure (#6), but the decomposition itself is immune — which is exactly why "fall back to a fixed workflow" is a real mitigation in Section F.

break down → delegate → synthesize, decided at runtime by a central LLM. §B = the cost of doing this; §C = doing each verb well; §D = how each verb fails; §E = designing the failures out; §F = when to not do it at all.

📈 Section 2⏱ ~14 min · core · simulator

The 15× economics, as a thing you operate

By the end you can
  • Explain the three rationales for multi-agent and the single mechanism underneath all three
  • Estimate total token spend from a fan-out and an effort level using the verified multipliers
  • Decide, with numbers, whether a task clears the value × parallelizability gate

Anthropic productionized orchestrator-workers in their Research feature: "a lead agent coordinates the process while delegating to specialized subagents that operate in parallel." (source: multi-agent-research-system) Three rationales justify the spend — but notice they collapse into one mechanism.

  1. Context isolation / compression. Subagents "operating in parallel with their own context windows… condensing the most important tokens for the lead research agent." The SDK: "Intermediate tool calls and results stay inside the subagent; only its final message returns to the parent." (sources: multi-agent-research-system; SDK subagents)
  2. Separation of concerns. "distinct tools, prompts, and exploration trajectories—which reduces path dependency."
  3. Parallelism / speed. "cut research time by up to 90% for complex queries."

The mechanism underneath: all three are ways of spending more tokens productively, in parallel. That reframing matters because of the headline finding — "token usage by itself explains 80% of the variance" in performance VERIFY (three factors — token usage, tool calls, model choice — explain 95% VERIFY). So the multi-agent win is, almost embarrassingly, a token story: more parallel tokens → more performance, until the task can't absorb more. That single sentence predicts the entire shape of the simulator below.

🤔 Predict before you touch the simulator
Before sliding anything: a multi-agent system uses "about 15× more tokens than chats" while single agents use "about 4×." If you spawn 6 workers each doing roughly chat-scale work, will your spend be closer to 15× or far past it? Predict, then drive the sliders and check.
Far past 15× — the 15× is a measured average, not a ceiling. The ~15× figure is Anthropic's observed average for their multi-agent system VERIFY. Spend scales roughly with (number of workers × work-per-worker) plus the lead's own overhead, so a heavy fan-out blows past the average quickly. The simulator makes the lead overhead + per-worker cost explicit so you can see where 15× lands — and how fast you leave it behind.

🔬 What the simulator is actually modeling (and what it isn't)▼

Cost unit = a chat (1×). The lead agent carries a fixed overhead (~4× a chat — it's an agent itself, planning and synthesizing). Each worker adds effort/8 × ~2.4× so that 4 medium workers + lead ≈ the ~15× the article reports VERIFY. Value-per-token rises with value and parallelizability but is dragged down as raw spend climbs — capturing the real lesson that more workers stop paying off once the task can't absorb parallel token spend. The inject-failure button removes one worker's contribution but keeps its spend, because in synchronous fan-out you've already paid for the tokens the failed worker burned (failure mode #6). What it is not: a pricing calculator. Real costs depend on models, caching, and tool latency. It exists to build the shape in your head, not a quote.

❓ Mechanism check
In the simulator, injecting a worker failure keeps the cost roughly the same but lowers value-per-token. Why is that the faithful model of synchronous fan-out?
In synchronous fan-out the lead "waits for each set of subagents to complete." A worker that fails late has already consumed its tool calls and tokens — the spend is sunk, the value isn't realized. That's failure mode #6 (partial failure) showing up directly in the economics, and why resume-from-failure beats full restart.
💡 The exam-relevant reframing▼

"Multi-agent systems excel at valuable tasks that involve heavy parallelization, information that exceeds single context windows, and interfacing with numerous complex tools." Read that as a gate: value × parallelizability must justify ~15× the spend. The simulator's verdict is just that gate computed live. If you can articulate why a task fails the gate, you've understood the section.

Job 1 — Decomposition + detailed delegation

"Each subagent needs an objective, an output format, guidance on the tools and sources to use, and clear task boundaries. Without detailed task descriptions, agents duplicate work, leave gaps, or fail to find necessary information." Vague instructions like "research the semiconductor shortage" caused subagents to "perform the exact same searches as other agents." (source: multi-agent-research-system) Detailed delegation is not politeness — it is the direct mechanism that prevents failure mode #5.

Job 2 — Effort heuristics scaled to complexity

"Agents struggle to judge appropriate effort… so we embedded scaling rules in the prompts." The explicit ladder — simple fact-finding: 1 agent, 3–10 calls; comparisons: 2–4 subagents, 10–15 calls each; complex research: >10 subagents with divided roles VERIFY counts. The SDK makes effort machine-settable: effort accepts 'low'|'medium'|'high'|'xhigh'|'max' or a number VERIFY, and maxTurns caps agentic turns.

SDK — worker scaled for a "direct comparison" task (escaped) · VERIFY enum, SDK-version-sensitive

  "comparison-worker": {
    description: "Compares two options across given dimensions.",
    prompt: "Compare EXACTLY the two named items. Output a table...",
    tools: ["Read", "Grep", "Glob"],   // restrict blast radius

  }
}

Job 3 — Synthesis (at a compression boundary)

The lead condenses findings — and crucially the compression already happened at the worker boundary: subagents return only "the most important tokens for the lead research agent." This is the mechanism that lets the lead synthesize without holding every worker's raw transcript — the same isolation the simulator's cost model depends on.

Job 4 — Citation / verification as its own stage

A dedicated "CitationAgent processes the documents and research report to identify specific locations for citations." Verification is architecturally separated from research: the agents that find are not the agent that checks — separation of concerns doubling as a reliability control.

❓ Spot the design intent
Why does Anthropic make the CitationAgent a separate stage rather than asking each research subagent to cite its own findings inline?
Isolating verification into its own role applies separation of concerns and creates a dedicated reliability checkpoint — the verifier reviews "the documents and research report" with one job, rather than each researcher self-grading mid-exploration.
🤔 Apply the heuristic
A user asks: "Compare the battery life of the iPhone 16 and Pixel 9." How many subagents and roughly how many tool calls each should a well-tuned orchestrator allocate — and why not more?
It's a direct comparison → ~2–4 subagents, 10–15 calls each VERIFY. Two clean options, bounded scope — one worker per device (plus maybe a synthesis pass) is plenty. Spawning >10 subagents here would burn the 15× spend with no parallel value to absorb it (fails the gate) and invites duplicated work (#5). The heuristic exists precisely to stop the orchestrator from over-fanning a simple task.
🔑 Key takeaway▼

The orchestrator's four jobs each defuse a specific failure: detailed delegation → no duplicated work (#5); effort scaling → no cost blowup on simple tasks (#3); compression at the boundary → no context exhaustion in the lead (#2); separate verification → catches errors before they compound (#1). Doing the job well is reliability engineering.

The coordination-cost curve — why "more agents" is not free

Anthropic names "a rapid growth in coordination complexity" as a defining property of multi-agent systems, and runs subagents synchronously — "the entire system can be blocked while waiting for a single subagent to finish." The secondary source quantifies the interaction explosion: "2 agents = 1 potential interaction; 4 agents = 6; 10 agents = 45." [SECONDARY] That is the pairwise count n·(n−1)/2 — quadratic, not linear.

🤔 Predict the curve
If 4 agents create 6 potential interactions, predict the count for 8 agents — then explain why doubling the agents more-than-doubles the coordination surface.
8 agents → 28 interactions (8·7/2). Doubling agents (4→8) takes interactions from 6 to 28 — nearly 5×, not 2×. Because each new worker can interact with every existing worker, the surface grows with the square of n. This is exactly why Anthropic avoids inter-agent coordination at runtime (synchronous execution + heavy upfront delegation) instead of letting workers negotiate among themselves.

The academic cross-reference (flagged unverified)

📚 [SECONDARY · academic] Cemri et al., "Why Do Multi-Agent LLM Systems Fail?" (UC Berkeley) identify 14 failure modes across 3 categories — "(i) system design issues, (ii) inter-agent misalignment, (iii) task verification" — from 1600+ annotated traces across 7 frameworks, κ=0.88 on a 150-trace subset. The design: A team builds a multi-agent researcher. The lead agent's delegation prompt to every subagent is a single line: "Research the semiconductor shortage." It spawns 12 subagents for every query regardless of complexity, runs them synchronously, and on any subagent error it restarts the entire run from scratch. 🚫 The compounding insight — why this section is the heart▼

"Agents are stateful and errors compound." A single bad runtime decision — vague delegation, wrong effort, a failed worker — doesn't stay local; it feeds the next step, which feeds the next. "Without effective mitigations, minor system failures can be catastrophic for agents." That's why §E's reliability patterns aren't optional polish — they're the only thing standing between a small error and a catastrophic run.

Worked example — resume beats restart (cost, not just convenience)

"When errors occur, we can't just restart from the beginning: restarts are expensive and frustrating." Tie this to §B: a worker that fails 12 tool-calls deep has already burned those tokens. A full restart pays for them again plus the new run. Resume-from-failure "picks up exactly where it stopped" — you pay once. In a 15× system, restart-on-failure can silently double or triple the real bill; durable execution is a cost control as much as a UX one.

❓ Map it
Your stateful agents are sometimes broken mid-run when you ship a new version. Which pattern directly fixes this — and which failure mode is it?
Rainbow deployments "gradually shift traffic from old to new versions," letting in-flight stateful agents finish on the version they started — the exact fix for #8.
🔬 Why end-state evals, not step-by-step▼

Because the system is non-deterministic (#7), "agents make different decisions on different runs." Grading every intermediate step punishes valid-but-different paths and is brittle. Anthropic instead evaluates "whether it achieved the correct final state" — and for complex flows, "discrete checkpoints where specific state changes should have occurred." This is the reliability-engineering version of "test behavior, not implementation."

🚦 The gate

When NOT to use orchestrator-worker at scale

The mark of an architect is refusing the fancy pattern when it doesn't earn its cost. Four disqualifiers — and the default-to-simplest rule that governs them all.

🚦 Section 6⏱ ~8 min · judgment

The economics gate, and when to refuse

By the end you can
  • State the four conditions under which multi-agent is the wrong choice
  • Apply the value × parallelizability gate to a concrete task
  • Name the fallbacks: single agent, or a fixed workflow

Four disqualifiers, each source-grounded:

  1. Fails the economics gate. Multi-agent costs ~15× VERIFY. Only use it for "valuable tasks that involve heavy parallelization, information that exceeds single context windows, and interfacing with numerous complex tools." Not valuable and parallelizable → don't.
  2. Needs shared state / tight coordination. "domains that require all agents to share the same context or involve many dependencies between agents are not a good fit… most coding tasks involve fewer truly parallelizable tasks than research." Subagents don't share context with each other, so tightly-coupled work suffers.
  3. Low-latency needs. Synchronous fan-out means the system "can be blocked while waiting for a single subagent" — bad for latency-sensitive use.
  4. The default-to-simplest principle. "find the simplest solution possible, and only increasing complexity when needed… Agentic systems often trade latency and cost for better task performance." Fall back to a single agent or a fixed workflow when the machinery isn't earning its cost.
🤔 Architect's judgment call
A user wants an agent to refactor a tightly-coupled module: every change depends on every other change. Multi-agent at scale — yes or no, and why?
No. It fails disqualifier #2: tightly-coupled work needs shared state, and subagents don't share context with each other. Anthropic explicitly notes "most coding tasks involve fewer truly parallelizable tasks than research." You'd pay ~15× for an architecture whose core advantage — independent parallel exploration — the task can't use. Fall back to a single agent (or a fixed workflow if the steps are predictable).
❓ Final judgment check
Which task is the best fit for orchestrator-worker at scale?
That last option is exactly Anthropic's sweet spot: "valuable tasks that involve heavy parallelization, information that exceeds single context windows, and interfacing with numerous complex tools." The others fail the gate on coupling, latency, or value respectively.
🔑 The whole lesson in one line▼

Orchestrator-worker spends ~15× the tokens to buy parallel exploration and isolated context. It pays only when the task is valuable AND parallelizable; it breaks in eight characteristic ways because errors compound in stateful chains; and the production patterns — durable resume, memory, tracing, end-state evals, rainbow deploys, worker limits, the Workflow escape hatch — exist to neutralize those specific failures. When the gate fails, the architect's move is to not use it.
