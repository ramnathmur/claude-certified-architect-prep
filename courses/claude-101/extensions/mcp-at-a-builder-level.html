MCP at a Builder Level

The Model Context Protocol is the wiring standard that lets any AI app talk to any tool, data source, or workflow through one shared contract. This lesson drops below the "USB-C for AI" headline into the actual mechanics — host / client / server roles, the three server primitives, transports, and the JSON-RPC handshake an architect has to reason about.

① Why it exists

Why MCP Exists — the M×N Problem

Before the standard: every (AI app × data source) pair was a bespoke integration. MCP collapses that.

🔌 Section 1⏱ ~11 minDomain 4 · Domain 1

The integration explosion MCP was built to kill

Learning objectives
  • State the M×N integration problem and how MCP collapses it to M+N
  • Define MCP and recall its origin (open standard, Anthropic, Nov 2024)
  • Draw the scope boundary: MCP standardizes context exchange, not the model's reasoning

Why before how

Imagine you have M AI applications (Claude, ChatGPT, an IDE, a custom agent) and N systems they each want to reach (GitHub, a database, a search engine, a calendar). Without a shared standard, every app needs a bespoke integration for every system — that's M×N custom connectors, and each one is built, tested, and maintained separately. Add one new app and you owe N new integrations; add one new data source and you owe M. The cost explodes multiplicatively.

Anthropic framed the pain directly: "Even the most sophisticated models are constrained by their isolation from data—trapped behind information silos and legacy systems." (source: Anthropic announcement) MCP is the answer: "a universal, open standard for connecting AI systems with data sources, replacing fragmented integrations with a single protocol." (source: Anthropic announcement) Each app implements MCP once; each data source implements MCP once. M×N collapses to M+N.

What MCP actually is

"MCP (Model Context Protocol) is an open-source standard for connecting AI applications to external systems." (source: modelcontextprotocol.io/introduction)

It connects three categories of thing — and keep these in mind, because they map directly onto the three server primitives in Section 3: "AI applications like Claude or ChatGPT can connect to data sources (e.g. local files, databases), tools (e.g. search engines, calculators) and workflows (e.g. specialized prompts)." (source: modelcontextprotocol.io/introduction)

Analogy — this one is the docs' own analogy. "Think of MCP like a USB-C port for AI applications. Just as USB-C provides a standardized way to connect electronic devices, MCP provides a standardized way to connect AI applications to external systems." (source: modelcontextprotocol.io/introduction) Before USB-C, every device had its own charger and cable; after it, one port fits everything. MCP is that port for AI integrations.

Origin and scope

MCP was announced by Anthropic on November 25, 2024 as "a new standard for connecting AI assistants to the systems where data lives, including content repositories, business tools, and development environments." Anthropic committed to "building MCP as a collaborative, open-source project and ecosystem." (source: Anthropic announcement) It is now "supported across a wide range of clients and servers," including Claude, ChatGPT, Visual Studio Code, and Cursor. (source: modelcontextprotocol.io/introduction)

🔑 Scope boundary — the testable nuance▼

"MCP focuses solely on the protocol for context exchange—it does not dictate how AI applications use LLMs or manage the provided context." (source: modelcontextprotocol.io/architecture) MCP standardizes the exchange — how context moves between an app and a system — not the model's reasoning, prompting, or context management. Those live above the protocol. Don't conflate "MCP" with "agent design."

🤔 Predict first
You have 5 AI apps and 8 data sources. Without a standard, roughly how many integrations? With MCP?
Without: 5 × 8 = 40 bespoke connectors. With MCP: 5 + 8 = 13 (each app speaks MCP once, each source speaks MCP once). That collapse from multiplicative to additive is the whole reason the standard exists — "replacing fragmented integrations with a single protocol." (source: Anthropic announcement)
❓ Quick check
Which statement correctly describes MCP's scope?
"MCP focuses solely on the protocol for context exchange—it does not dictate how AI applications use LLMs or manage the provided context." It's an open standard, and it packages the same tool-use loop rather than replacing it.
🔑 Key takeaways▼
  • MCP turns the M×N integration explosion into M+N: each side implements the protocol once.
  • It's "an open-source standard for connecting AI applications to external systems" — the docs' own analogy is a USB-C port for AI.
  • Open standard, announced by Anthropic, Nov 25 2024; broad client/server ecosystem.
  • Scope: standardizes context exchange, not the model's reasoning.
② Architecture

Host, Client, Server — and One Client per Server

The role distinctions are heavily tested. The one-client-per-server rule is the single most testable architecture fact.

🏛️ Section 2⏱ ~14 minDomain 4

The three participants and the connection rule

Learning objectives
  • Define host, client, and server, and explain how they relate
  • State and apply the one-client-per-server rule
  • Distinguish the data layer (inner) from the transport layer (outer)

The three participants

MCP "follows a client-server architecture where an MCP host — an AI application like Claude Code or Claude Desktop — establishes connections to one or more MCP servers." (source: modelcontextprotocol.io/architecture) The verbatim definitions:

ParticipantVerbatim definitionPlain English
MCP Host"The AI application that coordinates and manages one or multiple MCP clients."The app the user actually interacts with (VS Code, Claude Desktop, Claude Code).
MCP Client"A component that maintains a connection to an MCP server and obtains context from an MCP server for the MCP host to use."A protocol-level connector the host spins up internally — one per server.
MCP Server"A program that provides context to MCP clients."The program exposing tools/resources/prompts; local or remote.

Note the framing: "the host is the application users interact with, while clients are the protocol-level components that enable server connections... Each client handles one direct communication with one server." (source: modelcontextprotocol.io/client-concepts)

The one-client-per-server rule

This is the single most testable architecture fact, so internalize it precisely. The MCP host "accomplishes this by creating one MCP client for each MCP server. Each MCP client maintains a dedicated connection with its corresponding MCP server." (source: modelcontextprotocol.io/architecture) A host with three servers has three clients. There is no client that multiplexes across servers, and no server shared by two clients of the same host.

The worked example — VS Code + Sentry

"Visual Studio Code acts as an MCP host. When Visual Studio Code establishes a connection to an MCP server, such as the Sentry MCP server, the Visual Studio Code runtime instantiates an MCP client object that maintains the connection to the Sentry MCP server. When Visual Studio Code subsequently connects to another MCP server... the Visual Studio Code runtime instantiates an additional MCP client object." (source: modelcontextprotocol.io/architecture)
⚠️ The trap: two connections to one remote server▼

"Local MCP servers that use the STDIO transport typically serve a single MCP client, whereas remote MCP servers that use the Streamable HTTP transport will typically serve many MCP clients." But from a single host's perspective, a remote server reached over two connections is still served by two distinct clients — "Client3 and Client4 both connect to remote Server C" in the architecture diagram. The rule holds: each connection has its own client. (source: modelcontextprotocol.io/architecture) And remember: "MCP server refers to the program that serves context data, regardless of where it runs. MCP servers can execute locally or remotely." Server is a role, not a location.

Two layers: data (inner) vs transport (outer)

LayerWhat it defines (verbatim)
Data layer (inner)"Defines the JSON-RPC based protocol for client-server communication, including lifecycle management, and core primitives, such as tools, resources, prompts and notifications."
Transport layer (outer)"Defines the communication mechanisms and channels that enable data exchange between clients and servers, including transport-specific connection establishment, message framing, and authorization."

"Conceptually the data layer is the inner layer, while the transport layer is the outer layer." (source: modelcontextprotocol.io/architecture) The data layer is what you say (JSON-RPC messages, primitives, lifecycle); the transport layer is how the bytes travel (stdio or HTTP — Section 5). The same data-layer messages ride over either transport unchanged.

❓ Quick check
A single VS Code host connects to the Sentry server and the GitHub server. How many MCP clients exist?
The host "creat[es] one MCP client for each MCP server," each on a dedicated connection. Two servers ⇒ two clients. No multiplexing, no sharing.
🔑 Key takeaways▼
  • Host = the AI app; client = the per-connection protocol component; server = the program providing context.
  • One client per server — a dedicated 1:1 connection. N servers ⇒ N clients. (Most testable fact.)
  • "Server" is a role, not a location — local or remote.
  • Data layer (inner) = JSON-RPC, primitives, lifecycle; transport layer (outer) = how bytes travel + authorization.
③ Server primitives

Tools, Resources, Prompts — the M-A-U Split

The "who controls it" distinction is the highest-yield exam fact in all of Domain 4.

🧰 Section 3⏱ ~16 minDomain 4

Three building blocks, three controllers

Learning objectives
  • Name the three server primitives and what each does
  • Master the M-A-U control split — who controls Tools, Resources, and Prompts
  • Recall how resources are addressed (URIs) and how prompts surface (slash commands)

The control split — memorize this first

"Servers provide functionality through three building blocks." (source: modelcontextprotocol.io/server-concepts) The exam will test who controls each one more than anything else, so lead with it:

PrimitiveWhat it is (verbatim)Who controls it
Tools"Functions that your LLM can actively call, and decides when to use them based on user requests."Model
Resources"Passive data sources that provide read-only access to information for context, such as file contents, database schemas, or API documentation."Application
Prompts"Pre-built instruction templates that tell the model to work with specific tools and resources."User
🎯 The M-A-U mnemonic (highest-yield fact)▼

Tools = Model-controlled · Resources = Application-controlled · Prompts = User-controlled. Read top-to-bottom: M-A-U. The model decides when to call a tool; the application decides which resources to pull into context; the user explicitly invokes a prompt. "This three-way control split is the most heavily tested MCP fact." (source: modelcontextprotocol.io/server-concepts)

3.1 Tools — model-controlled

  • "Tools enable AI models to perform actions. Each tool defines a specific operation with typed inputs and outputs. The model requests tool execution based on context."
  • "Tools are schema-defined interfaces that LLMs can invoke. MCP uses JSON Schema for validation."
  • "Tools are model-controlled, meaning AI models can discover and invoke them automatically. However, MCP emphasizes human oversight." Tools "may require user consent prior to execution."
  • Methods: tools/list ("Discover available tools") and tools/call ("Execute a specific tool"). (source: modelcontextprotocol.io/server-concepts)

3.2 Resources — application-controlled

"Resources provide structured access to information that the AI application can retrieve and provide to models as context." The key distinction: the application, not the model, decides whether/how to include a resource — "Resources are application-driven."

  • URIs. "Each resource has a unique URI (e.g., file:///path/to/document.md) and declares its MIME type."
  • Direct resources — fixed URIs, e.g. calendar://events/2024.
  • Resource templates — dynamic URIs with parameters, e.g. travel://activities/{city}/{category}.
  • Methods: resources/list, resources/templates/list, resources/read, resources/subscribe. (source: modelcontextprotocol.io/server-concepts)

3.3 Prompts — user-controlled

"Prompts provide reusable templates... parameterized prompts for a domain." Crucially, "They are user-controlled, requiring explicit invocation rather than automatic triggering." They surface as "Slash commands (typing '/' to see available prompts like /plan-vacation), Command palettes... Dedicated UI buttons... Context menus."

  • Methods: prompts/list ("Discover available prompts") and prompts/get ("Retrieve prompt details"). (source: modelcontextprotocol.io/server-concepts)
🤔 Predict first
A server exposes a "send_email" capability that the model should be able to fire on its own, and a "vacation_planner" template a user picks from a menu. Which primitive is each?
send_email = a Tool (model-controlled — the model decides when to call it, though human oversight/consent is emphasized). vacation_planner = a Prompt (user-controlled — "requiring explicit invocation," typically via a / slash command). If instead you had read-only flight data the app pulls in for context, that'd be a Resource (application-controlled), addressed by a URI.
❓ Quick check
In the M-A-U split, who controls Resources?
Resources are Application-controlled (the "A" in M-A-U): "Resources are application-driven." The app decides what to pull into context. Tools are Model-controlled; Prompts are User-controlled.
🔑 Key takeaways▼
  • M-A-U: Tools = Model, Resources = Application, Prompts = User. (Highest-yield fact.)
  • Tools act (JSON-Schema typed), but "MCP emphasizes human oversight" + consent.
  • Resources are read-only, addressed by URIs (direct or templated).
  • Prompts require explicit user invocation — typically / slash commands.

④ Client primitives

Client Primitives — What the Host Offers Back

Sampling, Roots, and Elicitation flow the other direction: server → client.

🔁 Section 4⏱ ~13 minDomain 4 · Domain 2

The features clients expose to servers

Learning objectives
  • Name the three client primitives and their methods
  • Explain why Sampling keeps servers model-independent
  • Recall that Roots are advisory (not enforced) and Elicitation never asks for secrets

The direction flips

Section 3 was server → client (the server offers tools/resources/prompts). Now the direction flips: "clients may provide several features to servers. These client features allow server authors to build richer interactions." (source: modelcontextprotocol.io/client-concepts) Three primary client features:

FeatureWhat it does (verbatim)Method
Sampling"Sampling allows servers to request LLM completions through the client, enabling an agentic workflow. This approach puts the client in complete control of user permissions and security measures."sampling/createMessage
Roots"Roots allow clients to specify which directories servers should focus on, communicating intended scope through a coordination mechanism."roots/list (+ roots/list_changed)
Elicitation"Elicitation enables servers to request specific information from users during interactions, providing a structured way for servers to gather information on demand."elicitation/create

4.1 Sampling — keeps servers model-independent

"Sampling enables servers to perform AI-dependent tasks without directly integrating with or paying for AI models. Instead, servers can request that the client—which already has AI model access—handle these tasks on their behalf." This is the key architectural payoff: a server author can "want access to a language model, but want to stay model-independent and not include a language model SDK in their MCP server." (source: modelcontextprotocol.io/architecture)

It's human-in-the-loop by design with multiple checkpoints: "Users review and can modify both the initial request and the generated response before it returns to the server." Request params include messages, modelPreferences (with hints, costPriority, speedPriority, intelligencePriority), systemPrompt, and maxTokens.

4.2 Roots — advisory, not enforced

"Roots are a mechanism for clients to communicate filesystem access boundaries to servers... While roots communicate intended boundaries, they do not enforce security restrictions. Actual security must be enforced at the operating system level, via file permissions and/or sandboxing." Roots "are exclusively filesystem paths and always use the file:// URI scheme," e.g. {"uri": "file:///Users/agent/travel-planning", "name": "Travel Planning Workspace"}.

⚠️ SHOULD, not MUST — and why▼

"The specification requires that servers 'SHOULD respect root boundaries,' and not that they 'MUST enforce' them, because servers run code the client cannot control." (source: modelcontextprotocol.io/client-concepts) Roots are a coordination hint, not a security boundary — never treat them as a sandbox. Dynamic changes flow via the roots/list_changed notification.

4.3 Elicitation — ask the user, never for secrets

"Elicitation provides a structured way for servers to gather necessary information on demand. Instead of requiring all information up front or failing when data is missing, servers can pause their operations to request specific inputs from users." The request carries a message plus a JSON schema for the fields requested.

⚠ VERIFY Privacy guardrail: "Elicitation never requests passwords or API keys. Clients warn about suspicious requests and let users review data before sending." Maturity is version-sensitive: the architecture page labels Tasks as "(Experimental)." Elicitation and Sampling appear as standard client features in current docs with no experimental label, but confirm GA status against the dated spec at modelcontextprotocol.io/specification/latest before asserting it in production. (source: client-concepts, architecture)

A fourth client-side utility — Logging — "enables servers to send log messages to clients for debugging and monitoring," distinct from the three primary primitives.

❓ Quick check
Why does Sampling matter architecturally for a server author?
Sampling lets a server "perform AI-dependent tasks without directly integrating with or paying for AI models" — the client (which has model access) handles it. The server stays model-independent, and the client keeps human-in-the-loop control.
🔑 Key takeaways▼
  • Client primitives flow server → client: Sampling (sampling/createMessage), Roots (roots/list), Elicitation (elicitation/create).
  • Sampling keeps servers model-independent and is human-in-the-loop.
  • Roots are advisory ("SHOULD respect," not "MUST enforce") — OS-level security still required.
  • Elicitation never requests passwords or API keys. ⚠ VERIFY primitive maturity.
⑤ Transports

Transports — stdio vs Streamable HTTP

Two standard transports. The HTTP+SSE deprecation has a nuance you must get right.

🚚 Section 5⏱ ~15 minDomain 4

How the bytes actually travel

Learning objectives
  • Distinguish stdio (local subprocess) from Streamable HTTP (remote, current standard)
  • Get the HTTP+SSE deprecation nuance right — SSE survives inside Streamable HTTP
  • Recall the three Streamable-HTTP security warnings

The two standard transports

"The protocol currently defines two standard transport mechanisms... 1. stdio... 2. Streamable HTTP." "Clients SHOULD support stdio whenever possible." Over either, "MCP uses JSON-RPC to encode messages. JSON-RPC messages MUST be UTF-8 encoded." (source: transports docs)

5.1 stdio (local)

  • "The client launches the MCP server as a subprocess."
  • "The server reads JSON-RPC messages from its standard input (stdin) and sends messages to its standard output (stdout)."
  • "Messages are delimited by newlines, and MUST NOT contain embedded newlines."
  • "The server MAY write UTF-8 strings to its standard error (stderr) for logging purposes."
  • "The server MUST NOT write anything to its stdout that is not a valid MCP message." Use stdio for local subprocess servers with "optimal performance with no network overhead." (source: transports, architecture)

5.2 Streamable HTTP (remote — current standard)

  • "The server operates as an independent process that can handle multiple client connections. This transport uses HTTP POST and GET requests. Server can optionally make use of Server-Sent Events (SSE) to stream multiple server messages."
  • "The server MUST provide a single HTTP endpoint path (hereafter referred to as the MCP endpoint) that supports both POST and GET methods," e.g. https://example.com/mcp.
  • Clients "include an Accept header, listing both application/json and text/event-stream."
  • Session: the server MAY assign an Mcp-Session-Id header at init; clients MUST echo it on subsequent requests.
  • Version header: the client MUST include MCP-Protocol-Version: <protocol-version> on all subsequent requests, e.g. MCP-Protocol-Version: 2025-11-25 (the current canonical spec revision, verified 2026-06-06); if the header is absent the server SHOULD assume an earlier default per the spec's backwards-compatibility rule (historically 2025-03-26).
  • Auth: "supports standard HTTP authentication methods including bearer tokens, API keys, and custom headers. MCP recommends using OAuth."

5.3 The HTTP+SSE deprecation — the nuance

Streamable HTTP "replaces the HTTP+SSE transport from protocol version 2024-11-05" — explicitly called "the deprecated HTTP+SSE transport." But here is the nuance the exam loves:

🎯 SSE is NOT gone — read carefully▼

What was deprecated is the standalone HTTP+SSE transport — the older two-endpoint design with a separate SSE endpoint and an endpoint event. SSE itself survives as an optional streaming mechanism inside Streamable HTTP. So "SSE is deprecated" is only half-true: the standalone transport is deprecated; SSE-as-a-streaming-mechanism lives on. (source: transports docs)

Product-surface confirmation: Claude Code's docs say "The SSE (Server-Sent Events) transport is deprecated. Use HTTP servers instead, where available." Claude Code accepts streamable-http as an alias for http — "The MCP specification uses the name streamable-http for this transport." (source: code.claude.com/mcp)

✓ VERIFIED 2026-06-06 Protocol version strings. The current canonical "latest" spec revision is 2025-11-25 (confirmed at modelcontextprotocol.io/specification/latest). 2025-06-18 and 2025-03-26 are prior revisions (the latter is still the documented backwards-compatibility fallback). Re-verify before exam day — spec dates roll forward. (source: modelcontextprotocol.io/specification/latest, transports, mcp-connector)

5.4 Security warnings (Streamable HTTP)

  1. "Servers MUST validate the Origin header on all incoming connections to prevent DNS rebinding attacks."
  2. "When running locally, servers SHOULD bind only to localhost (127.0.0.1) rather than all network interfaces (0.0.0.0)."
  3. "Servers SHOULD implement proper authentication for all connections."

"Without these protections, attackers could use DNS rebinding to interact with local MCP servers from remote websites." (source: transports docs) MCP is also transport-agnostic: clients/servers "MAY implement additional custom transport mechanisms," but must "preserve the JSON-RPC message format and lifecycle requirements."

🤔 Predict first
An exam item says "SSE has been removed from MCP." True or false — and what's the precise correction?
False (and a classic trap). What was deprecated is the standalone HTTP+SSE transport (the two-endpoint 2024-11-05 design). SSE itself survives inside Streamable HTTP as an optional way for the server to stream multiple messages. Saying "SSE is gone" loses the nuance the spec is careful about.
❓ Quick check
Which transport runs the server as a local subprocess communicating over stdin/stdout?
stdio: "The client launches the MCP server as a subprocess," reading from stdin and writing to stdout. Streamable HTTP is the remote, multi-client transport. "Clients SHOULD support stdio whenever possible."
🔑 Key takeaways▼
  • Two standard transports: stdio (local subprocess, one client, no network) and Streamable HTTP (remote, many clients, one POST+GET endpoint).
  • The standalone HTTP+SSE transport (2024-11-05) is deprecated — but SSE survives inside Streamable HTTP.
  • Streamable HTTP security: validate Origin (MUST), bind localhost (SHOULD), authenticate (SHOULD).
  • ⚠ VERIFY the canonical protocol-version string and that the deprecation hasn't become removal.
⑥ Wire protocol

JSON-RPC 2.0 & the Initialize Handshake

The data layer's message format, and the capability negotiation that opens every connection.

📡 Section 6⏱ ~15 minDomain 4

How a connection opens and stays in sync

Learning objectives
  • Distinguish requests, responses, and notifications in JSON-RPC 2.0
  • Walk the initialize handshake and its three purposes
  • Read the real initialize request/response JSON and capability semantics

Three message types

"The data layer implements a JSON-RPC 2.0 based exchange protocol." "Client and servers send requests to each other and respond accordingly. Notifications can be used when no response is required." (source: architecture)

  • Requests — have an id; expect a response.
  • Responses — echo the same id.
  • Notifications — "without expecting a response"; no id.

MCP "is a stateful protocol that requires lifecycle management. The purpose of lifecycle management is to negotiate the capabilities that both client and server support."

The initialize handshake — three purposes

  1. Protocol Version Negotiation — "ensures both client and server are using compatible protocol versions... If a mutually compatible version is not negotiated, the connection should be terminated."
  2. Capability Discovery — "The capabilities object allows each party to declare what features they support."
  3. Identity Exchange — "The clientInfo and serverInfo objects provide identification and versioning information."

The real messages

Request · initialize
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2025-11-25",
    "capabilities": { "elicitation": {} },
    "clientInfo": { "name": "example-client", "version": "1.0.0" }
  }
}
Response · initialize result
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "protocolVersion": "2025-11-25",
    "capabilities": {
      "tools": { "listChanged": true },
      "resources": {}
    },
    "serverInfo": { "name": "example-server", "version": "1.0.0" }
  }
}

Capability semantics: "tools": {"listChanged": true} means "The server supports the tools primitive AND can send tools/list_changed notifications when its tool list changes." On the client side, "elicitation": {} means "The client declares it can work with user interaction requests (can receive elicitation/create method calls)." (source: architecture)

Notification · client signals ready
{ "jsonrpc": "2.0", "method": "notifications/initialized" }

Notice: no id field. After this, normal traffic begins. A later tool-list change is also a notification: {"jsonrpc": "2.0", "method": "notifications/tools/list_changed"} — "only sent by servers that declared 'listChanged': true." To cancel an in-flight request, "the client SHOULD explicitly send an MCP CancelledNotification."

🤔 Predict first
You see a JSON-RPC message with a method but no id field. What kind of message is it, and what does the sender expect back?
It's a notification. No id ⇒ no response is expected or sent ("Notifications can be used when no response is required"). Examples: notifications/initialized (client is ready) and notifications/tools/list_changed (the tool list changed). Requests, by contrast, carry an id and get a matching-id response.
❓ Quick check
What does the initialize handshake accomplish?
Three purposes: protocol-version negotiation (terminate if incompatible), capability discovery (the capabilities object), and identity exchange (clientInfo/serverInfo). Then the client sends notifications/initialized.
🔑 Key takeaways▼
  • Data layer = JSON-RPC 2.0: requests (have id), responses (echo id), notifications (no id, no response).
  • Every connection opens with initialize: version negotiation + capability discovery + identity exchange.
  • Capabilities like "tools": {"listChanged": true} declare both support and change-notification ability.
  • Client then sends notifications/initialized (no id) before normal traffic.
⑦ Discovery + invocation

The */list & */call Method Catalog

How a client discovers what a server offers, then invokes it.

🔎 Section 7⏱ ~12 minDomain 4

Discover, then call

Learning objectives
  • State the */list / */call discovery pattern and why it's dynamic
  • Read the real tools/list and tools/call messages
  • Place each method in the consolidated catalog

The discovery pattern

"Each primitive type has associated methods for discovery (*/list), retrieval (*/get), and in some cases, execution (tools/call). MCP clients will use the */list methods to discover available primitives... This design allows listings to be dynamic." (source: architecture) Because discovery is a live method call rather than a static manifest, a server can change what it offers and announce it with a */list_changed notification.

tools/list — discovery

"The tools/list request is simple, containing no parameters":

Request · tools/list
{ "jsonrpc": "2.0", "id": 2, "method": "tools/list" }

The response returns a tools array where each tool carries name, title, description, and inputSchema (JSON Schema).

tools/call — invocation

Request · tools/call
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "tools/call",
  "params": {
    "name": "weather_current",
    "arguments": { "location": "San Francisco", "units": "imperial" }
  }
}

The response returns a content array of typed blocks (e.g. {"type": "text", "text": "..."}). "Tool responses return an array of content objects, allowing for rich, multi-format responses (text, images, resources, etc.)." (source: architecture)

❓ Quick check
Which pair of methods discovers and then invokes a tool?
tools/list (discovery, no params) → tools/call (execution, with name + arguments). The prompts/list → prompts/get pair is for prompts, not tools.
🔑 Key takeaways▼
  • Pattern: */list discovers, */get retrieves, tools/call executes — and listings are dynamic.
  • tools/list takes no params; tools/call takes name + arguments and returns a typed content array.
  • The catalog spans server primitives (tools/*, resources/*, prompts/*) and client primitives (sampling/*, elicitation/*, roots/*).
⑧ MCP with Claude

MCP with Claude — Connector vs Claude Code Host

Keep MCP-the-standard distinct from Anthropic's product surfaces that consume it.

🤖 Section 8⏱ ~15 minDomain 4 · Domain 1

Two Anthropic surfaces that speak MCP

Learning objectives
  • Keep MCP-the-standard distinct from Anthropic's product implementations
  • Recall the Messages API MCP connector's limits (remote-only, tool-calls-only, beta header)
  • Recall Claude Code's claude mcp add, the three scopes, and the -- separator

One framing first: MCP the open standard (modelcontextprotocol.io) is distinct from Anthropic's product implementations that consume it — the Messages API connector, Claude Code, Claude Desktop, and Claude.ai connectors. Don't let a product limitation get mistaken for a protocol limitation.

8.1 The Messages API MCP connector

"Claude's Model Context Protocol (MCP) connector feature enables you to connect to remote MCP servers directly from the Messages API without a separate MCP client." Two components: an mcp_servers array (each with type: "url", an https:// url, a name, optional authorization_token) and a tools array with one mcp_toolset per server.

🚧 The connector's hard limits (testable)▼
  • Tool calls only: "Of the feature set of the MCP specification, only tool calls are currently supported." (No resources/prompts via this path.)
  • Remote only: "The server must be publicly exposed through HTTP... Local STDIO servers cannot be connected directly."
  • Platform matrix: available on the Claude API, Claude Platform on AWS, and Microsoft Foundry, but "not currently available on Amazon Bedrock or Vertex AI."
  • Data retention: "This feature is not eligible for Zero Data Retention (ZDR)."
  • Response blocks: mcp_tool_use (includes server_name) and mcp_tool_result (includes tool_use_id, is_error, content). Tool filtering supports allowlist/denylist — "Denylisting write or destructive tools is recommended when building read-only assistants."

(source: mcp-connector docs)

⚠ VERIFY Required beta header: "anthropic-beta": "mcp-client-2025-11-20". "The previous version (mcp-client-2025-04-04) is deprecated." Beta header strings rotate — re-verify before publishing. Platform availability is also subject to change. (source: mcp-connector docs)

For local servers, prompts, or resources, the TypeScript SDK offers client-side helpers (mcpTools, mcpMessages, mcpResourceToContent, mcpResourceToFile): "Use the mcp_servers API parameter when you have remote servers... and only need tool support. Use the client-side helpers when you need local servers, prompts, resources, or more control."

8.2 Claude Code as an MCP host

Claude Code "can connect to hundreds of external tools and data sources through the Model Context Protocol." Unlike the connector, Claude Code is a full host — it consumes all three server primitives and supports all transports.

claude mcp add — the three primary transports
# Remote HTTP (recommended)
claude mcp add --transport http notion https://mcp.notion.com/mcp

# Remote SSE (deprecated)
claude mcp add --transport sse <name> <url>

# Local stdio — note the -- separator
claude mcp add --env AIRTABLE_API_KEY=YOUR_KEY \
  --transport stdio airtable -- npx -y airtable-mcp-server
➖ The -- separator (testable detail)▼

"For stdio servers, the -- (double dash) separates Claude's own options... from the command and arguments that run the server. Everything after -- is passed to the server untouched." So --env and --transport are Claude Code's flags; npx -y airtable-mcp-server after the -- is the server's launch command. (source: code.claude.com/mcp)

The three scopes

ScopeVisibility (verbatim)Stored in

Precedence (highest → lowest): Local → Project → User → Plugin-provided → claude.ai connectors, and "fields are not merged across scopes." Claude Code surfaces the primitives natively: prompts as /mcp__servername__promptname slash commands; resources via @server:protocol://resource/path (e.g. @github:issue://123); elicitation as automatic interactive dialogs. Management: claude mcp list, claude mcp get, claude mcp remove, and the in-session /mcp panel. Claude Code can even be a server itself: claude mcp serve "Start[s] Claude as a stdio MCP server." (source: code.claude.com/mcp)

🤔 Predict first
You want Claude on the Messages API to use a local stdio MCP server's prompts. Will the mcp_servers connector parameter do it?
No — on two counts. The Messages API connector is (1) remote-only ("Local STDIO servers cannot be connected directly") and (2) tool-calls-only ("only tool calls are currently supported" — no prompts/resources). For local servers, prompts, or resources you use the TypeScript SDK's client-side helpers, or run a full host like Claude Code. These are product limits, not protocol limits.
❓ Quick check
In claude mcp add --transport stdio airtable -- npx -y airtable-mcp-server, what does the -- do?
"Everything after -- is passed to the server untouched." Before it: Claude Code's flags (--transport, --env). After it: the server's launch command (npx -y airtable-mcp-server).
🔑 Key takeaways▼
  • MCP the standard ≠ Anthropic's product surfaces that consume it.
  • Messages API connector: remote-only, tool-calls-only, ⚠ VERIFY beta header mcp-client-2025-11-20, not ZDR-eligible.
  • Claude Code is a full host: claude mcp add, three scopes (Local→Project→User precedence), the -- separator for stdio.
  • Claude Code consumes all three primitives (slash-command prompts, @-referenced resources, auto elicitation) and can be a server via claude mcp serve.
⑨ Security

Security & Builder Concerns

Permissioning, trust before connecting, and prompt-injection risk.

🛡️ Section 9⏱ ~12 minDomain 4

Trust is the first design decision

Learning objectives
  • Recall the human-oversight mechanisms for model-controlled tools
  • State the "trust the server before connecting" rule and the prompt-injection risk
  • Tie together the security boundaries: consent, advisory roots, transport hardening

Tools act — so gate them

Tools are model-controlled, "However, MCP emphasizes human oversight." The recommended mechanisms: "Displaying available tools in the UI... Approval dialogs for individual tool executions... Permission settings for pre-approving certain safe operations... Activity logs that show all tool executions." And "Tools may require user consent prior to execution, helping to ensure users maintain control over actions taken by a model." (source: server-concepts)

Trust before connecting — and the prompt-injection risk

"Verify you trust each server before connecting it. Servers that fetch external content can expose you to prompt injection risk." (source: code.claude.com/mcp)

A connected server can inject content into the model's context; if that server pulls in attacker-controlled external content, that content can carry instructions. This is why Claude Code "prompts for approval before using project-scoped servers from .mcp.json files" — a server checked into a shared repo shouldn't run on a teammate's machine without explicit consent.

The other boundaries, recalled

  • Sampling is permission-gated at the client: it "puts the client in complete control of user permissions and security measures," with human-in-the-loop review before and after the model call.
  • Roots are advisory, not a boundary: "Roots serve as a coordination mechanism... not a security boundary. Actual security must be enforced at the operating system level."
  • Transport hardening (Section 5): validate Origin, bind localhost, authenticate.
  • Elicitation "never requests passwords or API keys."
  • Self-served caveat: when exposing Claude Code via claude mcp serve, "your own client is responsible for implementing user confirmation for individual tool calls." (source: code.claude.com/mcp, client-concepts, transports)
❓ Quick check
Why does connecting an untrusted MCP server raise a prompt-injection concern?
"Servers that fetch external content can expose you to prompt injection risk." That content lands in context and may carry instructions — so "verify you trust each server before connecting it," and approve project-scoped .mcp.json servers explicitly.
🔑 Key takeaways▼
  • Model-controlled tools demand human oversight: approval dialogs, pre-approval settings, activity logs, consent.
  • Trust the server before connecting — external-content servers carry prompt-injection risk; project-scoped .mcp.json needs explicit approval.
  • Boundaries stack: sampling is client-gated, roots are advisory (OS enforces), transport hardening + no-secrets elicitation.
⑩ MCP vs raw tool use

How MCP Relates to Raw Tool Use

It's a standardized packaging/transport layer over the same C-E1 loop.

🔗 Section 10⏱ ~10 minDomain 4 · Domain 2

Same loop, standardized wrapper

Learning objectives
  • State the one-sentence relationship between MCP and native tool use
  • Explain why MCP tools and native tools are indistinguishable to the model
  • Connect tools/list/tools/call back to the C-E1 discover→call→observe cycle

The one clean sentence

MCP is a standardized packaging-and-transport layer over the same tool-use loop you already know. When Claude decides to call an MCP tool, "the AI application intercepts the tool call, routes it to the appropriate MCP server, executes it, and returns the results back to the LLM as part of the conversation flow" — exactly the discover → call → observe cycle of native tool use from C-E1, but with tools/list / tools/call standardized across every server. (source: architecture)

Why the model can't tell the difference

The host "fetches available tools from all connected MCP servers and combines them into a unified tool registry that the language model can access." So "from the model's perspective MCP tools and natively-defined tools are indistinguishable." On the Messages API this is literally true: MCP tools flow through the same tool-use machinery, surfacing as mcp_tool_use / mcp_tool_result blocks parallel to ordinary tool_use / tool_result. (source: architecture, mcp-connector)

🧭 The mental model to carry into the exam▼

C-E1 taught the loop (model decides → execute → feed result back). MCP doesn't change that loop — it standardizes the contract around it so the loop works against any server: the schema format (JSON Schema), the discovery method (tools/list), the invocation method (tools/call), the result shape (typed content array), and the transport. Write the loop once; point it at any MCP server. That reusability is the architect's payoff.

🤔 Predict first
Does adopting MCP change the fundamental agent loop from C-E1 (model decides → execute → feed result back)?
No. The loop is unchanged. MCP standardizes the packaging and transport around it — discovery (tools/list), invocation (tools/call), schema (JSON Schema), result shape, and how bytes travel. The host merges MCP tools into a unified registry, so "from the model's perspective MCP tools and natively-defined tools are indistinguishable." You're running the C-E1 loop over a standardized wire.
❓ Quick check
From the model's perspective, MCP tools versus natively-defined tools are…
The host "combines them into a unified tool registry," so "MCP tools and natively-defined tools are indistinguishable" to the model. MCP standardizes packaging/transport over the same C-E1 loop.
🔑 Key takeaways▼
  • MCP = a standardized packaging/transport layer over the same C-E1 tool-use loop — it doesn't change the loop.
  • The host merges MCP tools into a unified registry; MCP and native tools are indistinguishable to the model.
  • On the Messages API: mcp_tool_use/mcp_tool_result parallel ordinary tool_use/tool_result.

Q1
MCP exists primarily to…
"Replacing fragmented integrations with a single protocol." Open standard; each side implements MCP once ⇒ M+N.
Q2
A host connects to four MCP servers. The number of MCP clients is…
"One MCP client for each MCP server," each on a dedicated connection. Four servers ⇒ four clients.
Q3
In the M-A-U split, Prompts are controlled by the…
Prompts are User-controlled (the "U"): "requiring explicit invocation," typically via / slash commands.
Q4
Which client primitive lets a server use an LLM while staying model-independent?
Sampling: the client (which has model access) handles completions on the server's behalf, so the server needs no model SDK — it stays model-independent.
Q5
Which statement about transports is correct?
Streamable HTTP (one POST+GET endpoint) is the current remote standard. The standalone HTTP+SSE transport is deprecated, but SSE persists as an optional streaming mechanism within Streamable HTTP.
Q6
A JSON-RPC 2.0 message with a method but no id is a…
No id ⇒ a notification; "no response is expected or sent." E.g. notifications/initialized.
Q7
The initialize handshake exists to…
Three purposes: version negotiation (terminate if incompatible), capability discovery, identity exchange. Then the client sends notifications/initialized.
Q8
To discover then execute a tool, a client calls…
tools/list (no params) discovers; tools/call (name + arguments) executes and returns a typed content array.
Q9
The Messages API MCP connector supports…
"Only tool calls are currently supported" and "Local STDIO servers cannot be connected directly." A product limit, not a protocol limit. VERIFY the beta header.
Q10
How does MCP relate to the raw tool-use loop from C-E1?
Same discover→call→observe loop; the host merges MCP tools into a unified registry, so they're "indistinguishable" from native tools. MCP standardizes the wire, not the loop.
