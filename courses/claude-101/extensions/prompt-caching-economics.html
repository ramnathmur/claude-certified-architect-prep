Prompt Caching Economics

Every multi-turn conversation, agent loop, and document-Q&A app re-sends the same large prefix on every call — and pays full freight each time. Prompt caching lets Claude "resume from" an already-processed prefix at 0.1× the input price. This lesson drops below the headline into the actual math: the multipliers, the break-even algebra, the TTL gotcha, and when caching costs you more.

① The problem

What Caching Is — and the Repeated-Prefix Problem

The same large static content is re-processed on every request. Caching lets the model resume from it.

💰 Section A⏱ ~10 minDomain 5

Why you keep paying for the same tokens

Learning objectives
  • State the repeated-prefix problem and why it costs both latency and money
  • Define prompt caching in Anthropic's own words and name the twin benefit
  • Place the "~85% latency / ~90% cost" figures correctly (secondary vs. arithmetic)

Why before how

Picture any of the systems you already build: a multi-turn conversation, an agent loop, a document-Q&A app. Each of them sends the same large static block on every single request — the system prompt, the tool definitions, a long reference document or RAG context, the few-shot examples. That block is "sent and re-processed on every single request — paid for in full each time, in latency and input-token cost." (source: prompt-caching doc, pricing) Turn 1 pays for it. Turn 2 pays for the identical bytes again. Turn 10 pays a tenth time. The model has to prefill (re-encode) the whole prefix each call before it can generate a single new token.

That is the repeated-prefix problem — and it's the pattern name to carry forward: a large, byte-identical prefix re-sent many times, billed at full input price every time.

What prompt caching actually is

"Prompt caching optimizes your API usage by allowing resuming from specific prefixes in your prompts. This significantly reduces processing time and costs for repetitive tasks or prompts with consistent elements." (source: prompt-caching doc)
"Prompt caching reduces costs and latency by reusing previously processed portions of your prompt across API calls. Instead of reprocessing the same large system prompt, document, or conversation history on every request, the API reads from cache at a fraction of the standard input price." (source: pricing page)

So the fix is mechanical: write the processed prefix to a cache once, then on later calls "resume from" it. The cached prefix is "billed at the cache-read rate (0.1× base input) instead of full input price, and skips the prefill compute." (source: prompt-caching doc, pricing) That's the twin benefit: cost reduction on the cached prefix and latency reduction, because the cached prefix's prefill is skipped entirely.

How big is the win? Read the numbers carefully

⚠ VERIFY SECONDARY The "~85% latency / ~90% cost" headline. Secondary sources cite "up to ~85% latency reduction" and "up to ~90% cost reduction" for long prompts (a ~100K-token doc dropping from ~11.5s to ~2.4s). These trace to Anthropic's original 2024 launch framing, not the current doc — treat the 85% latency figure as illustrative/secondary. The 90%-on-the-cached-prefix figure, by contrast, is defensible as plain arithmetic: a read costs 0.1× input, so you save up to 90% on the cached portion. That's the floor of the read multiplier, not a marketing claim. (source: secondary aggregate for the % framing; pricing page for the 0.1× arithmetic)
🤔 Predict first
A 100,000-token reference document sits in the prefix of a Q&A app. Users ask 50 questions against it in five minutes. Without caching, how many times is that document re-encoded and billed? With caching?
Without: 50 times at full 1.0× input — the model re-prefills 100K tokens on every question. With caching: 1 write (1.25×) on the first question, then 49 reads at 0.1× — and because each hit refreshes the 5-minute clock, the cache stays warm across the whole burst. The prefill is skipped on all 49 reads, which is where the latency win comes from. This is exactly the "Large document processing" / "Knowledge base interaction" use case Anthropic recommends. (source: prompt-caching doc)
❓ Quick check
What is the "twin benefit" of prompt caching?
Caching "reduces costs and latency by reusing previously processed portions of your prompt." Cost drops because the cached prefix bills at 0.1×; latency drops because the cached prefix's prefill is skipped. It does NOT enlarge the context window or touch output pricing.
🔑 Key takeaways▼
  • Repeated-prefix problem: the same large static block (system + tools + long context + few-shot) is re-processed and re-billed at full price on every request.
  • Caching lets the model "resume from" an already-processed prefix — read at 0.1× base input, prefill skipped.
  • Twin benefit: cost ↓ on the cached prefix and latency ↓. It does not grow the context window.
  • The "~85% latency / ~90% cost" headline is SECONDARY/launch-era; the 90%-on-cached-prefix is arithmetic from the 0.1× read multiplier.
② Mechanism

The Mechanism — Breakpoints, Prefix Order & Invalidation

How you mark a cache, what's cacheable, the minimum lengths, and how a change busts everything after it.

⚙️ Section B⏱ ~16 minDomain 5 · Domain 3/4

cache_control, the prefix hierarchy, and the cascade

Learning objectives
  • Write a cache_control breakpoint and state the placement rule
  • Recall what is and isn't cacheable, the fixed tools→system→messages order, and the per-model minimums
  • Apply the invalidation cascade — why editing a tool definition is the most expensive change

B1 · The cache_control ephemeral block

You don't cache by flipping a global switch; you mark a breakpoint with a cache_control block. Default (5-minute) TTL:

cache_control · 5-minute default
{ "cache_control": {"type": "ephemeral"} }
cache_control · extended 1-hour TTL
{ "cache_control": {"type": "ephemeral", "ttl": "1h"} }

There are two ways to turn caching on (verbatim, pricing page):

"Automatic caching: Add a single cache_control field at the top level of your request. The system automatically manages cache breakpoints as conversations grow. This is the recommended starting point for most use cases."

"Explicit cache breakpoints: Place cache_control directly on individual content blocks for fine-grained control over exactly what gets cached." (source: pricing page)

Placement rule: cache_control goes on the last content block you want included in the cached prefix — inside the tools, system, or messages.content arrays. "Everything from the start of the request up to and including that marked block becomes the cached prefix." (source: prompt-caching doc)

B2 · Breakpoints, lookback, and exact match

  • Max breakpoints: "You can define up to 4 cache breakpoints" per request. ⚠ VERIFY
  • Lookback window: "The system checks at most 20 positions per breakpoint, counting the breakpoint itself as the first." It walks backward one block at a time within that window looking for a previously cached entry — so you get hits at earlier block boundaries, not only the exact breakpoint. ⚠ VERIFY
  • Exact-match requirement: "Cache hits require 100% identical prompt segments, including all text and images up to and including the block marked with cache control." (source: prompt-caching doc)
⚠️ "100% identical" is the rule everything else hangs on▼

A single changed byte before or at the breakpoint means no hit — the prefix is re-written from scratch. This is why a timestamp or username injected into the system prompt is so destructive (Section F). The exact-match rule is also why the invalidation cascade (B6) exists: change something early in the contiguous prefix and the identical-match fails for everything downstream.

B3 · What can — and can't — be cached

Cacheable content (verbatim list):

  • Tools — tool definitions in the tools array
  • System messages — content blocks in the system array
  • Text messages — content blocks in messages.content, for both user and assistant turns
  • Images & Documents — content blocks in messages.content, in user turns
  • Tool use and tool results — content blocks in messages.content, in both user and assistant turns

What CANNOT be cached: "Thinking blocks cannot be cached directly with cache_control"; "Sub-content blocks (like citations) themselves cannot be cached directly"; "Empty text blocks cannot be cached." (source: prompt-caching doc)

B4 · The fixed prefix order — tools → system → messages

"Cache prefixes are created in the following order: tools, system, then messages. This order forms a hierarchy where each level builds upon the previous ones." (source: prompt-caching doc)

The implication is structural and testable: the cache always covers a contiguous prefix starting at the very top of the request. You cannot cache a middle slice while leaving the start uncached. (Compare this to C-E3's tools→system→messages request anatomy — same ordering, now with a cost meaning.)

B6 · The invalidation cascade

"Changes at each level invalidate that level and all subsequent levels." (source: prompt-caching doc)
What changesInvalidates
Tool definitionsTools + System + Messages (the entire cache)
Web search toggleSystem + Messages
Citations toggleSystem + Messages
Speed settingSystem + Messages
Tool choiceMessages only
ImagesMessages only
Thinking parametersMessages only

Architect takeaway: because the cache is a contiguous prefix, changing anything earlier busts everything after it. Editing a tool definition is the most expensive change you can make — it invalidates the entire cache. This is the cost-side reason to keep tool definitions stable in an agent loop (a Domain 4 connection: every tool-def tweak forces a fresh full-price write of the whole prefix).

🤔 Predict first
Your agent caches a prefix of tools + a long system prompt + accumulated conversation. Mid-session you add one new tool to the tools array. What gets invalidated — and what does the next call cost on the prefix?
The entire cache busts — "changes at each level invalidate that level and all subsequent levels," and tools is the first level, so tools + system + messages all go. The next call must re-write the whole prefix at the write multiplier (1.25× for 5-min), not read it at 0.1×. That's why "cache the system + tools first and avoid editing tool defs casually" is a core pattern (Section F). Compare: changing only the thinking parameters or a tool-choice would invalidate messages only, leaving the expensive tools+system layers cached.
❓ Quick check
Where do you place the cache_control block to cache a stable prefix?
cache_control marks "the last content block you want included in the cached prefix." Everything from the start of the request up to and including that block becomes the contiguous cached prefix — which is why the order is fixed tools → system → messages.
❓ Quick check
On Claude Haiku 4.5, what is the minimum cacheable prefix length — and what happens if you mark a shorter prefix?
Haiku 4.5 requires 4,096 tokens (NOT the stale 2,048 — that's retired Haiku 3.5). Below the minimum, the request "will be processed without caching, and no error is returned" — a silent no-cache.
🔑 Key takeaways▼
  • cache_control: {"type":"ephemeral"} marks the last block in the cached prefix; add "ttl":"1h" for the extended window.
  • Up to 4 breakpoints; 20-position lookback per breakpoint; hits require 100% identical segments. ⚠ VERIFY
  • Cache is a contiguous prefix in fixed tools → system → messages order — no middle-slice caching.
  • Minimums are version-tied (Opus 4.8 = 1,024; Haiku 4.5 = 4,096); below it = silent no-cache. ⚠ VERIFY
  • Invalidation cascades downward — a tool-definition edit busts the entire cache.
③ TTL & refresh

TTL & Refresh — 5-Minute Default, 1-Hour Opt-In

The clock resets on every hit. The concurrency caveat trips up fan-out agents.

⏱️ Section C⏱ ~11 minDomain 5

How long a cache lives, and how it stays warm

Learning objectives
  • State the default 5-minute TTL and the refresh-on-hit behaviour
  • Recall the 1-hour opt-in and its extra write cost
  • Apply the concurrency caveat to parallel / fan-out agent calls

The 5-minute default — and the refresh that keeps it warm

⚠ VERIFY "By default, the cache has a 5-minute lifetime." But the lifetime is not a hard countdown from the write — it resets on use:

"The cache is refreshed for no additional cost each time the cached content is used." (source: prompt-caching doc)

So the 5-minute (or 1-hour) clock resets on every cache hit. A continuously-used prefix stays warm indefinitely at no extra write cost — you pay the one write, then every read both costs 0.1× and pushes the expiry out another full window. This is the single most important behaviour to internalize before the economics: a busy prefix never expires.

The 1-hour extended TTL

⚠ VERIFY The extended TTL is 1 hour, opt-in via "ttl": "1h", at an additional write cost — 2× base input versus 1.25× for the 5-minute write (the full multiplier table is Section D). Deletion timing: cached entries have a minimum lifetime, and after expiry they are "promptly, though not immediately, deleted." ⚠ VERIFY (source: prompt-caching doc, pricing)

The concurrency caveat — the fan-out trap

"For concurrent requests, note that a cache entry only becomes available after the first response begins. If you need cache hits for parallel requests, wait for the first response before sending subsequent requests." (source: prompt-caching doc)

This bites exactly the agentic / parallel-tool patterns you build. If you fan out N parallel calls that share a prefix before the first response returns, the cache entry isn't available yet — so each of those first calls pays a cache write, not a read. The fix: warm the cache with one call, wait for its response to begin, then fan out the rest as reads.

🤔 Predict first
You launch 8 parallel agent calls at t=0, all sharing the same 30K-token cached prefix. How many cache writes do you pay?
Up to 8 writes — one per call. "A cache entry only becomes available after the first response begins," so at t=0 none of the 8 can hit a cache that doesn't exist yet; each writes its own. To get 1 write + 7 reads, fire one call first, wait for its response to begin, then launch the other 7. The same prefix, fanned out naively, can cost 8× the write instead of one.
❓ Quick check
A prefix is hit once every 3 minutes for an hour on the default 5-minute TTL. How many cache writes do you pay?
"The cache is refreshed for no additional cost each time the cached content is used." With 3-minute gaps (< 5 min), every hit resets the clock, so the cache stays warm on a single initial write. (If gaps exceeded 5 minutes, it would expire and re-write each time — the TTL gotcha.)
🔑 Key takeaways▼
  • Default TTL = 5 minutes; the clock resets free on every hit, so a busy prefix stays warm indefinitely on one write. ⚠ VERIFY
  • Extended TTL = 1 hour via "ttl":"1h", at a higher write cost (2× vs 1.25×). ⚠ VERIFY
  • Concurrency caveat: a cache entry "only becomes available after the first response begins" — fan-out before that pays a write per call.

④ The economics

The Economics — Multipliers & the Break-Even Derivation

The heart of the lesson. We derive the break-even from the multipliers, then work a real dollar example.

🧮 Section D⏱ ~20 minDomain 5

The multipliers, the algebra, and the dollars

Learning objectives
  • Recall the three caching multipliers and the per-model dollar prices
  • Derive the break-even from scratch and predict the crossover before revealing it
  • Work the 10-turn dollar example and recall the stacking discounts

D1 · The exact multipliers

"Prompt caching uses the following pricing multipliers relative to base input token rates: 5-minute cache write = 1.25× base input price; 1-hour cache write = 2× base input price; Cache read (hit) = 0.1× base input price." (source: pricing page) ⚠ VERIFY
OperationMultiplier (vs base input = 1.0×) ⚠ VERIFYMeaning
Base input1.0×Uncached token, processed normally
Cache write — 5-min1.25×Store the prefix; valid 5 minutes
Cache write — 1-hour2.0×Store the prefix; valid 1 hour
Cache read (hit / refresh)0.1×10% of standard input price

And the doc states the break-even conclusion directly: "A cache hit costs 10% of the standard input price, which means caching pays off after just one cache read for the 5-minute duration (1.25× write), or after two cache reads for the 1-hour duration (2× write)." (source: pricing page) We'll derive exactly that in D3 — but predict it first.

D3 · Break-even — derive it, don't memorize it

This is the spine of the lesson. Set it up with three symbols:

  • N = input tokens in the stable prefix
  • P = base input price per token
  • R = number of cache reads (reuses after the first write)

Sending the prefix R + 1 times total (1 first call + R reuses):

Cost models (normalized later by N·P)
Without caching:   Cost = (R + 1) · N · P

With 5-min cache:  Cost = N·P·1.25 + R·N·P·0.1  =  N·P·(1.25 + 0.1R)
With 1-hour cache: Cost = N·P·2.0  + R·N·P·0.1  =  N·P·(2.0  + 0.1R)
🤔 Predict first — before you see the algebra
For the 5-minute cache, how many reuses R does it take for caching to beat sending uncached every time? Is it 1, 2, 3, or more? (Write your guess down before revealing.)
Just R = 1 — the very first reuse. Here's the algebra, normalized by dividing both sides by N·P (they're positive, so the inequality direction holds):

5-minute: we want caching cheaper than no-cache →
1.25 + 0.1R < R + 1 → 1.25 − 1 < R − 0.1R → 0.25 < 0.9R → R > 0.278.
Since R is a whole number of reads, the smallest R that works is R = 1. Check it: cached path = 1.25 + 0.1(1) = 1.35 units; uncached = 1 + 1 = 2.0 units. Caching wins by 0.65 units on the very first reuse. This is exactly the doc's "pays off after just one cache read for the 5-minute duration." ✓
🤔 Predict first — now the 1-hour case
The 1-hour write is more expensive (2.0× vs 1.25×). Predict: does break-even move to R = 2, or further out?
R = 2 — the second reuse.
1-hour: 2.0 + 0.1R < R + 1 → 2.0 − 1 < 0.9R → 1.0 < 0.9R → R > 1.11.
The smallest whole R is R = 2. Check it: cached = 2.0 + 0.1(2) = 2.2 units; uncached = 1 + 2 = 3.0 units. Caching wins by 0.8 units at two reuses. Matches the doc's "after two cache reads for the 1-hour duration." ✓

The pattern name: a bigger write premium pushes break-even one reuse further out. The 1-hour TTL trades a higher up-front write for surviving longer gaps between reuses (Section E).

Steady state — where it all heads

As R grows large, the per-request cached cost approaches the read rate: 0.1× base input, versus 1.0× uncached. That is up to 90% savings on the cached prefix portion — and it's the arithmetic floor of the 0.1× read multiplier, not a marketing claim. The one-time write premium amortizes to nothing across heavy reuse.

🧭 Framing nuance — what "pays off after one read" really compares▼

"Pays off after one/two reads" compares cache write + reads against sending uncached every time. A single 5-min write costs only +0.25× over a normal send; one hit saves 0.9× (you pay 0.1 instead of 1.0), so a single hit more than repays the write. It is not comparing against "never sending the prefix" — you always have to send it; the question is whether you send it cached or raw.

D4 · The worked dollar example

Assumptions (explicit): Claude Opus 4.8 — $5 base input / $0.50 read / $6.25 5-min write per MTok. Stable prefix = 20,000 tokens (system + tool defs + a long reference doc). A 10-turn conversation = 1 write + 9 reuses within the refreshing 5-min window. Ignore the varying suffix/output (identical in both scenarios) to isolate the prefix cost.

ScenarioPrefix cost calculationTotal
No caching10 turns × 20,000 tok × $5/MTok$1.00
5-min caching — write (turn 1)20,000 × $6.25/MTok$0.125

(Note it's ~78.5%, not the steady-state 90% — the single write premium and the small 10-turn count haven't fully amortized yet.) Each additional reuse costs one-tenth of the uncached rate. Scale it linearly: at 1,000 such sessions/day, the prefix line item drops from ~$1,000/day to ~$215/day under the same assumptions. ⚠ VERIFY (Derived from the doc's per-MTok pricing; assumptions explicit.)

📎 Anthropic's own worked example (for cross-checking)▼

The pricing page's Managed-Agents example shows a 1-hour Opus 4.8 coding session where moving 40,000 of 50,000 input tokens to cache reads drops token+runtime cost from $0.705 to $0.525 — the cache-read line is "40,000 × $5 × 0.1 / 1,000,000 = $0.02" versus the uncached "$0.20." Same arithmetic, different numbers: 40K read tokens at 0.1× cost a tenth of what they'd cost uncached. (source: pricing page)

D5 · Stacking with other discounts

  • Batch API: 50% off input/output, and "Batch API and prompt caching discounts can be combined." A cached read inside a batch job compounds: 0.1× × 0.5×. ⚠ VERIFY
  • Data residency (inference_geo: "us"): a 1.1× multiplier applies on all categories including cache writes and reads (Opus/Sonnet 4.6+). ⚠ VERIFY
  • Fast mode: caching multipliers apply on top of fast-mode premium pricing. ⚠ VERIFY

(all source: pricing page)

❓ Quick check
Caching breaks even versus sending uncached after how many reads, for the two TTLs?
From the algebra: 5-min needs R > 0.278 → first reuse (R=1); 1-hour needs R > 1.11 → second reuse (R=2). The doc states it directly: "pays off after just one cache read for the 5-minute duration, or after two cache reads for the 1-hour duration."
❓ Quick check
In the 10-turn Opus 4.8 example, why is the saving ~78.5% rather than the full 90%?
90% is the steady-state floor (per-request cost → 0.1× as R → ∞). With only 1 write + 9 reads, the single 1.25× write still weighs on the average, so the realized saving is ~78.5%. More reuses push it toward 90%.
🔑 Key takeaways▼
  • Multipliers: 1.25× (5-min write), 2.0× (1-hour write), 0.1× (read). ⚠ VERIFY
  • Break-even (derived): 5-min wins at R > 0.278 → 1 read; 1-hour at R > 1.11 → 2 reads.
  • Steady state → per-request cost approaches 0.1× = up to 90% off the cached prefix (arithmetic, not marketing).
  • Worked example: 10-turn / 20K-prefix / Opus 4.8 → $1.00 → $0.215 (~78.5%).
  • Stacks with Batch (50%), data residency (1.1×), and fast-mode premiums. ⚠ VERIFY
⑤ When NOT to cache

When Caching Pays Off — and When It Costs You More

Caching isn't free. Below the minimum, single-shot, changing prefixes, and the 5-minute TTL gotcha all lose money.

⚖️ Section E⏱ ~13 minDomain 5 · Domain 4

The pay-off conditions and the loss conditions

Learning objectives
  • State the common thread of every "pays off" use case
  • Name the five situations where caching costs you more, including the 5-min TTL gotcha
  • Apply the TTL decision rule (5-min vs 1-hour vs don't-cache)

E1 · Where it pays off

Anthropic's recommended use cases (verbatim) all share one shape:

  • Conversational agents: "Reduce cost and latency for extended conversations, especially those with long instructions or uploaded documents."
  • Coding assistants: "Improve autocomplete and codebase Q&A by keeping relevant sections or a summarized version of the codebase in the prompt."
  • Large document processing: "Incorporate complete long-form material including images in your prompt without increasing response latency."
  • Detailed instruction sets: "…with prompt caching you can get even better performance by including 20+ diverse examples of high quality answers." (few-shot caching)
  • Agentic tool use: "Enhance performance for scenarios involving multiple tool calls and iterative code changes, where each step typically requires a new API call."
  • Knowledge base interaction: "…embedding the entire document(s) into the prompt, and letting users ask it questions."

(all source: prompt-caching doc) Common thread: a large, byte-identical prefix (system + tools + long context / few-shot) reused across many requests within the TTL window. That's the pattern to recognize on the exam and in your own agentic workflows.

E2 · Where it does NOT pay off

(Derived from the mechanism + economics — general reasoning grounded in the doc, not a verbatim Anthropic "don't" list.)

Anti-patternWhy it loses
Prefix below the minimum length<1,024 or <4,096 tokens (per model) → silently not cached, no error, no savings.
Single-shot / one-and-done promptsYou pay the 1.25× (or 2×) write and never collect a read → net more expensive than a plain call.
Constantly-changing prefixesIf the start of the prefix changes each call (timestamp, username in the system prompt), exact-match fails → you re-write every time.
Sporadic traffic + 5-min TTLReuse gaps exceed the TTL → cache expires before the next hit → every request is a fresh 1.25× write (worse than no caching).
Parallel fan-out before first response"A cache entry only becomes available after the first response begins" → concurrent first-calls each pay a write.
⚠️ The 5-minute TTL gotcha (high-yield)▼

This is the most counter-intuitive loss case. If your traffic is bursty — reuses arrive every 6–20 minutes, say — the 5-minute cache expires between hits. Each call then pays a fresh 1.25× write and never collects a single 0.1× read. You've made the prefix 25% more expensive than not caching at all. Mitigations: (1) switch to the 1-hour TTL (2× write, break-even at 2 reads) so the cache survives the gaps; or (2) keep the cache warm with a cheap heartbeat call inside the window. The fix is matching the TTL to the reuse cadence — which is the decision rule below.

E3 · The TTL decision rule (architect heuristic, grounded in the doc math)

Reuse cadenceChooseWhy
Frequent / within 5 minDefault 5-min (1.25× write)Break-even at 1 read; refresh-on-hit keeps it warm.
Bursty / minutes-to-an-hour apart1-hour TTL (2× write)Break-even at 2 reads; survives gaps that would expire the 5-min cache.
Rare / unpredictable / singleDon't cacheYou'd pay a write you never recoup.
🤔 Predict first
A support bot gets a question against a cached 15K-token policy doc roughly every 12 minutes. On the default 5-min TTL, is caching helping or hurting? What should you change?
Hurting. With 12-minute gaps, the 5-minute cache expires before each next question, so every call pays a fresh 1.25× write and never gets a 0.1× read — making the prefix 25% more expensive than no caching. Fix: switch to the 1-hour TTL (2× write). Now a write survives the 12-minute gaps, and after the second reuse you're ahead (break-even at R=2), trending toward 90% savings as volume grows. Alternatively, keep it warm with a heartbeat — but matching TTL to cadence is cleaner. This is the TTL decision rule in action.
❓ Quick check
Which scenario makes caching more expensive than not caching?
A single-shot prompt pays the 1.25× (or 2×) write premium and collects zero reads → net more expensive than a plain 1.0× call. The other three are textbook pay-off cases: large identical prefix, many reuses within the TTL.
❓ Quick check
Traffic reuses a prefix every ~20 minutes. The best move is to…
20-minute gaps exceed the 5-min TTL, so it would expire and re-write each time (refresh-on-hit only helps if hits arrive within the window). The 1-hour TTL (2× write, break-even at 2 reads) survives the gaps — the TTL decision rule for bursty traffic.
🔑 Key takeaways▼
  • Pays off when a large, byte-identical prefix is reused many times within the TTL (conversations, coding/agent loops, doc Q&A, 20+ few-shot).
  • Loses on: sub-minimum prefixes (silent), single-shot prompts, changing prefixes, sporadic traffic vs 5-min TTL, and pre-first-response fan-out.
  • 5-min TTL gotcha: bursty reuse expires the cache → fresh write each time → 25% worse than no caching.
  • TTL rule: frequent → 5-min; bursty → 1-hour; rare → don't cache.
⑥ Patterns & gotchas

Patterns & Gotchas — Structuring for Cache Hits

Stable-prefix-first, breakpoint placement, multi-breakpoint layering, and measuring with the usage fields.

🧩 Section F⏱ ~15 minDomain 5 · Domain 3/4

Design the prompt so the cache actually hits

Learning objectives
  • Apply stable-prefix-first structuring and correct breakpoint placement
  • Use multiple breakpoints for layered stability and the 20-position lookback
  • Read the usage fields to confirm caching works and diagnose busting

F1 · Structure stable-prefix-first

"Place static content (tool definitions, system instructions, context, examples) at the beginning of your prompt. Mark the end of the reusable content for caching using the cache_control parameter."
"Place cached content at the prompt's beginning for best performance." (source: prompt-caching doc)

This is a Domain 3 prompt-structuring rule with a cost consequence: arrange the request so the most stable, most expensive-to-recompute material sits at the front, then mark the breakpoint where the stable part ends.

F2 · Put the breakpoint on the last identical block

For a prompt with a static prefix and a varying suffix (timestamps, per-request context, the incoming message), put the breakpoint at the end of the prefix, not the varying block. (source: prompt-caching doc)

🚫 Anti-pattern: dynamic content in the static prefix▼

Injecting the current date, user name, or session variables into the static system prompt changes the prefix byte-for-byte every call → exact-match fails → you re-write every time and never hit. Move dynamic content after the breakpoint (into the user turn or the varying suffix). (source: structuring guidance; corroborated SECONDARY PromptHub)

Anti-pattern vs fix (pseudocode — placeholders escaped)
# ✗ ANTI-PATTERN: dynamic date inside the cached system prompt
system = [
  { "type": "text",
    "text": "You are a support agent. Today is <current_date>. <long stable policy doc>",
    "cache_control": {"type": "ephemeral"} }
]   # the date changes every call → prefix busts → re-write every time

# ✓ FIX: stable text cached; dynamic date moved after the breakpoint
system = [
  { "type": "text",
    "text": "You are a support agent. <long stable policy doc>",
    "cache_control": {"type": "ephemeral"} }   # breakpoint at end of STABLE block
]
messages = [
  { "role": "user",
    "content": "Today is <current_date>. <the user question>" }  # varies AFTER the cache
]

F3 · Cache the system + tools first

Because invalidation cascades tools → system → messages, cache the most-stable, most-expensive-to-recompute layers (tool definitions and system prompt) and avoid editing tool defs casually — a tool-def change invalidates the entire cache (Section B6). In an agent loop, churning tool definitions is a silent money leak.

F4 · Use multiple breakpoints for layered stability

Up to 4 breakpoints let you cache layers with different change cadences: one after tools+system (rarely changes), another after a long doc (changes per-session), another after accumulated conversation. The 20-position lookback means earlier layers still hit even when a later layer changes. ⚠ VERIFY (source: prompt-caching doc)

F5 · Combine with Batch, and watch residency

Stack caching with the 50% Batch discount for non-time-sensitive bulk jobs; remember the 1.1× inference_geo:"us" premium applies to cache reads/writes too. ⚠ VERIFY (source: pricing page)

F6 · Measure with the usage fields

Confirm caching works by reading the response usage object. Field names (verbatim) ⚠ VERIFY:

response.usage
"usage": {
  "cache_creation_input_tokens": 5120,
  "cache_read_input_tokens": 1800,
  "input_tokens": 2048,
  "output_tokens": 503
}
  • cache_creation_input_tokens — tokens written to the cache when creating a new entry.
  • cache_read_input_tokens — tokens retrieved from the cache for this request.
  • input_tokens — "…only the tokens that come after the last cache breakpoint" (the uncached suffix).
  • Identity: total_input_tokens = cache_read_input_tokens + cache_creation_input_tokens + input_tokens.

(source: prompt-caching doc) Health signal: high cache_read + low cache_creation after warm-up = caching working; persistent high cache_creation = the prefix is busting (something earlier is changing) or the TTL keeps expiring. This is your debugging instrument — watch these two numbers.

F7 · The gotchas checklist

  • Min-length not met → silent no-cache, no error.
  • Anything before a breakpoint changes → whole downstream cache busts.
  • Ordering is fixed tools → system → messages; can't cache a non-prefix slice.
  • Caches are workspace-isolated (as of 2026-02-05 on Claude API / Claude Platform on AWS / MS Foundry beta; org-level on Bedrock/Vertex) — never shared across orgs. ⚠ VERIFY
  • Caching does not enlarge the context window; it's a cost/latency optimization only.
  • Thinking blocks, citation sub-blocks, and empty text blocks cannot be cached.

(all source: prompt-caching doc)

🤔 Predict first
After warm-up, your agent's usage still shows high cache_creation_input_tokens on every call and near-zero cache_read_input_tokens. What's almost certainly wrong?
The prefix is busting on every call — either something earlier in the prefix is changing (a timestamp/username in the system prompt, or churning tool definitions that bust the whole cache), or the TTL keeps expiring between calls (sporadic traffic vs 5-min). Persistent high cache_creation means you're re-writing, not reading. Fixes: move dynamic content after the breakpoint (F2), stop editing tool defs mid-session (F3), or switch to the 1-hour TTL (E3). The usage fields are exactly the instrument that surfaces this.
❓ Quick check
Which usage field reports only the tokens after the last cache breakpoint (the uncached suffix)?
input_tokens counts "only the tokens that come after the last cache breakpoint." Cached tokens are split into cache_read_input_tokens (hits) and cache_creation_input_tokens (writes); the three sum to total input.
❓ Quick check
Why move an injected timestamp out of the cached system prompt and into the user turn?
"Cache hits require 100% identical prompt segments." A per-call timestamp in the cached prefix busts the match every time. Put dynamic content after the breakpoint so the stable prefix keeps hitting.
🔑 Key takeaways▼
  • Stable-prefix-first: static content (tools, system, context, examples) at the front; breakpoint at the end of the last identical block.
  • Keep dynamic content (dates, names, the user message) after the breakpoint, or the prefix busts every call.
  • Multi-breakpoint layering (≤4) caches layers by change cadence; the 20-position lookback lets earlier layers still hit.
  • Read cache_read_input_tokens vs cache_creation_input_tokens to confirm hits and diagnose busting.

Q1
A cache read (hit) is billed at what multiplier of base input?
A read is 0.1× — "a cache hit costs 10% of the standard input price." Writes are 1.25× (5-min) and 2× (1-hour). ⚠ VERIFY the multipliers — they drift.
Q2
Caching breaks even versus uncached after…
5-min: 1.25+0.1R < R+1 → R > 0.278 → R=1. 1-hour: 2.0+0.1R < R+1 → R > 1.11 → R=2. Matches the doc.
Q3
The default cache TTL, and what happens on each hit, is…
Default = 5 minutes; "the cache is refreshed for no additional cost each time the cached content is used." A busy prefix stays warm on one write.
Q4
Editing one tool definition mid-session invalidates…
"Changes at each level invalidate that level and all subsequent levels." Tools is first, so a tool-def edit busts everything — the most expensive change.
Q5
Which case makes caching more expensive than not caching?
Single-shot pays the 1.25×/2× write and collects zero reads → net more than a plain 1.0× call. The others reuse a stable prefix many times.
Q6
Bursty traffic reuses a prefix every ~15 minutes. The 5-min TTL gotcha means…
15-min gaps > 5-min TTL → expiry before the next hit → fresh 1.25× write each time (worse than no caching). Fix: 1-hour TTL (break-even at 2 reads) or keep warm.
Q7
The minimum cacheable length below which a prefix is silently not cached is…
Minimums are version-tied. Opus 4.8 = 1,024; Haiku 4.5 = 4,096 (NOT the stale 2,048 — that's retired Haiku 3.5). Below the minimum: "processed without caching, and no error is returned."
Q8
Where should you place the cache_control breakpoint relative to a varying suffix?
Mark "the last content block you want included" — the end of the stable prefix. Keep dynamic content after it, or the 100%-identical match breaks and the cache busts.
Q9
Persistent high cache_creation_input_tokens with near-zero cache_read_input_tokens after warm-up indicates…
High creation + low read = re-writing, not reading. Cause: dynamic content in the prefix, churning tool defs (busts all), or TTL expiry. The usage fields are your debugging instrument.
Q10
In the steady state of heavy reuse, the cached prefix cost approaches…
As R grows, per-request cost → the 0.1× read rate vs 1.0× uncached → up to 90% off the cached portion. The one-time write premium amortizes away. It's arithmetic, not a marketing headline.
