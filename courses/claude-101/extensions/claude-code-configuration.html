Claude Code Configuration & Workflows

Claude Code is configured along two axes — scope (managed → user → project → local) and feature (settings, memory, permissions, hooks, skills, subagents, plugins, MCP). Master the mental model once and every config question becomes "which scope, which feature, and does it override or merge?" This lesson walks the whole surface, ending in the hooks contract and the headless automation patterns an architect ships in CI.

① The model

The Configuration Model & the .claude/ Directory

Two axes — scope × feature. Four scopes, and a map of which feature lives where.

🗺️ Section A⏱ ~12 minDomain 2 · Domain 4

Two axes: where config lives, and who it affects

Learning objectives
  • Describe the two axes of Claude Code configuration — scope and feature
  • Name the four scopes and who each affects, top of the org down to one repo
  • Map each feature (settings, subagents, MCP, plugins, CLAUDE.md) to its user/project/local location

Why before how: the two-axis mental model

lesson framing, not a verbatim Anthropic quote Almost every Claude Code config question is really asking two things at once: at what scope does this live, and which feature is it. Get those two coordinates and you can place any file, predict whose machine it affects, and reason about which one wins. The scope axis runs managed (enterprise) → user → project → local. The feature axis is settings, memory, subagents, skills/commands, hooks, MCP, plugins, output styles. The .claude/ directory is the per-scope home; managed config lives in OS-level policy locations instead.

📂 --add-dir grants file access, NOT a full config root▼

lesson framing Verbatim: "Adding a directory extends where Claude can read and edit files. It does not make that directory a full configuration root." Only Skills (with live reload), enabledPlugins/extraKnownMarketplaces, and (conditionally) CLAUDE.md load from --add-dir directories; subagents, commands, output styles, and hooks do not. (source: Permissions)

🔑 Key takeaways▼
  • Two axes: scope (managed → user → project → local) × feature (settings, memory, subagents, skills, hooks, MCP, plugins, output styles).
  • Managed = IT-deployed, all users, can't be overridden; local = you, this repo, gitignored.
  • .claude/ is the per-scope home; managed config lives in OS policy locations.
  • MCP servers are the odd one out — local/user both store in ~/.claude.json, project in .mcp.json.
② Memory

Memory — CLAUDE.md

Load order broad → specific. Concatenation, not override. @-imports (4 hops), rules, managed, and auto memory.

🧠 Section B⏱ ~12 minDomain 2 · Domain 1

How Claude remembers your project — and why it's context, not config

Learning objectives
  • State the CLAUDE.md load order and explain that files concatenate rather than override
  • Explain @-imports and the 4-hop depth limit
  • Distinguish CLAUDE.md (context) from a PreToolUse hook (enforcement)
  • Recognise .claude/rules/, managed claudeMd, and auto memory as separate mechanisms

Why before how: memory is context, not a rulebook

CLAUDE.md is the project's running briefing that Claude reads at the start of every conversation. The mental model that trips people up: it is concatenated context, not enforced configuration. Multiple CLAUDE.md files don't "override" each other — they are all stitched together, broadest scope first. So if you want a rule to be guaranteed (e.g. block a dangerous command), CLAUDE.md is the wrong tool — you need a permission rule or a PreToolUse hook.

Load order — broadest to most specific (verbatim table)

ScopeLocationShared with
Managed policymacOS: /Library/Application Support/ClaudeCode/CLAUDE.md; Linux & WSL: /etc/claude-code/CLAUDE.md; Windows: C:\Program Files\ClaudeCode\CLAUDE.mdAll users in organization
User instructions~/.claude/CLAUDE.mdJust you (all projects)
Project instructions./CLAUDE.md or ./.claude/CLAUDE.mdTeam members via source control
Local instructions./CLAUDE.local.md (add to .gitignore)Just you (current project)
"The table below lists them in load order, from broadest scope to most specific, so a project instruction appears in context after a user instruction." (source: Memory)

How loading actually works (verbatim)

"Claude Code reads CLAUDE.md files by walking up the directory tree from your current working directory... All discovered files are concatenated into context rather than overriding each other. Across the directory tree, content is ordered from the filesystem root down to your working directory... Within each directory, CLAUDE.local.md is appended after CLAUDE.md, so your personal notes are the last thing Claude reads at that level." (source: Memory)

Subdirectory files are discovered lazily: "Instead of loading them at launch, they are included when Claude reads files in those subdirectories." (source: Memory)

🚧 Trap — memory is context, not enforced config▼

Verbatim: "Both are loaded at the start of every conversation. Claude treats them as context, not enforced configuration. To block an action regardless of what Claude decides, use a [PreToolUse hook] instead." (source: Memory) Managed-policy CLAUDE.md cannot be excluded by individual settings. Exam translation: "Put it in CLAUDE.md" never guarantees a behavior — for a hard block you need a permission deny rule or a PreToolUse hook that exit 2s (Section E).

@-imports — and the 4-hop limit (verbatim)

"CLAUDE.md files can import additional files using @path/to/import syntax. Imported files are expanded and loaded into context at launch... Both relative and absolute paths are allowed. Relative paths resolve relative to the file containing the import... Imported files can recursively import other files, with a maximum depth of four hops." (source: Memory)
CLAUDE.md import lines (verbatim)
See @README for project overview and @package.json for available npm commands for this project.

# Additional Instructions
- git workflow @docs/git-instructions.md

Size guidance (verbatim): "target under 200 lines per CLAUDE.md file" and "Bloated CLAUDE.md files cause Claude to ignore your actual instructions!" (sources: Memory; Best practices)

Three more memory mechanisms

  • .claude/rules/ (path-scoped rules): rules without paths frontmatter "are loaded at launch with the same priority as .claude/CLAUDE.md." Path-scoped rules use a YAML paths glob list and "only apply when Claude is working with files matching the specified patterns." User-level rules in ~/.claude/rules/ "are loaded before project rules, giving project rules higher priority." (source: Memory)
  • Managed claudeMd: embeds managed CLAUDE.md content directly in managed-settings.json; "honored: managed and policy settings only." (source: Memory)
  • Auto memory ⚠ VERIFY — machine-local notes Claude writes itself, stored per-repo at ~/.claude/projects/<project>/memory/ with a MEMORY.md index. "The first 200 lines of MEMORY.md, or the first 25KB, whichever comes first, are loaded at the start of every conversation." On by default (autoMemoryEnabled); requires v2.1.59+. (source: Memory)
managed claudeMd in managed-settings.json (verbatim)
{
  "claudeMd": "Always run `make lint` before committing.\nNever push directly to main."
}

AGENTS.md: "Claude Code reads CLAUDE.md, not AGENTS.md." Bridge with an @AGENTS.md import or a symlink. (source: Memory)

🤔 Predict first
A teammate adds "NEVER run rm -rf" to the project CLAUDE.md and tells the team they're now protected. Predict: is the team actually protected from a destructive deletion?
No. CLAUDE.md is "context, not enforced configuration." Claude usually will follow the instruction, but nothing guarantees it. For a hard block you need a permission deny rule (e.g. Bash(rm *) in deny) or a PreToolUse hook that exit 2s. The verbatim guidance: "To block an action regardless of what Claude decides, use a PreToolUse hook instead." (source: Memory)
❓ Quick check
When two CLAUDE.md files (user and project) both apply, how does Claude Code combine them?
Verbatim: "All discovered files are concatenated into context rather than overriding each other... ordered from the filesystem root down to your working directory." Memory "precedence" is ordering, not override — and it's context, not enforced config.
🔑 Key takeaways▼
  • Concatenation, not override: managed → user → project (root→cwd) → local; CLAUDE.local.md appended last per directory.
  • Context, not config: for a guaranteed block use a permission deny or a PreToolUse hook.
  • @-imports expand at launch, max 4 hops; keep files under ~200 lines.
  • Separate mechanisms: .claude/rules/ (path-scoped), managed claudeMd, and auto memory (MEMORY.md).
③ settings.json

settings.json — Hierarchy & Precedence

The 5-level ladder. The exam trap: settings override, permission rules merge. Key fields, governance, env, reload.

🪜 Section C⏱ ~13 minDomain 2 · Domain 4

Which setting wins — and the one place precedence breaks

Learning objectives
  • Recite the 5-level settings precedence ladder, highest to lowest
  • State the exam trap: settings override by precedence, but permission rules merge across scopes
  • Recognise key fields, managed/governance fields, and which keys live-reload vs need a restart

The precedence ladder (verbatim)

  1. Managed (highest) — can't be overridden by anything
  2. Command line arguments — temporary session overrides
  3. Local — overrides project and user settings
  4. Project — overrides user settings
  5. User (lowest) — applies when nothing else specifies the setting
⚖️ Exam trap — settings override, permission rules merge▼

Verbatim: "Permission rules behave differently because they merge across scopes rather than override." (source: Settings) This is one of the most frequently tested Domain-2 facts. A plain setting like model uses the ladder — the higher level wins. But permissions.allow/ask/deny arrays from every scope are combined, then evaluated deny → ask → allow (Section D). So a deny in user settings still applies even if project settings say allow.

Key fields (verbatim selections)

  • model — "Override the default model to use for Claude Code"
  • env — "Environment variables applied to every session and to subprocesses Claude Code spawns"
  • permissions — allow/deny/ask arrays; also defaultMode, additionalDirectories, disableBypassPermissionsMode
  • hooks — "Configure custom commands to run at lifecycle events"
  • apiKeyHelper, statusLine, outputStyle
  • autoMemoryEnabled (default true), autoMemoryDirectory, claudeMd (managed only), claudeMdExcludes
  • cleanupPeriodDays (default 30), includeCoAuthoredBy (Deprecated — "Use attribution instead"), attribution
  • enableAllProjectMcpServers, enabledMcpjsonServers, disabledMcpjsonServers
Managed-only / governance fields (verbatim selections)
  • allowManagedHooksOnly — "only managed hooks, SDK hooks, and force-enabled plugin hooks are loaded"
  • allowManagedPermissionRulesOnly — "prevents user and project settings from defining allow, ask, or deny permission rules"
  • allowManagedMcpServersOnly, allowedMcpServers, deniedMcpServers
  • strictKnownMarketplaces, blockedMarketplaces, strictPluginOnlyCustomization ("true locks all four surfaces; an array such as ["skills", "hooks"] locks only the named ones")
  • forceLoginMethod, forceLoginOrgUUID, requiredMinimumVersion, requiredMaximumVersion, forceRemoteSettingsRefresh
settings.json (verbatim example)
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "permissions": {
    "allow": [
      "Bash(npm run lint)",
      "Bash(npm run test *)",
      "Read(~/.zshrc)"
    ],
    "deny": [
      "Bash(curl *)",
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)"

  "env": {
    "CLAUDE_CODE_ENABLE_TELEMETRY": "1",
    "OTEL_METRICS_EXPORTER": "otlp"
  }
}

Reload behavior (verbatim)

"Claude Code watches your settings files and reloads them when they change, so edits to most keys apply to the running session without a restart." Live-reload keys include permissions, hooks, apiKeyHelper. Read once at session start (apply on next restart): model (use /model to switch mid-session) and outputStyle ("rebuilt on /clear or restart"). (source: Settings) /status shows active "Setting sources" for the current session.

🪟 Managed-settings deployment locations ⚠ VERIFY▼
  • macOS: /Library/Application Support/ClaudeCode/managed-settings.json (+ managed-settings.d/), or the com.anthropic.claudecode plist
  • Linux & WSL: /etc/claude-code/managed-settings.json (+ managed-settings.d/)
  • Windows: registry HKLM\SOFTWARE\Policies\ClaudeCode (admin) / HKCU\SOFTWARE\Policies\ClaudeCode (user, lowest); file C:\Program Files\ClaudeCode\managed-settings.json. "Legacy path (no longer supported as of v2.1.75): C:\ProgramData\ClaudeCode\managed-settings.json"
⚠ VERIFY: the Windows legacy path retirement (v2.1.75) and other OS policy locations are version-noted — re-verify for the current release. Selected env vars set via env: CLAUDE_CODE_ENABLE_TELEMETRY, OTEL_METRICS_EXPORTER, CLAUDE_CODE_DISABLE_AUTO_MEMORY, ANTHROPIC_MODEL, DISABLE_AUTOUPDATER, CLAUDE_CODE_USE_POWERSHELL_TOOL.
❓ Quick check
Project settings set model: opus and add an allow rule; user settings set model: sonnet and a deny rule. What happens?
Plain settings like model use precedence — project overrides user, so opus wins. But "permission rules... merge across scopes rather than override," so the user deny and project allow both apply, evaluated deny → ask → allow. The deny still bites.
🔑 Key takeaways▼
  • Ladder: Managed > CLI args > Local > Project > User.
  • Trap: settings override; permission rules merge across scopes, then evaluate deny → ask → allow.
  • Live-reload: permissions, hooks, apiKeyHelper. Restart/needs /clear: model, outputStyle.
  • Governance lives in managed-only keys (allowManaged*Only, required*Version, marketplace locks).
④ Permissions

Permissions — allow / ask / deny

deny → ask → allow. Bare vs scoped. Six modes. The //absolute trap and fragile-argument warning.

🔐 Section D⏱ ~15 minDomain 2 · Domain 4

The decision flow that decides whether a tool runs

Learning objectives
  • Walk the deny → ask → allow evaluation order and explain why deny is absolute
  • Distinguish a bare tool name from a scoped pattern, and apply the absolute-path trap
  • Name the six permission modes and the permission precedence ladder

Why before how: three rule types, evaluated in a fixed order

"Allow rules let Claude Code use the specified tool without manual approval. Ask rules prompt for confirmation... Deny rules prevent Claude Code from using the specified tool." (source: Permissions)
"Rules are evaluated in order: deny -> ask -> allow. The first matching rule wins, so deny rules always take precedence." ... "If a tool is denied at any level, no other level can allow it." (source: Permissions)

Bare tool name vs scoped pattern (verbatim)

"A bare tool name like Bash removes the tool from Claude's context entirely, so Claude never sees it. A scoped rule like Bash(rm *) leaves the tool available and blocks matching calls when Claude attempts them." (source: Permissions)

Rule syntax — Tool or Tool(specifier)

  • Bash(npm run build) → exact command; Bash(npm run test *) → prefix. The space before * enforces a word boundary: Bash(ls *) matches ls -la but not lsof; Bash(ls*) matches both. A :* suffix ≡ a trailing * (only at the end of a pattern).
  • Read(./.env), Edit(/src/**/*.ts) — gitignore-style patterns with four anchors: //path (absolute), ~/path (home), /path (relative to project root), path/./path (relative to cwd).
  • WebFetch(domain:example.com); MCP: mcp__puppeteer (whole server), mcp__puppeteer__* (wildcard), mcp__puppeteer__puppeteer_navigate (single tool); Agent(Explore), Agent(my-custom-agent).
🪤 The //absolute trap (high-yield)▼

Verbatim: "A pattern like /Users/alice/file is NOT an absolute path. It's relative to the project root. Use //Users/alice/file for absolute paths." A single leading slash means project-root-relative; you need a double slash for a true absolute path. This is a favorite exam gotcha.

🧨 Shell-operator awareness + "argument constraints are fragile"▼

Verbatim: "Claude Code is aware of shell operators, so a rule like Bash(safe-cmd *) won't give it permission to run the command safe-cmd && other-cmd. The recognized command separators are &&, ||, ;, |, |&, &, and newlines." But also: "Bash permission patterns that try to constrain command arguments are fragile." For reliable URL filtering, deny curl/wget and use WebFetch(domain:...), or use a PreToolUse hook. (source: Permissions)

The six permission modes (verbatim table)

Set via defaultMode in settings or the --permission-mode flag.

ModeDescription
default"Standard behavior: prompts for permission on first use of each tool"
acceptEdits"Automatically accepts file edits and common filesystem commands (mkdir, touch, mv, cp, etc.) for paths in the working directory or additionalDirectories"
plan"Plan Mode: Claude reads files and runs read-only shell commands to explore but does not edit your source files"
auto ⚠ VERIFY"Auto-approves tool calls with background safety checks... Currently a research preview"
dontAsk ⚠ VERIFY"Auto-denies tools unless pre-approved via /permissions or permissions.allow rules"

⚠ VERIFY: default, acceptEdits, plan, bypassPermissions are stable; auto (research preview) and dontAsk are newer and could be renamed or removed (--enable-auto-mode was already removed in v2.1.111). Admins can block bypass via permissions.disableBypassPermissionsMode: "disable" and auto via permissions.disableAutoMode.

Permission precedence (verbatim)

  1. Managed settings — cannot be overridden by any other level, including command line arguments
  2. Command line arguments
  3. Local project settings (.claude/settings.local.json)
  4. Shared project settings (.claude/settings.json)
  5. User settings (~/.claude/settings.json)

"a managed settings deny cannot be overridden by --allowedTools, and --disallowedTools can add restrictions beyond what managed settings define." lesson framing Permissions are client-enforced, not model-enforced — "Permission rules are enforced by Claude Code, not by the model." For OS-level enforcement use sandboxing (sandbox.enabled), which complements permissions. (source: Permissions)

🤔 Predict first
You want to stop Claude from fetching a malicious URL. A teammate proposes Bash(curl https://safe.example.com *) in allow to "only allow the safe domain." Predict whether this reliably restricts which URLs get fetched.
No — it's fragile. "Bash permission patterns that try to constrain command arguments are fragile." Argument-level allow rules for curl don't reliably bound the URL. The robust pattern: deny curl/wget outright and route web access through WebFetch(domain:safe.example.com), or enforce with a PreToolUse hook. (source: Permissions)
❓ Quick check
User settings allow a tool; project settings deny the same tool. With deny → ask → allow evaluation, what happens?
Rules merge across scopes, then evaluate deny → ask → allow with first-match-wins. Deny is checked first: "If a tool is denied at any level, no other level can allow it." The deny wins.
🔑 Key takeaways▼
  • deny → ask → allow, first match wins; deny is absolute across all levels.
  • Bare Bash removes the tool from context; scoped Bash(rm *) leaves it available but blocks matches.
  • Single / = project-root-relative; // = true absolute path.
  • Six modes; auto/dontAsk are newer/volatile. Argument-constraint Bash rules are fragile — prefer WebFetch(domain:) or a hook.
  • Permission precedence: Managed > CLI > Local > Project > User. Client-enforced; sandbox adds OS-level enforcement.
⑤ Hooks

Hooks — Deterministic Lifecycle Control

The richest section. Events, matchers, the stdin contract, the exit-code table (only 2 blocks), the JSON output schema, and security.

🪝 Section E⏱ ~22 minDomain 2 · Domain 4 · Domain 1

The contract that gives you guaranteed, deterministic control

Learning objectives
  • Define what a hook is and where the classic stable events fire in the lifecycle
  • Read the matcher rules, the stdin input contract, and the exit-code table
  • Explain why only exit 2 blocks — and write a PreToolUse permissionDecision and a SessionStart additionalContext

Why before how: hooks are the enforcement layer

"Claude Code hooks are user-defined shell commands that execute at various points in Claude Code's lifecycle and provide deterministic control over Claude Code's behavior." (source: Hooks)

CLAUDE.md asks Claude to behave; permissions gate tool calls by pattern; hooks run your own code at lifecycle moments and can deterministically allow, block, inject context, or react. They're configured under the hooks key in settings.json (or hooks/hooks.json in a plugin), and you can browse them with /hooks.

The event roster — stable subset vs the growing set

The full reference lists roughly 30 hook events. The classic stable subset — the safest to learn for the exam — is: PreToolUse, PostToolUse, UserPromptSubmit, Notification, Stop, SubagentStop, PreCompact, SessionStart, SessionEnd. ⚠ VERIFY

⚠ VERIFY — the full list is volatile and likely to keep growing. Newer events include Setup, UserPromptExpansion, PermissionRequest, PermissionDenied, PostToolUseFailure, PostToolBatch, MessageDisplay, SubagentStart, TaskCreated, TaskCompleted, StopFailure, TeammateIdle, InstructionsLoaded, ConfigChange, CwdChanged, FileChanged, WorktreeCreate, WorktreeRemove, PostCompact, Elicitation, ElicitationResult. Re-verify against the live Hooks reference. (source: Hooks)

Matchers (verbatim rules)

Matcher valueEvaluated as
"*", "", or omittedMatch all
Only letters, digits, _, |Exact string or |-separated list (e.g. Bash, Edit|Write)

Tool-name matching applies to PreToolUse, PostToolUse, PostToolUseFailure, PermissionRequest, PermissionDenied. The exit-code table (verbatim) — the single most-tested hook fact

Exit codeBehaviorJSON processing
0SuccessParses stdout for JSON output
2Blocking errorIgnores stdout/JSON; feeds stderr to Claude as error
Other (1, 3, …)Non-blocking errorstderr shown in transcript; execution continues
🛑 Security-critical — only exit 2 blocks (exit 1 does NOT)▼

Verbatim: "For most hook events, only exit code 2 blocks the action. Claude Code treats exit code 1 as a non-blocking error and proceeds with the action, even though 1 is the conventional Unix failure code. If your hook is meant to enforce a policy, use exit 2." (source: Hooks) This is counter-intuitive — almost every other CLI tool treats exit 1 as failure. A policy hook that exit 1s will let the action through.

Which events can block via exit 2 (selections): PreToolUse (blocks the tool call), PermissionRequest (denies), UserPromptSubmit (blocks + erases the prompt), Stop/SubagentStop (prevents stopping), PreCompact (blocks compaction), ConfigChange (blocks the change except policy_settings), WorktreeCreate (any non-zero fails), PostToolBatch (stops the agentic loop before the next model call). Cannot block: PostToolUse/PostToolUseFailure ("tool already ran"), StopFailure, Notification, SessionStart, SessionEnd.

SessionStart can inject context (verbatim): hookSpecificOutput.additionalContext adds context to Claude's window; "Output strings ... are capped at 10,000 characters."

⚠️ Hook decisions do NOT bypass permission rules▼

Verbatim: "Hook decisions do not bypass permission rules. Deny and ask rules are evaluated regardless of what a PreToolUse hook returns... A hook that exits with code 2 stops the tool call before permission rules are evaluated." (source: Permissions) So a PreToolUse hook returning permissionDecision: "allow" still can't override a deny rule — but an exit 2 stops the call earlier, before the permission flow runs at all.

A complete hook config (verbatim)

PreToolUse hook in settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "if": "Bash(rm *)",
            "command": "${CLAUDE_PROJECT_DIR}/.claude/hooks/block-rm.sh",
            "args": []
          }
        ]
      }
    ]
  }
}
🔒 Security notes (verbatim)▼
  • "On macOS and Linux, command hooks run in their own session without a controlling terminal as of v2.1.139... To surface a message to the user on any platform, return systemMessage in JSON output."
  • Bash if-matching "fails open (runs hook regardless) when Bash command cannot be parsed. Use the [permission system] rather than a hook to enforce hard allow/deny."
  • Prompt-injection guidance for additionalContext: "Write the text as factual statements rather than imperative system instructions."
  • Managed governance: allowManagedHooksOnly, allowedHttpHookUrls, httpHookAllowedEnvVars, disableAllHooks.

Hooks can also be declared inside a subagent's frontmatter (e.g. a db-reader agent with a PreToolUse matcher running validate-readonly-query.sh). Note: plugin subagents do not support hooks (Section F.2).

🤔 Predict first
A security team writes a PreToolUse hook that runs a policy check and exit 1s when the command is forbidden, logging the violation to stderr. Predict: does the forbidden command get blocked?
No — the command runs anyway. "Only exit code 2 blocks the action. Claude Code treats exit code 1 as a non-blocking error and proceeds with the action." The hook needs exit 2 to block. Worse, if the enforcement relies on Bash if-matching that can't parse the command, it "fails open." For hard allow/deny, the docs say use the permission system, not a hook. (source: Hooks)
❓ Quick check
You need to guarantee that a dangerous tool call is stopped before it executes. Which event + exit code does the job?
PostToolUse can't block ("tool already ran"). Exit 1 is non-blocking. PreToolUse fires before the tool; exit 2 blocks the call before permission rules even run, and permissionDecision: "deny" denies it. SessionStart only injects context.
🔑 Key takeaways▼
  • Hooks = deterministic control via your own shell commands at lifecycle events.
  • Exit 0 = success (parse stdout JSON); exit 2 = block (stderr → Claude); exit 1/other = non-blocking. Only 2 blocks.
  • PreToolUse can block; PostToolUse cannot. Stable subset is 9 events; the full roster (~30) is volatile.
  • Matchers: bare/|-list = exact; other chars = regex. PreToolUse uses permissionDecision; SessionStart uses additionalContext (cap 10,000 chars).
  • For hard allow/deny use the permission system — Bash if-matching fails open.

F.1 · Skills (custom commands merged in)

"Custom commands have been merged into skills. A file at .claude/commands/deploy.md and a skill at .claude/skills/deploy/SKILL.md both create /deploy and work the same way. Your existing .claude/commands/ files keep working." (source: Skills)

Skill scopes & precedence: Enterprise (managed) → Personal (~/.claude/skills/) → Project (.claude/skills/) → Plugin (<plugin>/skills/). "When skills share the same name across levels, enterprise overrides personal, and personal overrides project... if a skill and a command share the same name, the skill takes precedence." The command name comes from the directory under .claude/skills/ (deploy-staging/SKILL.md → /deploy-staging), the file name under .claude/commands/, and is namespaced plugin-name:skill-name for plugin skills.

Frontmatter (selections): name, description (recommended; "truncated at 1,536 characters in the skill listing"), when_to_use, argument-hint, arguments, disable-model-invocation: true ("prevent Claude from automatically loading this skill... trigger manually with /name"), user-invocable: false ("hide from the / menu... for background knowledge"), allowed-tools, model, context: fork, hooks, paths, shell. Substitutions: $ARGUMENTS (all args; appended as ARGUMENTS: <value> if not present), $ARGUMENTS[N], $N (0-based: $0 = first arg), $name, ${CLAUDE_SKILL_DIR}. Dynamic context: !`git diff HEAD` "replaces the line with its output before Claude sees the skill content."

F.3 · Plugins & marketplaces

A plugin bundles .claude-plugin/ (holds plugin.json only), skills/, commands/, agents/, hooks/, .mcp.json, .lsp.json, monitors/, bin/, and a limited settings.json. "Common mistake: Don't put commands/, agents/, skills/, or hooks/ inside the .claude-plugin/ directory. Only plugin.json goes inside .claude-plugin/."

plugin.json manifest (verbatim)
{
  "name": "my-first-plugin",
  "description": "A greeting plugin to learn the basics",
  "version": "1.0.0",
  "author": { "name": "Your Name" }
}

"name — Unique identifier and skill namespace. Skills are prefixed with this (e.g., /my-first-plugin:hello)." If version is omitted and the plugin is distributed via git, "the commit SHA is used and every commit counts as a new version." Dev: claude --plugin-dir ./my-plugin (also accepts a .zip), /reload-plugins.

.claude-plugin/marketplace.json (verbatim)
{
  "name": "my-plugins",
  "owner": { "name": "Your Name" },
  "plugins": [
    {
      "name": "quality-review-plugin",
      "source": "./plugins/quality-review-plugin",
      "description": "Adds a quality-review skill for quick code reviews"
    }
  ]
}

Install flow: /plugin marketplace add ./my-marketplace then /plugin install quality-review-plugin@my-plugins; refresh with /plugin marketplace update. Anthropic maintains claude-plugins-official (auto-available) and a community marketplace you add explicitly.

"Output styles change how Claude responds, not what Claude knows. They modify the system prompt..." Built-in: Default, Proactive, Explanatory, Learning.   • Pick the surface: knowledge → Skill; isolation → Subagent; guarantee → Hook; integration → MCP; share/version → Plugin.

MCP in Claude Code

Transports, three install scopes (with the rename trap), .mcp.json, scope precedence, governance.

🔗 Section G⏱ ~12 minDomain 2 · Domain 4 · Domain 5

Connecting Claude Code to external tools and data

Learning objectives
  • Name the four MCP transports and which is deprecated / recommended
  • State the three install scopes — and the local-was-project / user-was-global rename trap
  • Recall .mcp.json, scope precedence (no field merging), and governance keys

Transports (verbatim commands)

  • HTTP (recommended for remote): claude mcp add --transport http <name> <url>. In JSON, type accepts streamable-http as an alias for http.
  • SSE (deprecated): claude mcp add --transport sse <name> <url>
  • stdio (local): claude mcp add [options] <name> -- <command> [args...] — "the -- (double dash) separates Claude's own options... from the command and arguments that run the server."
  • WebSocket: type: "ws" via .mcp.json or claude mcp add-json (not supported by --transport).

The three install scopes (verbatim)

ScopeLoads inShared with teamStored in
LocalCurrent project onlyNo~/.claude.json
ProjectCurrent project onlyYes, via version control.mcp.json in project root
UserAll your projectsNo~/.claude.json
🪤 The scope-rename trap ⚠ VERIFY▼

Verbatim: "local (default): Available only to you in the current project (was called project in older versions)" and "user: Available to you across all projects (was called global in older versions)." Older study material — and possibly older exam phrasing — uses the old names. So "project scope" in old material may mean today's local, and "global" may mean today's user. Don't confuse today's project scope (the team-shared .mcp.json) with the legacy meaning.

project .mcp.json (checked into version control)
{
  "mcpServers": {
    "shared-server": {
      "command": "/path/to/server",
      "args": [],
      "env": {}
    }
  }
}

"For security reasons, Claude Code prompts for approval before using project-scoped servers from .mcp.json files." Reset with claude mcp reset-project-choices. Scope precedence (verbatim): 1. Local → 2. Project → 3. User → 4. Plugin-provided → 5. claude.ai connectors; "the entire server entry from that source is used; fields are not merged across scopes."

Management: claude mcp list, claude mcp get <name>, claude mcp remove <name>, in-session /mcp (also handles OAuth 2.0 for remote servers). Governance: managed-only allowedMcpServers, deniedMcpServers, allowManagedMcpServersOnly, plus enableAllProjectMcpServers/enabledMcpjsonServers/disabledMcpjsonServers; CLI --mcp-config <file> and --strict-mcp-config.

❓ Quick check
An older guide says "install the server at project scope so it stays on your machine only." In today's terminology, that maps to…
The old "project" meaning ("only you, current project") is today's local scope: "was called project in older versions." Today's project scope is the team-shared .mcp.json. Old "global" = today's user.
🔑 Key takeaways▼
  • Transports: http (recommended), sse (deprecated), stdio (local, after --), ws (JSON only).
  • Scopes: local (you/project, ~/.claude.json), project (team, .mcp.json), user (all projects). Rename trap: local=old "project", user=old "global".
  • Precedence: local → project → user → plugin → connectors; the whole entry is used, no field merge.
  • Project servers prompt for approval; governance via managed allow/deny keys and --strict-mcp-config.

Headless mode and the Agent SDK

📦 --bare, output formats, and the reproducible-CI story▼

--bare "skip[s] auto-discovery of hooks, skills, plugins, MCP servers, auto memory, and CLAUDE.md" for reproducible CI; it "will become the default for -p in a future release" ⚠ VERIFY. for file in $(cat files.txt); do
  claude -p "Migrate $file from React to Vue. Return OK or FAIL." \
    --allowedTools "Edit,Bash(git commit *)"
done

The canonical loop — explore → plan → code → commit (verbatim)

  1. Explore — "Enter plan mode. Claude reads files and answers questions without making changes."
  2. Plan — "Ask Claude to create a detailed implementation plan." (Ctrl+G to edit the plan)
  3. Implement — "Switch out of plan mode and let Claude code, verifying against its plan."
  4. Commit — "Ask Claude to commit with a descriptive message and create a PR."
"Letting Claude jump straight to coding can produce code that solves the wrong problem." (source: Best practices)

Verification gates (verbatim guidance): "Give Claude a check it can run: tests, a build, a screenshot to compare." Options: in-prompt, a /goal condition, a Stop hook ("blocks the turn from ending until it passes. Claude Code overrides the hook and ends the turn after 8 consecutive blocks"), or an adversarial review subagent / /code-review.

Context hygiene (verbatim): "Use /clear frequently between tasks"; "/compact <instructions>"; "After two failed corrections, /clear and write a better initial prompt." Built-in commands to know: /init, /memory, /permissions, /mcp, /agents, /hooks, /skills, /plugin, /config, /status, /model, /context, /compact, /clear, /plan, /doctor, /code-review, /security-review.
