Context Management & Reliability

The context window is your agent's working memory — finite, shared, and the thing that quietly decides whether a long-running agent stays coherent or falls apart. This lesson takes you from "what's a context window" down to the API mechanics of compaction, context editing, and token budgeting.

① Why it matters

Context Is the Reliability Backbone

Why managing the window — not maximizing it — is what keeps production agents alive.

🧠 Section 1⏱ ~10 minDomain 5 · Domain 1

The hook: a great agent that slowly forgets

Learning objectives
  • Frame context management as a reliability problem, not a capacity problem
  • State why "more context" is not the same as "better answers"
  • Preview the three levers — compaction, context editing, context engineering — you'll learn to pull

Why before how

Picture an agent that nails the first ten turns of a task, then — twenty turns later — starts repeating itself, forgetting decisions it already made, and getting slower and more expensive with every reply. The model didn't get worse. Its working memory filled up. Everything the agent has ever seen this session is competing for one finite resource, and nobody curated it.

That is the central insight of Domain 5: a production agent's reliability is mostly a function of what is in its context window and how it's managed over time. The docs put it bluntly — context is "a finite resource with diminishing returns, and irrelevant content degrades model focus." (source: context-editing docs)

Analogy (clearly labeled): Analogy, not a product fact. Think of the context window like a chef's cutting board, not a pantry. The pantry (the model's training) is enormous. The cutting board is small and shared — every ingredient you pile on leaves less room to actually work, and a cluttered board slows the chef down. Good agents keep the board clean; they don't just buy a bigger board.

The three levers you're about to learn

This lesson is organized around the moves an architect makes to keep that board clean:

  • Compaction — let the server summarize old history when the window fills (Section 5).
  • Context editing — surgically clear stale tool results and thinking blocks (Section 6).
  • Context engineering — put the smallest set of high-signal tokens in front of the model in the first place (Section 7).

Plus the discipline that ties them together: token budgeting (Section 8) and reliability patterns like external state and checkpointing (Section 9). But first, you have to understand the resource itself.

🤔 Predict first
Your agent has a 1M-token window and you're only using 400K. Is there any reliability reason to trim the context anyway?
Yes. "More context isn't automatically better. As token count grows, accuracy and recall degrade, a phenomenon known as context rot." Having headroom in the window doesn't mean the model uses all 400K equally well — irrelevant tokens still dilute its attention. Curating what's in context is "just as important as how much space is available." (source: context-windows docs)
🔑 Key takeaways▼
  • Context management is a reliability discipline, not just a way to fit more in.
  • "More context isn't automatically better" — accuracy and recall degrade as the window fills (context rot).
  • Three levers: compaction, context editing, and context engineering, backed by token budgeting and external state.
② The shared budget

The Context Window as Shared Working Memory

One window. Input, history, tools, thinking, and output all draw on it.

🪟 Section 2⏱ ~13 minDomain 5

The single budget everything spends from

Learning objectives
  • Define the context window and explain why it's "working memory," not training data
  • List every consumer of the window: input, history, tool config, thinking, and output
  • Recall current window sizes (200K vs 1M) and the overflow stop reason

What the context window actually is

"The 'context window' refers to all the text a language model can reference when generating a response, including the response itself. This is different from the large corpus of data the language model was trained on, and instead represents a 'working memory' for the model." (source: context-windows docs)

Two things to internalize. First, the response counts. The tokens Claude generates come out of the same budget as everything you sent in. Second, this is working memory — distinct from the vast training corpus in the "pantry." The model can only reason over what's currently on the board.

Everything spends from one budget

This is the fact that trips up most architects. The window is not "your prompt." It's a single pool that five different things draw on simultaneously:

  • Input / current message — the current user turn.
  • Conversation history — each input phase "contains all previous conversation history plus the current user message." (source: context-windows docs)
  • Tool configuration — tool definitions count as input tokens; tool-use requests, results, and accompanying thinking blocks all count toward the window. (source: context-windows docs)
  • Extended thinking — "all input and output tokens, including the tokens used for thinking, count toward the context window limit." Thinking tokens "are a subset of your max_tokens parameter, are billed as output tokens, and count towards rate limits." (source: context-windows docs)
  • Output — the generated response "becomes part of a future input." (source: context-windows docs)

Overflow is a signal, not a crash

"On Claude 4.5 models and newer, if input tokens plus max_tokens exceeds the context window size, the API accepts the request. If generation then reaches the context window limit, it stops with stop_reason: 'model_context_window_exceeded'." (source: context-windows docs)

Earlier models return a validation error unless you opt in via the model-context-window-exceeded-2025-08-26 beta header. We'll return to this stop reason in Section 9 — it's a reliability hook you can build on.

❓ Quick check
Which of these does NOT consume the context window?
The training corpus is the "pantry" — it's not in the window. Input, history, tool config, thinking, and output all consume the window; the response counts too.
🔑 Key takeaways▼
  • The window is working memory, distinct from training data — and "includes the response itself."
  • One shared budget: input + history + tool config + thinking + output.
  • Opus 4.8 / Sonnet 4.6 = 1M; Haiku 4.5 = 200k. ⚠ VERIFY — lineup drifts.
  • Overflow on 4.5+ returns model_context_window_exceeded rather than crashing.
③ Accumulation

Linear Accumulation, Quadratic Bill

Why long sessions get slow and expensive even though the window grows only linearly.

📈 Section 3⏱ ~11 minDomain 5 · Domain 1

The accumulation problem

Learning objectives
  • Describe progressive token accumulation across multi-turn conversations
  • Explain why per-turn growth is linear but billed-across-the-session cost is roughly quadratic
  • Connect this to the transformer's n² attention cost

Each turn keeps everything

"Progressive token accumulation: As the conversation advances through turns, each user message and assistant response accumulates within the context window. Previous turns are preserved completely." … "The context usage grows linearly with each turn, with previous turns preserved completely." (source: context-windows docs)

So the window per turn grows linearly — turn 2 holds turn 1, turn 3 holds turns 1–2, and so on. That part is intuitive. The non-obvious part is what it does to your bill.

Why the bill grows worse than linearly

Here's the mechanism. Every turn re-sends the entire prior history as input. Turn N pays to process roughly N turns' worth of history. Add that up across a session and the total tokens billed grow roughly quadratically with the number of turns — even though the window itself only grows linearly at any single moment.

Anthropic frames the underlying model cost as quadratic at the attention level too: the transformer creates "n² pairwise relationships for n tokens," which "stretches thin as context lengthens." (source: effective-context-engineering)

🤔 Predict first
A chat agent runs 50 turns with no context management. Compared to a 5-turn session, roughly how does the total tokens billed scale — 10×, or much more?
Much more than 10×. Because every turn re-sends the whole history, cumulative billed tokens scale roughly with turns² — so 10× the turns is closer to ~100× the work, not 10×. This is exactly why "server-side compaction is the primary strategy for context management" in long-running conversations. (source: context-windows docs)

What to do about it

The docs are explicit about the remedy ladder: for long-running conversations and agentic workflows, "server-side compaction is the primary strategy for context management." For finer control, context editing clears old tool results and thinking blocks. (source: context-windows docs) And prompt caching mitigates the cost of re-sending a stable prefix (Section 9) — but note: caching does not stop the window from filling. The two solve different problems.

❓ Quick check
Per-turn the window grows linearly. Why does total session cost grow faster than linearly?
Each turn re-processes the entire accumulated history as input, so summed over a session the billed tokens scale ~quadratically with the number of turns. The transformer's n² attention reinforces this.
🔑 Key takeaways▼
  • "Previous turns are preserved completely" — window grows linearly per turn.
  • Re-sending history every turn makes cumulative billed cost ~quadratic in turn count.
  • Compaction is the primary remedy; context editing is the fine-grained one; caching cuts cost but not occupancy.

④ Context rot

Context Rot & the Attention Budget

More tokens ≠ better. The deeper reason curation beats capacity.

🌫️ Section 4⏱ ~12 minDomain 5

Why a fuller window thinks worse

Learning objectives
  • Define context rot and the "attention budget"
  • Explain the architectural cause (n² relationships, training-length mismatch)
  • Reason about why newer models degrade more gracefully but never escape the effect

Context rot, defined

"As token count increases, the model's ability to accurately recall information from that context decreases." … "This constraint applies across all models, though some degrade more gradually." (source: effective-context-engineering)

And the mechanism the docs use to explain it — the attention budget:

LLMs possess "an 'attention budget' that they draw on when parsing large volumes of context. Every new token introduced depletes this budget by some amount." (source: effective-context-engineering)

So every token you add isn't free even when it fits — it spends from a budget that's separate from raw token capacity. This is why curation beats capacity.

Why it happens architecturally

Two structural reasons, per the engineering post:

  • The transformer creates "n² pairwise relationships for n tokens" — attention spreads thinner as n grows. (source: effective-context-engineering)
  • Models have "less experience with, and fewer specialized parameters for, context-wide dependencies" because training sequences are typically shorter than the max context. (source: effective-context-engineering)

The docs corroborate from the product side: context is "a finite resource with diminishing returns, and irrelevant content degrades model focus." (source: context-editing docs)

The benchmark anchor

Anthropic measures long-context capability on needle-in-a-haystack benchmarks (MRCR v2, GraphWalks), and notes gains "depend on what's in context, not just how much fits." (source: context-windows docs) A comparative datapoint from launch material: on the 8-needle 1M MRCR v2 variant, Opus 4.6 scored 76% vs Sonnet 4.5 at 18.5% — newer models degrade far more gracefully at depth, but degradation still exists.

⚠ VERIFY The MRCR v2 8-needle stat (Opus 4.6 76% vs Sonnet 4.5 18.5%) came from search-summary launch material, not a fetched primary page. Treat the exact figures as version-sensitive and re-verify before quoting.
🤔 Predict first
You upgrade from a 200K model to a 1M model. Does context rot stop being a concern?
No. A bigger window raises the ceiling but doesn't remove the effect — "this constraint applies across all models, though some degrade more gradually." You still spend an attention budget per token, so curating what goes in matters at 1M just as it did at 200K.
❓ Quick check
"Every new token introduced depletes this budget by some amount." What budget is the docs referring to?
The attention budget — separate from raw capacity. Filling the window depletes the model's ability to attend to and recall what's in it, which is context rot.
🔑 Key takeaways▼
  • Context rot: recall and focus drop as the window fills — universal across models.
  • The attention budget is depleted by every token, independent of raw capacity.
  • Architectural cause: n² attention relationships + shorter training sequences than max context.
  • Newer models degrade more gradually but never escape it — curate, don't just expand.
⑤ Compaction

Compaction — Server-Side Summarization

The primary strategy: let the server summarize old history when the window fills.

🗜️ Section 5⏱ ~16 minDomain 5 · Domain 1

How compaction works

Learning objectives
  • Walk the four-step compaction lifecycle and what gets dropped
  • Recall the 150K default trigger (min 50K) and the request/response shapes
  • State the key tradeoffs and how Claude Code's auto-compaction relates

The lifecycle

When enabled, Claude automatically summarizes the conversation as it approaches the configured threshold. It:

  1. Detects when input tokens exceed the trigger threshold;
  2. Generates a summary of the current conversation;
  3. Creates a compaction block containing the summary;
  4. Continues the response with the compacted context.

On subsequent requests, "The API automatically drops all message blocks prior to the compaction block, continuing from the summary." (source: compaction docs)

The trigger

Default trigger: 150,000 input tokens. Configurable, but must be at least 50,000 tokens. (source: compaction docs)

⚠ VERIFY Compaction is dated/beta. Supported models: "Mythos Preview, Opus 4.8, Opus 4.7, Opus 4.6, Sonnet 4.6 (beta)." Beta header anthropic-beta: compact-2026-01-12 must be in all requests using compaction, and the edit type string is compact_20260112. Both are dated and may rev. (source: compaction docs)

The request shape

Request · enabling compaction
{
  "model": "claude-opus-4-8",
  "max_tokens": 4096,
  "messages": [{"role": "user", "content": "Help me build a website"}],
  "context_management": {
    "edits": [
      {
        "type": "compact_20260112",
        "trigger": {"type": "input_tokens", "value": 150000},
        "pause_after_compaction": false,
        "instructions": null
      }
    ]
  }
}

Parameters: type (required, "compact_20260112"); trigger (default 150,000 input tokens); pause_after_compaction (default false); instructions (default null — a custom prompt replaces the default summarization prompt). (source: compaction docs)

The response shape — note the compaction block and iterations

Response · after compaction fires
{
  "content": [
    {"type": "compaction", "content": "Summary of the conversation: ..."},
    {"type": "text", "text": "Based on our conversation so far..."}
  ],
  "stop_reason": "end_turn",
  "usage": {
    "input_tokens": 23000,
    "output_tokens": 1000,
    "iterations": [
      {"type": "compaction", "input_tokens": 180000, "output_tokens": 3500},
      {"type": "message",    "input_tokens": 23000,  "output_tokens": 1000}
    ]
  }
}
💸 Billing trap (QA-critical)▼

The top-level input_tokens/output_tokens do not include the compaction-iteration usage. To get true session cost you must "sum across all iterations array entries for total tokens consumed." In the example above, the summarization step alone cost 180,000 input + 3,500 output tokens — invisible if you only read the top-level numbers. (source: compaction docs)

Tradeoffs

BenefitsCosts
Extends effective contextAdds a sampling step for summarization (counts toward rate limits & billing)
Automatic, minimal integrationCan't use a cheaper model to summarize — uses the same model as the request
Keeps active context focused"May fail when tools are defined" — the model may call a tool during summarization instead of summarizing

On the craft: "The art of compaction lies in the selection of what to keep versus what to discard… overly aggressive compaction can result in the loss of subtle but critical context whose importance only becomes apparent later." The recommendation: "Start by maximizing recall… then iterate to improve precision by eliminating superfluous content." (source: effective-context-engineering)

Claude Code's auto-compaction

At the product level, Claude Code "automatically optimizes costs through prompt caching… and auto-compaction, which summarizes conversation history when approaching context limits." You can steer it: /compact Focus on code samples and API usage, or add a # Compact instructions section to CLAUDE.md. (source: code.claude.com/costs)

⚠ VERIFY The ~83.5% / ~167K auto-compact trigger figure circulating in third-party blogs is not confirmed on the official code.claude.com/docs/en/costs page. Treat the specific percentage as unverified. (source: code.claude.com/costs)
🤔 Predict first
You enable compaction and define several tools on the same request. What's the failure mode the docs warn about?
Compaction "may fail when tools are defined" — during the summarization step the model may call a tool instead of producing the summary. It's a documented cost/limitation, so test compaction behavior carefully in tool-heavy agents.
❓ Quick check
After compaction fires, what happens to the messages before the compaction block on the next request?
"The API automatically drops all message blocks prior to the compaction block, continuing from the summary." You just pass the compaction block back.
🔑 Key takeaways▼
  • Lifecycle: detect → summarize → compaction block → drop prior → continue.
  • Default trigger 150K input tokens (min 50K). ⚠ VERIFY dated beta headers.
  • True cost = sum the iterations array; top-level usage excludes the summary step.
  • Can fail with tools defined; summarizes with the same (not a cheaper) model.
  • Claude Code auto-compacts; steer it with /compact or # Compact instructions.
⑥ Context editing

Context Editing — Fine-Grained Clearing

The scalpel: clear stale tool results and thinking blocks while keeping the rest.

✂️ Section 6⏱ ~15 minDomain 5

When the summary is too blunt

Learning objectives
  • Distinguish context editing from compaction and know when to reach for each
  • Read the tool-result-clearing config: trigger, keep, clear_at_least, exclude_tools
  • Recall the server-side, no-client-sync property and the cache interaction

What it is and when to use it

"Context editing allows you to selectively clear specific content from conversation history as it grows. Beyond optimizing costs and staying within limits, this is about actively curating what Claude sees." (source: context-editing docs)

The relationship to compaction is explicit: "For most use cases, server-side compaction is the primary strategy… The strategies on this page are useful for specific scenarios where you need more fine-grained control over what content is cleared." (source: context-editing docs) Compaction is the blunt instrument (summarize everything old); context editing is the scalpel (drop just the stale tool results, keep recent ones).

⚠ VERIFY Context editing is dated/beta: beta header context-management-2025-06-27 (supported on all supported Claude models). Strategy strings clear_tool_uses_20250919 and clear_thinking_20251015 are dated and may rev. (source: context-editing docs)

The strategies

StrategyType stringPurpose
Tool result clearingclear_tool_uses_20250919Clear old tool results in heavy tool-use workflows

Client-side compaction (SDK)—Summary-based, via tool_runner in Python/TS/Ruby SDKs

Tool-result clearing — the mechanism

"When activated, the API automatically clears the oldest tool results in chronological order. The API replaces each cleared result with placeholder text so Claude knows it was removed. By default, only tool results are cleared. You can optionally clear both tool results and tool calls (the tool use parameters) by setting clear_tool_inputs to true." (source: context-editing docs)
Config · tool-result clearing
{
  "type": "clear_tool_uses_20250919",
  "trigger": {"type": "input_tokens", "value": 30000},
  "keep": {"type": "tool_uses", "value": 3},
  "clear_at_least": {"type": "input_tokens", "value": 5000},
  "exclude_tools": ["web_search"]
}
  • trigger — clearing fires when input tokens exceed this value.
  • keep — number of recent tool uses to retain after clearing.
  • clear_at_least — guarantees a minimum number of tokens cleared each time (so prompt-cache invalidation is worthwhile).
  • exclude_tools — tools whose results are never cleared.
  • clear_tool_inputs — when true, also clears the tool-call parameters (default clears only results).

Thinking-block clearing

The clear_thinking_20251015 strategy "manages thinking blocks… you can choose to keep more thinking blocks to maintain reasoning continuity, or clear them more aggressively to save context space." Config: keep = {type: "thinking_turns", value: N} (N > 0) or "all". Per-model defaults differ:

Model classKeep ALL prior thinkingKeep ONLY last turn's thinking
OpusOpus 4.5 and laterOpus 4.1 and earlier
SonnetSonnet 4.6 and laterSonnet 4.5 and earlier
Haiku(none)all through Haiku 4.5

"If your code runs across multiple model tiers, set keep explicitly." And when combining strategies: "the clear_thinking_20251015 strategy must be listed first in the edits array." (source: context-editing docs)

Two properties architects must remember

🖥️ Server-side — no client sync needed▼

"Context editing is applied server-side before the prompt reaches Claude. Your client application maintains the full, unmodified conversation history. You do not need to sync your client state with the edited version." So your durable record stays complete; the server does the trimming on the way in. (source: context-editing docs)

💾 Cache interaction▼

Tool-result clearing "invalidates cached prompt prefixes when content is cleared." So you want to "clear enough tokens to make the cache invalidation worthwhile" — that's what clear_at_least is for. For thinking-block clearing: kept thinking preserves the cache; cleared thinking invalidates the cache at the clearing point. (source: context-editing docs)

⚠ VERIFY The response context_management.applied_edits object (number of tool uses cleared, tokens freed) is referenced in product summaries, but the exact JSON field names were not captured in the fetched section of the docs. Confirm against the live page before quoting field names. (source: context-editing docs)
❓ Quick check
Why does clear_at_least exist?
Clearing invalidates the cached prefix at the clear point, so clear_at_least guarantees a minimum tokens cleared each time — otherwise you'd pay the cache-invalidation cost for trivial savings.
🔑 Key takeaways▼
  • Context editing = the scalpel; compaction = the primary blunt strategy.
  • Tool-result clearing drops the oldest results, keeps keep recent ones, leaves a placeholder.
  • Config knobs: trigger, keep, clear_at_least, exclude_tools, clear_tool_inputs.
  • Server-side — your client keeps the full history; no sync required.
  • Clearing invalidates the cache at the clear point; clear_at_least makes it worthwhile.
⑦ Context engineering

Effective Context Engineering

The smallest set of high-signal tokens — and the Claude Code tactics that embody it.

🎯 Section 7⏱ ~13 minDomain 5 · Domain 1

Curate before you compress

Learning objectives
  • Distinguish context engineering from prompt engineering
  • State the guiding maxim and apply system-prompt + tool hygiene
  • Explain just-in-time retrieval vs pre-loading, and name concrete Claude Code tactics

Context engineering vs prompt engineering

Context engineering is "the set of strategies for curating and maintaining the optimal set of tokens (information) during LLM inference, including all the other information that may land there outside of the prompts." (source: effective-context-engineering)

Prompt engineering is "writing and organizing LLM instructions." Context engineering is broader: it addresses "the entire context state (system instructions, tools, Model Context Protocol, external data, message history, etc)" across multiple turns. Where Section 5 and 6 were about removing tokens, this section is about not adding the wrong ones in the first place.

The guiding maxim

Find "the smallest possible set of high-signal tokens that maximize the likelihood of some desired outcome." Overall: "be thoughtful and keep your context informative, yet tight." (source: effective-context-engineering)

System prompt & tool hygiene

  • Altitude. Prompts should strike the right "altitude" — avoiding both "brittle if-else hardcoded prompts" and "prompts that are overly general or falsely assume shared context." Organize with XML tags or Markdown headers, pursuing "the minimal set of information that fully outlines your expected behavior." (source: effective-context-engineering)
  • Tool curation. Tools should be "self-contained, robust to error, and extremely clear with respect to their intended use." Avoid "bloated tool sets that cover too much functionality or lead to ambiguous decision points about which tool to use." (source: effective-context-engineering)

Just-in-time retrieval vs pre-loading

Maintain "lightweight identifiers (file paths, stored queries, web links, etc.)" and "dynamically load data into context at runtime using tools" rather than pre-processing everything upfront. (source: effective-context-engineering)

The trade-off is real: "Runtime exploration is slower than retrieving pre-computed data," requiring "opinionated and thoughtful engineering" to keep agents from "wasting context by misusing tools, chasing dead-ends, or failing to identify key information." JIT keeps the window lean; pre-loading keeps latency low. An architect chooses per workload.

🤔 Predict first
Your agent must occasionally consult a 10,000-line log. Do you pre-load the whole log into context, or hand the agent a path and a grep tool?
Hand it a path + retrieval tool (JIT) — or preprocess with a hook. Pre-loading 10,000 lines burns the attention budget on every turn. The docs' tactic: use hooks to grep down to just the ERROR lines, "reducing context from tens of thousands of tokens to hundreds." Keep lightweight identifiers; load just-in-time.

  • Context engineering curates the whole context state, not just the prompt.
  • Maxim: smallest set of high-signal tokens; informative, yet tight.
  • Right "altitude" prompts + self-contained, non-bloated tools.
  • JIT retrieval keeps the window lean (trade: slower); pre-load for latency.
  • Claude Code: deferred MCP defs, CLI over MCP, sub-agents, hooks, /clear, CLAUDE.md < 200 lines.
⑧ Token budgeting

Token Budgeting in Practice

The usage object, the count_tokens API, and allocating the window like a budget.

🧮 Section 8⏱ ~14 minDomain 5 · Domain 1

Treat the window like a budget

Learning objectives
  • Read the usage object and compute total input tokens
  • Use the count_tokens API to pre-flight a budget
  • Allocate the window across system/tools, history, JIT retrieval, and reserved output

The budget allocation

An architect allocates the single window across four buckets, each with its own management strategy:

The usage object

Response · usage (with caching)
{
  "input_tokens": 2048,
  "output_tokens": 503,
  "cache_creation_input_tokens": 5120,
  "cache_read_input_tokens": 1800
}
  • input_tokens — "Tokens after the last cache breakpoint (not eligible for cache)."
  • output_tokens — generated tokens (includes thinking tokens, billed as output).
  • cache_creation_input_tokens — "Number of tokens written to cache (new entries)."
  • cache_read_input_tokens — "Number of tokens retrieved from cache."

Total input = cache_read_input_tokens + cache_creation_input_tokens + input_tokens. Reading only input_tokens understates your true input. (source: prompt-caching docs)

Pre-flight with count_tokens

Plan a budget before you send. The endpoint POST /v1/messages/count_tokens "accepts the same structured inputs as message creation" (system, tools, images, PDFs) and returns just { "input_tokens": N }. Examples: a basic message → 14; message + one tool → 403; message + image → 1551; message + PDF → 2188. (source: token-counting docs)

  • "The token count should be considered an estimate… the actual number… may differ by a small amount." System-added optimization tokens aren't billed.
  • Free, but rate-limited separately (tier 1 = 100 RPM, tier 2 = 2,000, tier 3 = 4,000, tier 4 = 8,000). It does not use caching logic.
  • Thinking nuance: "Thinking blocks from previous assistant turns are ignored and do not count," but "current assistant turn thinking does count."

The programmatic budget signal (context awareness)

Some models self-manage their remaining budget. At conversation start they receive a token-budget tag, and after each tool call a usage warning:

System signals · context awareness
<budget:token_budget>1000000</budget:token_budget>
<system_warning>Token usage: 35000/1000000; 965000 remaining</system_warning>

These models "track their remaining context window (that is, 'token budget') throughout a conversation." Smaller-window models get 200000 instead of 1000000. (source: context-windows docs)

⚠ VERIFY Context awareness is listed for Sonnet 4.6, Sonnet 4.5, and Haiku 4.5 — Opus 4.8 is not listed as context-aware. Confirm whether Opus has any context-awareness signal before relying on it. (source: context-windows docs)

Claude Code cost telemetry

/usage shows session token stats and a local cost estimate; /context shows what's consuming the window; the status line can display context-window usage continuously. (source: code.claude.com/costs)

❓ Quick check
A response shows input_tokens: 2048, cache_creation_input_tokens: 5120, cache_read_input_tokens: 1800. What was the total input?
Total input = cache_read + cache_creation + input_tokens = 1,800 + 5,120 + 2,048 = 8,968. Reading only input_tokens (2,048) badly understates it.
🔑 Key takeaways▼
  • Total input = cache_read + cache_creation + input_tokens.
  • Pre-flight with the free count_tokens endpoint (an estimate; separately rate-limited).
  • Allocate the window: cache stable parts, compact history, JIT retrieval, reserve output.
  • Sonnet 4.6/4.5 + Haiku 4.5 get a token_budget signal. ⚠ VERIFY — Opus 4.8 not listed.
⑨ Reliability

Reliability Patterns Tied to Context

External state, checkpointing, recovery, and overflow as a signal — the Domain 1 crossover.

🛟 Section 9⏱ ~13 minDomain 1 · Domain 5

Designing for survival across sessions

Learning objectives
  • Explain why durable state belongs outside the context window
  • Recall the checkpointing / recovery patterns from Anthropic's long-running-agent harness
  • Treat the overflow stop reason as a reliability hook, not a crash

Put durable state outside the window

The window is volatile working memory — it gets compacted, cleared, and reset. So for agents spanning multiple sessions, "design your state artifacts so that context recovery is fast when a new session starts." (source: context-windows docs) Anthropic's long-running-agent harness uses concrete artifacts:

  • A claude-progress.txt log "that keeps a log of what agents have done."
  • A structured feature-requirements JSON file with per-item completion status.
  • An initial git commit "that shows what files were added" as a recovery mechanism.

(source: effective-harnesses-for-long-running-agents)

Checkpointing & recovery

  • Session init: "Read the git logs and progress files to get up to speed on what was recently worked on."
  • State verification: "run a basic test on the development server to catch any undocumented bugs" before new work.
  • Clean handoffs: "leave the environment in a clean state at the end of a session" — commit with descriptive messages and write progress summaries.
  • Incremental progress: "work on only one feature at a time" prevents context exhaustion and incomplete implementations across sessions.

(source: effective-harnesses-for-long-running-agents) In Claude Code, recovery from a mid-session wrong turn uses /rewind or double-tap Escape "to restore conversation and code to a previous checkpoint"; press Escape to stop early; use plan mode (Shift+Tab) before implementing to prevent expensive re-work. (source: code.claude.com/costs)

Overflow as a reliability signal

Recall Section 2's stop reason. On 4.5+ models, hitting the limit yields stop_reason: "model_context_window_exceeded" "rather than a hard validation failure — handle this stop reason to trigger compaction/checkpoint-and-continue instead of crashing." (source: context-windows docs) That single branch in your loop is the difference between an agent that gracefully checkpoints and one that falls over.

⚠ VERIFY Retries / idempotency: the fetched reliability page emphasizes external-state and checkpointing, not HTTP-level retries or idempotency keys. The architect-level guidance — make tool calls idempotent and persist state externally so a retried/restarted session reconstructs from durable artifacts (git, progress file, memory tool) rather than from in-window history — is inferred from that pattern. Confirm any dedicated Anthropic idempotency-key page before asserting a specific mechanism. (source: effective-harnesses-for-long-running-agents)

How prompt caching intersects (brief — full treatment in C-E4)

Caching is "a way to manage cost of large stable context" — it reduces the per-turn cost of re-sending a stable prefix but does not reduce window occupancy. On a request the system checks whether a prompt prefix up to a cache breakpoint is already cached; if so it reuses it, otherwise it caches the prefix once the response begins. (source: prompt-caching docs)

cache_control shape
{ "cache_control": { "type": "ephemeral", "ttl": "5m" } }
⚠ VERIFY Cache pricing/min-prefix are model- and price-sensitive: 5-min cache write ≈ 1.25× base input, 1-hour write ≈ 2×, reads ≈ 0.1×; minimum cacheable prefix is 1,024 tokens (Opus 4.8 / Sonnet 4.6 / Sonnet 4.5 / Opus 4.1) vs 4,096 (Opus 4.7 / 4.6 / 4.5 / Haiku 4.5). Re-verify against the pricing page; full caching treatment lives in extension C-E4. (source: prompt-caching docs)
🤔 Predict first
A long agent run dies mid-task and restarts. Should it reconstruct state from its previous in-window conversation, or from somewhere else?
From external durable artifacts. The window doesn't survive the restart. A reliable agent reads the git logs and progress files "to get up to speed," verifies state with a basic test, and continues — reconstructing from claude-progress.txt, the requirements JSON, git commits, and the memory tool, not from a context window that's already gone.
❓ Quick check
On a 4.5+ model your generation hits stop_reason: "model_context_window_exceeded". The reliable response is to…
It's a handleable signal, not a crash. Branch on it to compact, clear, or checkpoint-and-continue. Retrying unchanged would just hit the limit again.
🔑 Key takeaways▼
  • Durable state lives outside the window: progress log, requirements JSON, git commits, memory tool.
  • Session init reads logs + progress; verify state; leave clean handoffs; one feature at a time.
  • Treat model_context_window_exceeded as a hook → compact / checkpoint-and-continue.
  • Caching cuts the cost of a stable prefix but not window occupancy (full treatment: C-E4).
