① Why

Why Prompt Engineering — and Why Model Choice Is a Peer Lever

Be empirical first. Prompting beats fine-tuning for iteration speed. And not every failure is a prompting problem.

🧪 Section A⏱ ~10 minDomain 3 · Domain 1

The empirical loop, and the lever most people forget

Learning objectives
  • State the three prerequisites Anthropic assumes before you prompt-engineer
  • Explain why prompting is reached for before fine-tuning
  • Recognise that model selection is a peer lever — some failures are best fixed by changing the model, not the prompt

Why before how

Before you touch a prompt, Anthropic's overview assumes you already have three things. Verbatim: "1. A clear definition of the success criteria for your use case 2. Some ways to empirically test against those criteria 3. A first draft prompt you want to improve." If you lack them, "we highly suggest you spend time establishing that first." (source: prompt-engineering overview) The pattern name to carry forward is the empirical-iteration loop: define success criteria → build evals → draft a prompt → iterate. Prompt engineering is the tuning step inside that loop, not a substitute for it.

Andrew-Ng framing: don't start by polishing prose. Start by deciding how you'll know the prompt is good. If you can't measure it, you're guessing — and a guess that "looks better" is how teams ship regressions.

Not every failure is a prompting problem

"Not every success criteria or failing eval is best solved by prompt engineering. For example, latency and cost can be sometimes more easily improved by selecting a different model." (source: prompt-engineering overview)

This single sentence is the bridge to Section F. Model selection is a peer lever to prompting — not a last resort. If your eval is failing on latency or cost, the highest-leverage move may be to drop from Opus to Sonnet (or Haiku), or to lower the effort setting — not to wordsmith the prompt. Hold that thought; we make it concrete in F.

Why prompting before fine-tuning

Reach for prompting first because the change-test cycle is minutes. SECONDARY "Prompt engineering is far faster and cheaper than fine-tuning" (source: aiwithgrant guide, summarising Anthropic docs). The mechanics behind that claim: editing a prompt is editing a string and re-running your evals; a fine-tune is hours of compute, a dataset, and a new artifact to host. The prompt also stays transparent (you can read exactly what changed) and portable (it travels to the next model — with a re-test, see Section F).

🛠️ The tooling that supports the loop▼

Anthropic ships a prompt generator, prompt templates and variables, and a prompt improver in the Claude Console to "help you build and refine prompts quickly" (source: prompt-engineering overview). These exist to shorten the draft→test→iterate loop — they don't replace having success criteria and evals.

🤔 Predict first
Your support-bot prompt is accurate enough, but p95 latency is too high and per-conversation cost is over budget. You've been told to "improve the prompt." Predict: is more prompt engineering the right first move?
Probably not. Anthropic explicitly notes "latency and cost can be sometimes more easily improved by selecting a different model." A failing latency/cost eval is the textbook case for the model-selection lever: descend from a heavier model to Sonnet/Haiku, or lower the effort setting, and re-run your evals. Wordsmithing won't move a latency budget the way a tier change will. (source: prompt-engineering overview; effort/model framing in Section F)
❓ Quick check
According to Anthropic's overview, which three things should be in place before you start prompt engineering?
Verbatim: "1. A clear definition of the success criteria… 2. Some ways to empirically test against those criteria 3. A first draft prompt you want to improve." Without these, "spend time establishing that first." Prompting is the tuning step inside the empirical loop.
🔑 Key takeaways▼
  • Empirical loop: success criteria → evals → draft → iterate. Prompting is the tuning step inside it.
  • Prompting before fine-tuning — faster, cheaper, transparent, portable change-test cycle.
  • Model selection is a peer lever: "latency and cost can be sometimes more easily improved by selecting a different model."
  • Console tooling (generator, templates, improver) shortens the loop but doesn't replace evals.
② The ladder

The Technique Ladder — Reconciling Both Orderings

Clarity → examples → thinking → XML → role → chaining → long-context. Why that order, with each rung's mechanics and a snippet.

🪜 Section B⏱ ~18 minDomain 3

The rungs, and why you climb them in this order

Learning objectives
  • Reconcile the classic docs ladder with the interactive-tutorial teaching sequence
  • State each rung's mechanics and recall a verbatim snippet for clarity and role
  • Articulate the ordering rationale — and recognise it as pedagogical framing, not an Anthropic quote

Two canonical orderings

(1) The classic docs ladder — the ordered list the original Claude 101 extends: be clear and direct → use examples (multishot) → let Claude think (chain of thought) → use XML tags → give Claude a role (system prompts) → chain complex prompts → long-context tips SECONDARY (source: aiwithgrant guide). Important: the docs do not label this a "ladder" and do not print a one-line "cheap-first" rationale.

(2) The interactive tutorial teaching sequence (PRIMARY repo): Beginner — Basic Prompt Structure → Being Clear and Direct → Assigning Roles; Intermediate — Separating Data from Instructions → Formatting Output & Speaking for Claude → Precognition (Thinking Step by Step) → Using Examples; Advanced — Avoiding Hallucinations → Building Complex Prompts; Appendix — Chaining, Tool Use, Search & Retrieval (source: interactive tutorial repo).

⚖️ The ordering rationale is reasoning, not a verbatim Anthropic quote▼

⚠ VERIFY The framing "techniques are ordered cheapest-and-highest-leverage first" is pedagogical reasoning, corroborated by a secondary source and the tutorial's chapter order — it is not a one-line Anthropic quote. The reasoning: clarity costs nothing and fixes the most failures; examples are the next-highest lever; thinking/CoT trades tokens and latency for accuracy; structural tools (XML, roles) shape output; chaining and long-context handling are heavier, situational moves. Try them roughly in this order and stop when your evals pass — that's the loop from Section A applied to the ladder. It's consistent with Anthropic's "when to prompt engineer" note, but the explicit "cheap-first" wording is yours, not theirs.

Rung 1 · Be clear and direct

"Claude responds well to clear, explicit instructions. Being specific about your desired output can help enhance results. If you want 'above and beyond' behavior, explicitly request it rather than relying on the model to infer this from vague prompts." (source: best practices)

Mental model (verbatim): "Think of Claude as a brilliant but new employee who lacks context on your norms and workflows. The more precisely you explain what you want, the better the result." analogy, not a product fact — Anthropic's own framing. The golden rule (verbatim): "Show your prompt to a colleague with minimal context on the task and ask them to follow it. If they'd be confused, Claude will be too." Mechanics: be specific about output format/constraints; use numbered lists/bullets "when the order or completeness of steps matters."

// Less effective:
Create an analytics dashboard

// More effective:
Create an analytics dashboard. Include as many relevant
features and interactions as possible. Go beyond the basics
to create a fully-featured implementation.
Rung 1b · Add context / motivation (clarity's twin)
"Providing context or motivation behind your instructions, such as explaining to Claude why such behavior is important, can help Claude better understand your goals… Claude is smart enough to generalize from the explanation." (source: best practices)
motivation beats a bare rule
// Bare rule:
NEVER use ellipses

// With motivation (Claude generalizes from the why):
Your response will be read aloud by a text-to-speech engine,
so never use ellipses since the text-to-speech engine will
not know how to pronounce them.

Rung 2 · Use examples / multishot

"Examples are one of the most reliable ways to steer Claude's output format, tone, and structure. A few well-crafted examples (known as few-shot or multishot prompting) can dramatically improve accuracy and consistency." (source: best practices)

This is the highest-leverage rung — Section C is the deep-dive. Mechanically, examples let the model see the target instead of inferring it from a description.

Rung 3 · Let Claude think (chain of thought)

On current models this is mostly handled by adaptive thinking; manual "think step by step" is now a fallback when thinking is off. Section D is the deep-dive and covers the version shift.

Rung 4 · Structure prompts with XML tags

"XML tags help Claude parse complex prompts unambiguously, especially when your prompt mixes instructions, context, examples, and variable inputs. Wrapping each type of content in its own tag (e.g. <instructions>, <context>, <input>) reduces misinterpretation." (source: best practices)

Best practices (verbatim): "Use consistent, descriptive tag names across your prompts" and "Nest tags when content has a natural hierarchy." Key point: there is no canonical or required tag vocabulary — consistency matters more than the specific names you pick.

XML-structured prompt skeleton
<instructions>
  Summarize the report for an executive audience.
</instructions>
<context>
  The reader has 2 minutes and no technical background.
</context>
<input>
  {{REPORT_TEXT}}
</input>

Rung 5 · Give Claude a role (system prompts)

"Setting a role in the system prompt focuses Claude's behavior and tone for your use case. Even a single sentence makes a difference." (source: best practices)

Use the top-level system parameter:

role via the system parameter (verbatim)
import anthropic
client = anthropic.Anthropic()
message = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    system="You are a helpful coding assistant specializing in Python.",
    messages=[{"role": "user", "content": "How do I sort a list of dictionaries by key?"}],
)

Rung 6 · Chain complex prompts

"With adaptive thinking and subagent orchestration, Claude handles most multi-step reasoning internally. Explicit prompt chaining (breaking a task into sequential API calls) is still useful when you need to inspect intermediate outputs or enforce a specific pipeline structure." (source: best practices)

Canonical self-correction pattern (verbatim): "generate a draft → have Claude review it against criteria → have Claude refine based on the review. Each step is a separate API call so you can log, evaluate, or branch at any point." Pattern name: chaining = a hand-built pipeline you control; subagent orchestration = the model doing it internally. This is the direct tie to the agent lessons — when you want to inspect intermediate state, chain; when you want the model to manage it, orchestrate.

Rung 7 · Long-context tips

Verbatim threshold: "When working with large documents or data-rich inputs (20k+ tokens), structure your prompt carefully." ⚠ VERIFY Three rules:

  • Documents-first ordering (verbatim): "Put longform data at the top: Place your long documents and inputs near the top of your prompt, above your query, instructions, and examples. This can significantly improve performance across all models." Quantified (verbatim): "Queries at the end can improve response quality by up to 30% in tests, especially with complex, multi-document inputs." ⚠ VERIFY the 30% figure
  • Structure with XML: wrap each document in <document> tags with <document_content> and <source> subtags; use <document index="n">.
  • Ground responses in quotes (verbatim): "For long document tasks, ask Claude to quote relevant parts of the documents first before carrying out its task. This helps Claude cut through the noise of the rest of the document's contents."

(See Section B's long-context diagram in the structured-output section for the docs-at-top layout.) Note on prefill: the classic ladder included "prefill Claude's response" as a rung. On current models this is largely removed — see Section E and the [VERIFY] list. LEGACY (pre-4.6)

🔑 Key takeaways▼
  • Ladder: clarity → examples → thinking → XML → role → chaining → long-context.

③ Examples

Examples / Multishot — the Highest-Leverage Rung

3–5 diverse examples in <example> tags. Show the target; positive beats negative.

🎯 Section C⏱ ~12 minDomain 3

Why examples beat instructions — and how to build a good set

Learning objectives
  • Explain why examples are the highest-leverage rung after clarity
  • Recall the count (3–5) and the three qualities of a good example set
  • Apply positive-over-negative and the examples-compound-with-thinking patterns

Why it's the highest-leverage rung

"Examples are one of the most reliable ways to steer Claude's output format, tone, and structure. A few well-crafted examples (known as few-shot or multishot prompting) can dramatically improve accuracy and consistency." (source: best practices)

The mechanism, in plain terms: an instruction describes the target; an example shows it. Showing is higher-bandwidth — the model infers format, tone, edge handling, and structure all at once from a concrete instance, instead of reconstructing them from prose. That's why, once your prompt is clear, examples are the next lever to reach for.

How many — and what makes a set good

Verbatim: "Include 3–5 examples for best results. You can also ask Claude to evaluate your examples for relevance and diversity, or to generate additional ones based on your initial set." ⚠ VERIFY the exact "3–5"

The three qualities of a good example set (verbatim):

  • Relevant: "Mirror your actual use case closely."
  • Diverse: "Cover edge cases and vary enough that Claude doesn't pick up unintended patterns."
  • Structured: "Wrap examples in <example> tags (multiple examples in <examples> tags) so Claude can distinguish them from instructions."
structured multishot · <examples> wrapping <example> blocks
<examples>
  <example>
    <input>The wifi keeps dropping every few minutes.</input>
    <category>connectivity</category>
  </example>
  <example>
    <input>I was double-charged on my last invoice.</input>
    <category>billing</category>
  </example>
  <example>
    <input>How do I export my data before cancelling?</input>
    <category>account</category>
  </example>
</examples>

Positive over negative

"Positive examples showing how Claude can communicate with the appropriate level of concision tend to be more effective than negative examples or instructions that tell the model what not to do." (source: best practices)

Pattern name: show the target, don't enumerate what to avoid. A "don't do X / don't do Y" list makes the model hold a mental blacklist; one example of the right output gives it a single, concrete pattern to copy. This is the same principle as Tier-1 output control in Section E ("tell Claude what to do instead of what not to do").

Examples compound with thinking

"Multishot examples work with thinking. Use <thinking> tags inside your few-shot examples to show Claude the reasoning pattern. It will generalize that style to its own extended thinking blocks." (source: best practices)

So examples don't just steer the final output — put a <thinking> block inside an example and you're steering the reasoning style too. This is the bridge from rung 2 (examples) to rung 3 (thinking, Section D).

❓ Quick check
Which describes a good example set, per Anthropic?
"Include 3–5 examples for best results" ⚠ VERIFY. Good sets are Relevant (mirror the use case), Diverse (cover edge cases, vary so Claude doesn't learn unintended patterns), and Structured (wrapped in <example>/<examples> tags). Positive examples beat negative ones.
❓ Quick check

"Multishot examples work with thinking. Use <thinking> tags inside your few-shot examples to show Claude the reasoning pattern. It will generalize that style to its own extended thinking blocks." Examples steer reasoning, not just output.
🔑 Key takeaways▼
  • Examples are the highest-leverage rung after clarity — showing beats describing.
  • 3–5 examples ⚠ VERIFY; qualities = Relevant, Diverse, Structured (in <example>/<examples> tags).
  • Positive > negative — show the target, don't list what to avoid.
  • Examples compound with thinking via <thinking> tags inside the example.
④ Thinking

Chain-of-Thought — the Adaptive-Thinking Version Shift

Manual "think step by step" is now a fallback. Adaptive thinking + the effort dial is the modern primary.

🧠 Section D⏱ ~16 minDomain 3 · Domain 1

From hand-written CoT to adaptive thinking + effort

Learning objectives
  • Describe the think-space spectrum: off → manual CoT fallback → adaptive + effort
  • State why manual budget_tokens thinking is legacy and what replaced it
  • Apply show-the-thinking, self-check, general-over-prescriptive, and the cost/latency tradeoff
⏳ Major version shift — read first. On current models (Sonnet 4.6, Haiku 4.5, Opus 4.6/4.7/4.8) the dominant mechanism is adaptive thinking, not hand-written "think step by step." LEGACY (pre-4.6) Manual extended thinking with budget_tokens is deprecated/removed on Opus 4.7+ and returns a 400. MODERN Manual CoT prompting now serves only as the fallback when thinking is off. (source: migration guide; best practices)

The think-space spectrum

  • Off / direct (verbatim): "Thinking is off by default when you omit the thinking parameter." On Opus 4.8, "thinking is off unless you explicitly set thinking: {type: \"adaptive\"}."
  • Manual CoT fallback fallback (verbatim): "When thinking is off, you can still encourage step-by-step reasoning by asking Claude to think through the problem. Use structured tags like <thinking> and <answer> to cleanly separate reasoning from the final output."
  • Adaptive primary (verbatim): "Claude calibrates its thinking based on two factors: the effort parameter and query complexity. Higher effort elicits more thinking, and more complex queries do the same. On easier queries that don't require thinking, the model responds directly. In internal evaluations, adaptive thinking reliably drives better performance than extended thinking."

Show the thinking, and self-check

The structured <thinking>/<answer> pattern keeps reasoning visible and inspectable — the modern equivalent of the classic "always have Claude output its thinking." And ask the model to verify itself (verbatim): "Ask Claude to self-check. Append something like 'Before you finish, verify your answer against [test criteria].' This catches errors reliably, especially for coding and math."

Prefer general over prescriptive (verbatim): "A prompt like 'think thoroughly' often produces better reasoning than a hand-written step-by-step plan. Claude's reasoning frequently exceeds what a human would prescribe." The architect lesson: don't micromanage the steps — set the goal and the verification criteria, and let the model's reasoning find the path.

The cost / latency tradeoff and over-thinking

"Extended thinking adds latency and should only be used when it will meaningfully improve answer quality — typically for problems that require multi-step reasoning. When in doubt, respond directly." (source: best practices)

Over-thinking is a real failure mode in the opposite direction (verbatim): "Claude Opus 4.6 may think extensively, which can inflate thinking tokens and slow down responses… you can lower the effort setting to reduce overall thinking and token usage." So the effort dial cuts both ways — raise it for hard reasoning, lower it when the model is burning tokens on something easy. (This connects straight to C-E4: thinking tokens are billed output tokens.)

adaptive thinking + effort (verbatim shape) — the MODERN primary
client.messages.create(
    model="claude-opus-4-8",
    max_tokens=64000,
    thinking={"type": "adaptive"},
    output_config={"effort": "high"},  # or "max", "xhigh", "medium", "low"
    messages=[{"role": "user", "content": "..."}],
)
🔤 Opus-4.5 word-sensitivity caveat▼

⚠ VERIFY — model-specific Verbatim: "When extended thinking is disabled, Claude Opus 4.5 is particularly sensitive to the word 'think' and its variants. Consider using alternatives like 'consider,' 'evaluate,' or 'reason through' in those cases." This is an Opus-4.5-specific quirk when thinking is off — not a general rule. Mentioned so you recognise it if a prompt over-triggers on that model.

Manual extended thinking with budget_tokens is deprecated/removed on Opus 4.7+, so this is the LEGACY path and will fail (the version shift the migration guide calls out) ⚠ VERIFY. The MODERN fix is adaptive thinking + effort: thinking={"type":"adaptive"} plus output_config={"effort":"high"}, letting the model calibrate by effort and query complexity. "Adaptive thinking reliably drives better performance than extended thinking." (source: migration guide; best practices)

Manual <thinking>/<answer> CoT is the fallback "when thinking is off." Manual budget_tokens thinking is removed on Opus 4.7+.

Over-thinking is fixed by lowering effort: "you can lower the effort setting to reduce overall thinking and token usage." The effort dial cuts both ways — raise it for hard reasoning, lower it when the model over-thinks an easy task.
🔑 Key takeaways▼
  • Spectrum: off → manual CoT fallback (<thinking>/<answer>) → adaptive + effort (primary).
  • Manual budget_tokens thinking is LEGACY — removed on Opus 4.7+ ⚠ VERIFY.
  • Prefer general over prescriptive ("think thoroughly" beats a hand-written plan); ask for a self-check.
  • Effort cuts both ways: up for hard reasoning, down to fix over-thinking; thinking tokens cost money/latency.
  • Opus-4.5 is "think"-word sensitive when thinking is off ⚠ VERIFY — use "consider / evaluate / reason through."
⑤ Structured output

Structured Output — Three Tiers, and the Prefill Migration

Prompt format → XML contracts → constrained-decoding JSON. Prefill is now legacy.

📐 Section E⏱ ~16 minDomain 3 · Domain 4

From "please format it nicely" to a hard schema guarantee

Learning objectives
  • Place the three tiers of output control and when each is appropriate
  • State what Structured Outputs guarantees, the limits, and the supported models
  • Explain why prefill is now legacy and how the classic prefill uses migrate

The three tiers — soft to hard

Tier 1 · Prompt-level format control

Four levers (verbatim):

  1. "Tell Claude what to do instead of what not to do" — instead of "Do not use markdown," try "Your response should be composed of smoothly flowing prose paragraphs."
  2. "Use XML format indicators" — "Write the prose sections of your response in <smoothly_flowing_prose_paragraphs> tags."
  3. "Match your prompt style to the desired output" — "removing markdown from your prompt can reduce the volume of markdown in the output."
  4. "Use detailed prompts for specific formatting preferences."

Note lever 1 is the same positive-over-negative principle from Section C.

Tier 2 · XML output contracts

Have the model write its answer inside named tags (<quotes>, <info>, <answer>) so downstream code parses deterministically. This pairs with the long-context "ground in quotes" pattern from Section B — ask for the quotes inside <quotes>, then the answer inside <answer>.

Tier-2 XML output contract
Quote the passages you used inside <quotes> tags,
then give your final answer inside <answer> tags.

<quotes>
  ...relevant source passages...
</quotes>
<answer>
  ...final answer parsed by downstream code...
</answer>

Tier 3 · Structured Outputs — the hard guarantee

"Structured outputs constrain Claude's responses to follow a specific schema, ensuring valid, parseable output for downstream processing." (source: Structured Outputs)

Two features, both via constrained decoding:

  • JSON outputs (output_config.format) — "Get Claude's response in a specific JSON format."
  • Strict tool use (strict: true) — "Guarantee schema validation on tool names and inputs."

The payoff (verbatim): output is "Always valid: No more JSON.parse() errors," "Type safe," and "Reliable: No retries needed for schema violations." For classification tasks specifically (verbatim): "use either tools with an enum field containing your valid labels or structured outputs."

Tier-3 JSON outputs (verbatim shape)
response = client.messages.parse(
    model="claude-opus-4-8",
    max_tokens=1024,
    messages=[...],
    output_config={"format": {"type": "json_schema",
        "schema": {"type": "object",
            "properties": {"name": {"type": "string"}, "email": {"type": "string"}},
            "required": ["name", "email"], "additionalProperties": False}}},
)
🚧 Structured Outputs — the limits (don't memorise blindly, but know they exist)▼

⚠ VERIFY NOT supported: recursive schemas; external $ref; numerical constraints (minimum, maximum…); string constraints (minLength, maxLength). additionalProperties must be false. Limits: 20 strict tools / request; 24 optional params; 16 union-typed params. Performance: "Compiled grammars are cached for 24 hours," with "additional latency while the grammar compiles" on the first request. The tradeoff axis — and the second dial

The three tiers trade capability ↔ speed ↔ cost: Opus = max capability, moderate latency, highest price; Sonnet = the balance; Haiku = fastest + cheapest with "near-frontier" capability. But there's a second, orthogonal dial: the effort parameter trades "Claude's intelligence vs. token spend, trading off capability for faster speed and lower costs" within a model. Architect point: you tune cost/quality on two dials — model tier AND effort level.

The decision framework — start capable, descend by evals

"If you're unsure which model to use, consider starting with Claude Opus 4.8 for the most complex tasks. It is Anthropic's most capable model for complex reasoning, long-horizon agentic coding, and high-autonomy work." (source: Models overview)

SECONDARY A complementary framing: "Teams should start with Sonnet as the default, measure actual quality and cost against their specific tasks, and then specialize by routing simpler tasks down to Haiku and only the most demanding tasks up to Opus." (source: SitePoint, secondary)

Reconcile the two: start with a capable model to establish a quality ceiling, get it working, then optimize down to the cheapest model that still passes your evals — escalating to Opus only for the hardest, longest-horizon work. This is the same "climb only as far as evals require" discipline from the ladder, now applied to model tier. It matches the overview's note that "latency and cost can be sometimes more easily improved by selecting a different model."

"For the hardest, longest-horizon problems (large-scale code migrations, deep research, extended autonomous work), Opus 4.8 remains the right choice. Sonnet 4.6 is optimized for workloads where fast turnaround and cost efficiency matter most." (source: migration guide)

The prompt is portable — but re-test on a new model

Migration guidance repeatedly notes new models have "strong out-of-the-box performance on existing… prompts and evals" — prompts are broadly portable. But every migration section lists behavioral shifts to re-check: verbosity calibration, literal instruction-following, effort recalibration, tone/voice changes, prefill removal. Verbatim: "As with any new model, prose style on long-form writing may shift… If your product relies on a specific voice, re-evaluate style prompts against the new baseline."

🧭 Architect rule: portable as a starting point, never as a guarantee▼

A prompt is portable as a starting point, but you must re-run your evals on the new model before trusting it in production (reasoning grounded in the migration-guide quotes). The things that drift: verbosity, how literally instructions are followed, the right effort level, tone/voice, and anything that relied on prefill. "Don't blind-transfer" is itself a quiz-worthy anti-pattern (Section G).

Connecting to the agent lessons (C-E4 cost + orchestration)

The cost matrix is exactly what motivates the standard agentic pattern: a smarter, more expensive orchestrator (Opus/Sonnet) directing cheaper workers (Haiku) on bounded subtasks. The docs support delegating to specialized subagents that "require isolated context… or involve independent workstreams," while doing simple/sequential work directly. Pair this with prompt caching (Opus 4.8's cacheable minimum dropped to 1,024 — see C-E4) and the Batch API for further cost control. The C-E4 cost lever, made concrete: route by difficulty, cache the static system prompt, batch the non-interactive work. This is S-E3's orchestrator-worker pattern wearing its price tag.

Putting It Together + Anti-Patterns

The integrated prompt and the seven anti-patterns

Learning objectives

The integrated prompt

A strong production prompt typically stacks the rungs (synthesized from best-practices): a role in the system field → XML-structured sections → 3–5 diverse examples in <example> tags → long docs at the top, query at the end → grounding-in-quotes for long context → adaptive thinking + effort for hard reasoning → Structured Outputs for a machine-parseable result → optional prompt chaining for self-correction.

the integrated prompt — rungs stacked
system: "You are a senior support-triage assistant."   # role (rung 5)

# user turn — XML-structured (rung 4), docs-first (rung 7):
<documents>
  <document index="1"><document_content>...long policy...</document_content></document>
</documents>
<examples>                                              # 3–5 diverse (rung 2)
  <example><input>...</input><category>billing</category></example>
</examples>
<instructions>
  Quote the relevant policy in <quotes> tags first,    # ground in quotes (rung 7)
  then classify. Respond directly without preamble.    # positive instruction
</instructions>
<query>...the actual ticket...</query>                  # query at the END

# request config:
#   thinking={"type":"adaptive"}, effort tuned by evals  (rung 3 / Section D)
#   output_config JSON schema for the category           (Tier 3 / Section E)

The anti-patterns

Anti-patternWhy it fails / the fix
Vague instructionsFails the golden rule — "If they'd be confused, Claude will be too." Fix: clarity + motivation (rung 1).
No examplesSkips the single highest-leverage lever. Fix: 3–5 diverse examples (rung 2).
Negative-only instructions"Tell Claude what to do instead of what not to do"; positive examples beat "what not to do" lists.
Hiding the thinking / over-thinkingUse <thinking>/<answer> to keep reasoning inspectable; over-thinking is the opposite failure — fix by lowering effort.
Over-prompting / shoutingAggressive language causes overtriggering — see callout below.
Relying on prefillBroken on 4.6+ (400). Migrate to Structured Outputs / direct instructions (Section E).
Assuming the prompt transfers blindlyRe-baseline verbosity, effort, tone on a new model (Section F).

📢 Over-prompting / shouting → overtriggering▼

Verbatim: "Where you might have said 'CRITICAL: You MUST use this tool when…', you can use more normal prompting like 'Use this tool when…'." Aggressive, all-caps, "you MUST" language on current models causes overtriggering — tools fire when they shouldn't. Also verbatim: "Remove over-prompting. Tools that undertriggered in previous models are likely to trigger appropriately now." The mental model: newer models follow instructions more literally, so the emphatic phrasing that compensated for older models now over-corrects. Calm, direct phrasing wins. (source: best practices)

Close the loop

The lesson ends where it began (Section A): define success criteria → build evals → draft prompt → climb the ladder only as far as your evals require → if cost/latency is the blocker, change model or lower effort rather than over-prompting → re-run evals on any model change. Prompting and model selection are the two peer levers; the empirical loop is the discipline that keeps both honest. (source: prompt-engineering overview)
